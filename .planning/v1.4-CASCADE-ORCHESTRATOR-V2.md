# v1.4 Cascade V2 — Orchestrator-Worker Pattern (Anthropic-Aligned)

**Reorganization rationale:** V1 used coarse 4-subagent waves per phase. V2 applies Anthropic's published multi-agent research pattern: **lead orchestrator + wide tree of narrow Haiku workers + dedicated verifier**. Token cost ~3-5× higher per phase, wall-clock ~40% faster, error rate substantially lower (verifier catches BLOCKER-4-style "file did not persist" failures before they reach the user).

**Mission:** Phases 5→9 to v1.4 sign-off, zero human intervention, all waves automated.

---

## Anthropic Pattern Applied

Reference: Anthropic "How we built our multi-agent research system" + Claude best practices for agentic workflows.

### Roles

| Role | Model | Job | Context |
|---|---|---|---|
| **Lead Orchestrator** | Sonnet 4.6 | Phase planning, wave dispatch, synthesis, escalation | Full phase scope |
| **Plan-Lead** (1 per plan) | Haiku 4.5 | Decompose plan into ≤4 narrow tasks; spawn workers; aggregate | One plan |
| **Specialist Worker** (≤4 per plan-lead) | Haiku 4.5 | Single narrow task: schema OR callable OR UI OR tests | One task |
| **Verifier** (1 per wave) | Sonnet 4.6 | Read claimed deliverables, verify on disk, run tests, gate next wave | Wave summaries + grep/read |
| **Persistence Auditor** (1 per wave) | Haiku 4.5 | After every Worker claims completion, `Glob`+`Read` to confirm files exist on disk (catches BLOCKER-4 sandbox failures) | File paths claimed |

### Why Haiku 4.5 for workers
- Narrow, well-specified tasks (schema/callable/UI/tests) fit Haiku's strength
- 1/3 the cost of Sonnet, 2× speed
- Anthropic blog: subagent fanout amortizes orchestrator cost — wide+cheap > narrow+expensive
- Sonnet reserved for orchestration/verification where reasoning + cross-context synthesis matter

### Parallelism rules (Anthropic guidance)
1. **All independent tool calls in one message.** Workers fire `Read`/`Grep`/`Glob`/`Edit` calls in parallel batches.
2. **Wave-level parallelism.** Plans within a wave run simultaneously; waves run sequentially (dependency-driven).
3. **Compressed handoffs.** Workers return ≤200-line summaries, not transcripts. Orchestrator never sees worker tool output.
4. **Fresh context per worker.** No conversation history bleed between specialists.
5. **Prompt caching.** Reuse system prompts across waves (5-min cache TTL respected — workers spawn in batches inside cache window).

## Failure Modes V2 Closes (V1 missed these)

| V1 failure | V2 mitigation |
|---|---|
| Worker reports "file written" but disk empty (BLOCKER 4) | **Persistence Auditor** does `Glob` of claimed paths immediately after worker returns. If missing, marks worker FAILED, respawns with explicit `Write` retry. |
| HMAC signature drift undetected (BLOCKER 1 was false alarm — but real ones happen) | **Verifier** runs targeted greps (`hash:\s*''` etc.) and ADR-keyed regex on worker output. |
| Compliance claim inflation (BLOCKER 2 — Phase 5 credited Art. 167 it didn't deliver) | **Verifier** cross-checks SUMMARY.md claims against `docs/DICQ_CLOSURE_TRACKER_v1.4.md` deltas + actual file evidence. Inflated claims rolled back. |
| Doc-driven DICQ gains unsupported by code (BLOCKER 6) | DICQ tracker writes are gated on Verifier signoff. Until verifier confirms code-level evidence, the wave's DICQ delta stays at zero. |
| Cascade deadlock if next-phase planner stalls | Orchestrator timeout = 30 min for `/gsd-plan-phase`; on timeout spawns a fresh plan-lead with the same scope. Idempotent re-entry. |

## Auto-Trigger Sequence (no polling script needed)

V1 used a 5-min polling shell script. V2 uses GSD Skill chaining inside the orchestrator:

```
Lead Orchestrator (one continuous session):
  for phase in [5, 6, 7, 8, 9]:
    plan = invoke_skill("gsd-plan-phase", {phase, auto_approve, all_permissions})
    wait_for_plan_md(timeout=30min, retry_on_timeout=true)
    for wave in plan.waves:
      workers = spawn_parallel_haiku(wave.tasks)
      summaries = await_all(workers)
      audit = spawn_persistence_auditor(summaries.claimed_files)
      if audit.failed: respawn_failed_workers()
      verify = spawn_verifier(wave, summaries)
      if verify.failed: spawn_fix_worker(verify.failures); retry up to 3
    write_phase_summary(phase)
    update_state_md(phase, status=COMPLETE)
```

No polling, no cron, no shell script. The orchestrator session itself is the loop. If the session dies, GSD checkpoint resumes from last-completed wave.
