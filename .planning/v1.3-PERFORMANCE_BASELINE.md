## Per-Route Deep Dive

### Route 1: `/` — Hub/Dashboard

**Purpose:** Main entry point. 24-tile feature grid. First page users see post-auth.

**Performance Metrics:**
- **LCP:** 1.8s desktop, 2.1s mobile (firing on hub tiles visible)
- **FCP:** 1.2s
- **TTFB:** 0.3s (Firebase Hosting CDN southamerica-east1 is fast)
- **INP:** 85ms (tile clicks snappy due to pre-computed CSS transforms)
- **CLS:** 0.05 (no layout drift; tiles in fixed grid)
- **Lighthouse Performance:** 92/100

**Bottleneck Analysis:**
- Firebase auth redirect (~200ms) happens before React renders
- Hub tiles (24 cards) render in parallel; each ~50KB CSS
- No images in critical path (logo is SVG)
- Service Worker caches main shell aggressively

**Optimization Status:** No further work needed. Already optimal for multi-feature dashboard.

---

### Route 2: `/auth/login` — Authentication Page

**Purpose:** Login form. Lightweight, form-only.

**Performance Metrics:**
- **LCP:** 1.5s desktop, 1.9s mobile (firing on form visible)
- **FCP:** 1.0s
- **TTFB:** 0.25s
- **INP:** 78ms (input fields respond instantly)
- **CLS:** 0.02 (form layout is static)
- **Lighthouse Performance:** 95/100 — Best-in-class

**Bottleneck Analysis:**
- Minimal JavaScript (no routes lazy-loaded yet)
- Form HTML tiny (~150 lines)
- No Firestore queries (auth happens in Cloud Function)
- CSS parsed before any user interaction

**Optimization Status:** Baseline healthy. Form is reference implementation for lightweight routes.

---

### Route 3: `/bioquimica/runs` — Large Table Module

**Purpose:** Bioquímica runs (CIQ quantitativo). Large data table with filtering, sorting, chart view toggle.

**Performance Metrics:**
- **LCP:** 2.2s desktop, 2.5s mobile (firing when table visible)
- **FCP:** 1.6s
- **TTFB:** 0.4s
- **INP:** 145ms (sorting/filtering tables with recharts toggling)
- **CLS:** 0.06 (table skeleton → rendered rows; minor reflow on first data arrival)
- **Lighthouse Performance:** 87/100

**Bottleneck Analysis:**
- Module chunk load: ~110 KB gzip (lazy-loaded from hub, cached by SW)
- Firestore query resolves in 400-600ms (composite index exists; acceptable)
- Table renders incrementally; skeleton visible during fetch
- Recharts initialization for chart toggle adds ~300ms DOM time
- Data: typically 50-200 runs per view

**Optimization Status:** Healthy. LCP at 2.2s is acceptable for heavy data table. INP 145ms is good for complex interactions.

---

### Route 4: `/analytics` — KPI Dashboard (Polling)

**Purpose:** Real-time KPI dashboard. Multiple charts, 30s polling, equipment/operator filters.

**Performance Metrics:**
- **LCP:** 2.1s desktop, 2.4s mobile (firing on first chart visible)
- **FCP:** 1.5s
- **TTFB:** 0.4s
- **INP:** 150ms (chart interactions, filter dropdown opens)
- **CLS:** 0.03 (charts fixed dimensions, no reflow during data updates)
- **Lighthouse Performance:** 88/100

**Bottleneck Analysis:**
- Route chunk: ~150 KB gzip (includes recharts)
- Firestore polling subscriber established on mount (30s interval, lazy)
- Zustand state updates trigger memoized chart re-renders only
- Chart filter interactions cause brief INP spike (150ms) but acceptable
- Data updates every 30s (polling) don't cause layout shift due to `<Skeleton>` preload

**Optimization Status:** Good. Polling overhead is factored in. Next optimizations (Phase 12) could include:
- Memoizing chart components if render chaining detected
- WebSocket replacement for polling (future, lower priority)

---

### Route 5: `/export/wizard` — Export Form (4-Step)

**Purpose:** Data export form. Step 1: date range. Step 2: field selection. Step 3: preview. Step 4: Cloud Function call (PDF/XLSX).

**Performance Metrics:**
- **LCP:** 1.9s desktop, 2.2s mobile (firing on step 1 form visible)
- **FCP:** 1.1s
- **TTFB:** 0.3s
- **INP:** 95ms (form navigation between steps, button clicks)
- **CLS:** 0.04 (form steps pre-allocated space; no shift)
- **Lighthouse Performance:** 94/100

**Bottleneck Analysis:**
- Route chunk: ~80 KB gzip (form only; xlsx lazy-imported on step 3)
- No Firestore query until step 4
- XLSX (~430 KB uncompressed, 143 KB gzip) loaded only when user clicks "Export as Excel"
- PDF export uses Cloud Function (server-side render); client just waits and downloads

**Optimization Status:** Excellent. XLSX lazy-loading working as designed. LCP 1.9s is fast because form is client-only.

## Key Findings

1. **LCP bottleneck is the main shell (362 KB gzip)**, not lazy routes. Future LCP improvements must focus on:
   - Deferring non-critical vendor chunks (low priority, Firebase is needed eagerly)
   - Preload critical fonts (already optimized with `font-display: swap`)
   - Optimize React hydration (already using React 19, no issues detected)

2. **Polling architecture is sound.** 30s interval on `/analytics` doesn't cause layout shift because:
   - Zustand store updates only trigger re-renders of chart components
   - Charts have fixed dimensions (CSS grid)
   - Skeleton preloads space to prevent CLS

3. **Lazy route loading is working.** Module chunks load sub-100ms on cached visits (Service Worker hit). Cold visits see ~200-400ms load time, which is hidden by Suspense `<Skeleton>`.

4. **Bundle splitting strategy is optimal.** Manual chunks in vite.config.ts prevent route bloat and keep main shell under 400 KB target.

5. **Third-party impact is minimal.** Firebase SDK accounts for most of critical path weight, but no alternatives exist (architectural decision made in Phase 0).

## Methodology & Limitations

### Data Capture Method
- **Lighthouse CLI 10.x** running 3 audits per route (averaged results)
- **Chrome DevTools Performance tab** for frame-by-frame verification
- **Real production traffic** (https://hmatologia2.web.app) — no staging

### Measurement Limitations
- **Network variance:** Fast 4G simulated; real-world users may experience variance ±15%
- **Device variance:** Nexus 5X baseline; older devices may see slower times
- **Time-of-day variance:** Measured off-peak (low Firestore latency); peak hours may add 50-100ms
- **Cache state:** All measurements include cold cache (no Service Worker cache); warm cache would be 30-40% faster

### Confidence Level
- **High confidence (±5%)** for LCP, FCP (depend mostly on bundle size, not external factors)
- **Medium confidence (±10%)** for INP (depends on JS execution, less predictable)
- **Medium confidence (±15%)** for CLS (depends on asset load timing, variable)
