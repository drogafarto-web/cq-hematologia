  - Authority: Approve rollbacks, emergency changes, escalations

ERROR BUDGET POLICY
  - 30-day allowance: 99.9% uptime = 43.2 minutes acceptable downtime

  - 15-60 min incidents: Full duration consumed
  - >60 min incidents: Critical incident + post-mortem required

KEY METRICS & TARGETS
================================================================================

MONITORING TARGETS (All Phase 4 production):

  Portal Auth:
    - Auth success rate: >99% (Alert if <95%)
    - Email delivery: >95% (Alert if <80%)
    - Auth latency p95: <1.5s (Alert if >2s)
    - Active sessions: Baseline 50-200 concurrent

  NOTIVISA Queue:
    - Submission success: >99% (Alert if <95%)
    - Queue processing: <5 min avg (Alert if >15min pending)
    - Webhook ACK: >99% (should be immediate)

  Firestore:
    - Rule violation rate: <0.01% (Alert if >0.1%)
    - Multi-tenant isolation: 0 cross-lab reads (Alert if >0)
    - Quota usage: <50% daily (Alert if >80%)

  System:
    - Function error rate: <0.1% (Alert if >0.5%)
    - Firestore permission errors: 0 unexpected (normal: 1-2 expected)
    - HTTP 5xx errors: 0 (sustained >5min = escalate)

RUNBOOK USAGE GUIDE
================================================================================

When alert fires in Slack:

  1. READ alert message carefully (which service? which severity?)
  2. CONFIRM alert is real (not false positive) — check logs
  3. FIND matching runbook from table below:

     Alert Name                    | Severity | Runbook Path
     ──────────────────────────────┼──────────┼──────────────────────────────
     Portal Auth Failures Spike    | P1       | phase-4-auth-failures.md
     NOTIVISA Queue Stuck          | P1       | phase-4-notivisa-queue.md
     Firestore Rules Rejections    | P2       | phase-4-firestore-rules.md
     Email Delivery Failure        | P2       | phase-4-email-delivery.md
     Function Latency Degradation  | P3       | phase-4-function-latency.md

  4. FOLLOW runbook step-by-step (copy-paste gcloud commands)
  5. DETERMINE root cause (decision tree in each runbook)
  6. EXECUTE fix (deploy, restart, rotate secret, etc.)
  7. VALIDATE recovery (error rate drops, service healthy)
  8. DOCUMENT incident (create ticket with template from quick ref)
  9. ESCALATE if unresolved >threshold (CTO for P1 >30min)

All runbooks include:
  - Immediate triage (2-3 min)
  - Root cause diagnosis (5-10 min)
  - Fix procedures with copy-paste gcloud commands
  - Recovery validation criteria
  - Post-incident checklist
