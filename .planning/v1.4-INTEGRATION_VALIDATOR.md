## 3. Cross-Wave E2E Test Scenarios (8–12 scenarios)

### Scenario 1: Portal Auth → Laudo Review → Sign (Wave 1)

**Phase 4 → Phase 4-02 (same phase, but multi-plan)**

```
Flow:
  1. RT login (Phase 4-01 callable) → get auth token
  2. List laudos (Phase 4-01 callable) → paginated results
  3. Fetch laudo (Phase 4-01 callable) → full document + signature context
  4. Render DraftEditor (Phase 4-02 UI) → allow edits in draft state
  5. Submit signature (Phase 4-02 UI) → call Phase 4-01 signLaudo callable
  6. Verify audit trail (Phase 4-02 audit log) → entry logged with operatorId + ts

Validation:
  - Auth token JWT valid + claims match RT identity
  - Laudo state = "draft" when fetched
  - Signature hash validates (chainHash)
  - Audit trail entry ✓
```

**Integration blockers:**

- Callable signature stability (return type must match UI expectations)
- Firestore permissions (callable able to write to `laudos/{laudoId}/assinatura`)

### Scenario 2: NOTIVISA Queue → Event Processing (Wave 1 + Ops)

**Phase 4-03 → Phase 8-06 (audit trail validation)**

```
Flow:
  1. Generate critical result (any module)
  2. Phase 4-03 NOTIVISA queue processor detects in `notivisa-outbox`
  3. Queue scheduler polls every 30s (with exponential backoff)
  4. Event sent to NOTIVISA sandbox API
  5. Success → mark `processado=true` in Firestore
  6. Phase 8-06 audit trail logs event (entry: who triggered, timestamp, result)
  7. Verify no reprocessing (idempotency check: Art. 6º §1 compliance)

Validation:
  - Event schema matches NOTIVISA Art. 6º §1 payload spec
  - Polling interval 30s ± 5s (not too aggressive)
  - Exponential backoff kicks in after 3 retries
  - Audit trail captures event id, timestamp, operator, status
```

**Integration blockers:**

- NOTIVISA sandbox credentials (Phase 10 pre-req)
- Firestore rules allow Phase 4-03 callable to write to `notivisa-outbox`
- Callable retry logic must not exceed Firestore transaction limits

## 5. Escalation Protocol

### When to Escalate (Red Flags)

🔴 **IMMEDIATE escalation to Wave coordinator + CTO:**

1. **Schema conflict** — two phases modify same collection without coordination
2. **Orphaned callable** — function deployed but no consuming phase/test
3. **Permission denial** — callable/trigger blocked by Firestore rules (not in spec)
4. **Test data incompatibility** — fixture generated by Phase X breaks Phase Y tests
5. **Auditor blocker** — Phase 11 feedback contradicts deployed code (CAPA, compliance)
6. **Security finding critical** — Phase 10 pen-test finds critical vulnerability with no mitigation assigned

### Escalation Template

```
ESCALATION: [Wave] [Phase] — [Blocker category]

Blocker:
  [1-line description]

Details:
  - Discovery: [where/when/how found]
  - Impact: [which phases blocked + hours of delay estimate]
  - Root cause: [schema change / callable sig / perm issue / test fixture / auditor feedback]
  - Evidence: [PR link / test failure / error log / meeting notes]

Proposed mitigation:
  - Option A: [fix option 1]
  - Option B: [fix option 2]
  - CTO decision: [awaiting approval]

Deadline to resolve:
  [date] — [reason: blocks Phase X kickoff on Y, auditor ceremony on Z, etc.]
```

### Escalation Routing

| Scenario                             | Route                                | SLA                        |
| ------------------------------------ | ------------------------------------ | -------------------------- |
| Schema conflict between phases       | Validator → Wave coordinator → CTO   | 2 hours                    |
| Auditor feedback contradicts code    | Validator → CTO + RT lead → Auditor  | 4 hours (meeting next day) |
| Pen-test critical finding (Phase 10) | Validator → Security engineer → CTO  | 1 hour                     |
| Test data fixture incompatibility    | Validator → QA lead → Phase owner    | 24 hours                   |
| Callable signature regression        | Validator → Phase owner → CTO review | 2 hours                    |
