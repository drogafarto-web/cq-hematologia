## Architecture Overview

### What's Running Right Now

**Phase 5 Execution (started 2026-05-08):**

- **4 Subagents** spawned via `/gsd-execute-phase phase:5 waves:15`
- **Wave 1:** Plans 05-01 + 05-02 in parallel (days 1-5)
- **Wave 2:** Plans 05-03 + 05-04 in parallel (days 5-7)
- **Est. completion:** 2026-05-14 00:00 UTC

**Why this architecture:**

- **Parallel waves** = faster execution (2 weeks per 4-plan phase)
- **Async subagents** = runs in background while you work
- **State-driven triggers** = automatic phase kickoffs when previous completes
- **Sleep-safe** = survives restarts, network interruptions, system sleeps

### How Auto-Triggers Work

```
Polling Loop (every 5 minutes):
  1. Read .planning/STATE.md
  2. Check each phase status (5-9)
  3. For each phase that's ✅ COMPLETE:
       → If next phase queued but not started:
           ✓ Invoke /gsd-plan-phase phase:N+1
           ✓ Invoke /gsd-execute-phase phase:N+1 waves:15
           ✓ Log trigger event
  4. Sleep 5 minutes, repeat
  5. When Phase 9 ✅ COMPLETE → cascade exits (mission complete)
```

**Polling scripts:**

- Windows: `scripts/v1.4-cascade-orchestrator.ps1` (PowerShell)
- macOS/Linux: `scripts/v1.4-cascade-orchestrator.sh` (Bash)

Both do the same thing. Pick one for your OS.

## If Something Breaks (Troubleshooting)

### Scenario 1: Phase stuck >24 hours

**Check:**

```bash
# Is the polling script running?
ps aux | grep "v1.4-cascade-orchestrator"

# Are subagents in that phase still active?
ls -lh .planning/phases/0N-*/.checkpoint 2>/dev/null  # (recent timestamp = active)

# Check for errors in logs
tail -50 .planning/ORCHESTRATOR_LOG.md | grep ERROR
```

**Fix:**

1. **Check CHECKPOINT.md** in the stalled phase directory
2. **Inspect error logs** in Cloud Logs (if deployed)
3. **If safe to retry:** `git reset --hard HEAD~1` (undo last commit, restart phase)
4. **If unsafe:** Email drogafarto@gmail.com with logs

### Scenario 2: Polling script crashed

**Restart:**

```bash
# PowerShell (Windows)
pwsh .\scripts\v1.4-cascade-orchestrator.ps1 &

# Bash (macOS/Linux)
bash ./scripts/v1.4-cascade-orchestrator.sh &

# (& runs in background; script automatically polls and continues)
```

### Scenario 3: Phase triggered but execution didn't start

**Check:**

```bash
# Does the PLAN.md exist?
ls .planning/phases/0N-*/*.PLAN.md

# If yes, manually invoke:
/gsd-execute-phase phase:N waves:15 auto-approve all-permissions sleep-safe

# Log the manual intervention:
echo "$(date): Manual Phase N restart after polling failure" >> .planning/ORCHESTRATOR_LOG.md
```

### Scenario 4: Need to pause cascade temporarily

```bash
# Stop polling script
pkill -f "v1.4-cascade-orchestrator"  # (kills polling, doesn't affect running phases)

# Pause running phases
/gsd-pause-work  # (saves checkpoint, can resume later)

# When ready to resume:
/gsd-resume-work  # (restores context, resumes from checkpoint)
bash ./scripts/v1.4-cascade-orchestrator.sh &  # (restart polling)
```

## FAQ

**Q: Can I interrupt the cascade?**  
A: Yes. `pkill -f "v1.4-cascade-orchestrator"` stops polling. Running phases continue (graceful). Use `/gsd-pause-work` to checkpoint, then `/gsd-resume-work` to continue.

**Q: What if Phase 5 takes longer than 6 days?**  
A: Cascade adjusts. Polling detects completion whenever it happens and auto-triggers Phase 6. Timeline slips proportionally, but sequencing is maintained.

**Q: Do I need to monitor every day?**  
A: No. Cascade is fully autonomous. Check status 1-2 times per week if desired, but zero action required.

**Q: What if a subagent dies?**  
A: GSD automatically respawns it. Logged to ORCHESTRATOR_LOG.md. No action needed.

**Q: Can I run the cascade script on a laptop while I travel?**  
A: Yes, if you have stable internet. Cascade survives network blips (GSD checkpointing). If connection lost >30 min, manually restart script when back online.

**Q: What about holidays/weekends?**  
A: Cascade runs 24/7. No pause for weekends. If you need to pause, use `/gsd-pause-work` + `/gsd-resume-work`.
