# Phase 3.1: Foundation (Mobile + Analytics + Export) — Research

**Researched:** 2026-05-04  
**Domain:** React Native CLI + Firestore Analytics + Cloud Functions Export Infrastructure  
**Confidence:** HIGH  

## Summary

Phase 3.1 Foundation establishes infrastructure for three parallel workstreams: mobile app (React Native CLI with Firebase auth), analytics aggregation (Cloud Functions + Firestore queries), and export (Pub/Sub + SheetJS + Cloud Storage). Architecture decisions are locked via CONTEXT.md; this research verifies implementation patterns against Firebase best practices, RDC 978 compliance requirements, and existing HC Quality conventions.

**Key finding:** All three architectural decisions (native RN CLI, hourly aggregation, server-side export) are supported by official Firebase SDKs and align with Phase 2 patterns. Primary risks are around Cloud Function timeout handling for large datasets and mobile iOS/Android provisioning complexity — both have documented mitigation strategies.

**Primary recommendation:** Start mobile and analytics workstreams in parallel (dependencies are minimal); export can follow after Day 3 when Cloud Function callable patterns are validated. Unit test infrastructure already exists (Jest in functions/ for server, Vitest in web for client); extend to mobile via `@testing-library/react-native` + Jest preset.

**Data flow walkthrough:**

1. **Mobile offline read:** App subscribes to `/labs/{labId}/analytics-meta` + `/analytics-agg/{timestamp}` via Firestore onSnapshot. LocalStorage caches results. No write possible offline (buttons disabled).

2. **Analytics hourly aggregation:** CloudSchedule trigger fires. `analyticsAggregation()` CF queries `/labs/{labId}/runs`, `/lots/{lotId}/runs`, `/nc/...` (multi-collection). Computes turnaround%, compliance%, retrabalho%. Writes `/labs/{labId}/analytics-agg/2026-05-04T14:00:00Z` + updates `/labs/{labId}/analytics-meta` with `latestAggregateId`, `freshness`, `nextUpdateAt`. Emits audit log.

3. **Web dashboard refresh:** Dashboard component watches `analytics-meta` (lightweight). On change, refetch aggregate. Zustand caches in memory until next update.

4. **Export initiation:** User picks format + date range in ExportWizard. Calls `initiateExport({labId, format, dateRange})` callable. Callable validates claims, creates `/labs/{labId}/export-jobs/{jobId}` with `status: 'pending'`, `initiatedBy`, `ts`. Publishes message to `export-jobs` Pub/Sub topic. Returns `jobId` to UI.

5. **Export background processing:** Pub/Sub subscriber Cloud Function `processExportJob()` dequeues. Reads job spec. Fetches 100k+ docs in batches (streaming). Generates XLSX via SheetJS. Uploads to Cloud Storage as `/exports/{labId}/{jobId}/ciq-runs-2026-05.xlsx`. Generates 7-day signed URL. Updates job record: `status: 'done'`, `gsPath`, `signedUrl`, `completedAt`. Audit log with size + operator.

6. **Job status polling:** Web UI polls `/labs/{labId}/export-jobs/{jobId}` or subscribes (preferred). When `status === 'done'`, renders download button. User clicks → browser downloads via signedUrl.

### Pattern 1: Firebase Auth Session Persistence in React Native

**What:** Automatic login restoration via `initializeAuth()` with `AsyncStorage` persistence.

**When to use:** Every React Native app with Firebase Auth needs this to survive app restarts.

**Example:**

```typescript
// src/mobile/services/firebaseConfig.ts
import { initializeApp } from 'firebase/app';
import { initializeAuth, getReactNativePersistence } from 'firebase/auth';
import AsyncStorage from '@react-native-async-storage/async-storage';

const firebaseConfig = { /* ... */ };
const app = initializeApp(firebaseConfig);

// Critical: provide AsyncStorage to persist auth state
export const auth = initializeAuth(app, {
  persistence: getReactNativePersistence(AsyncStorage),
});

// src/mobile/hooks/useMobileAuth.ts
import { useEffect, useState } from 'react';
import { onAuthStateChanged, signInWithEmailAndPassword, signOut } from 'firebase/auth';
import { auth } from '../services/firebaseConfig';

export function useMobileAuth() {
  const [user, setUser] = useState(null);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    // onAuthStateChanged fires once at app start with persisted user (or null)
    const unsubscribe = onAuthStateChanged(auth, (currentUser) => {
      setUser(currentUser);
      setIsLoading(false);
    });

    return unsubscribe; // cleanup
  }, []);

  return {
    user,
    isLoading,
    login: (email: string, password: string) =>
      signInWithEmailAndPassword(auth, email, password),
    logout: () => signOut(auth),
  };
}
```

**Key:** `onAuthStateChanged` is asynchronous and resolves once Firebase connects. Always initialize loading state to true and guard render tree with `isLoading` check. [CITED: Firebase docs + rnfirebase.io/auth/usage]

### Pattern 2: Hourly Cloud Function Aggregation with Firestore Composite Indices

**What:** CloudSchedule trigger → Cloud Function that queries 100k+ docs, computes aggregates, writes atomically.

**When to use:** Pre-computed analytics that must be reproducible, versionable, and compliant (RDC 978).

**Example:**

```typescript
// functions/src/modules/analytics/analyticsAggregation.ts
import { onSchedule } from 'firebase-functions/v2/scheduler';
import { getFirestore, writeBatch, query, collection, where, getDocs } from 'firebase-admin/firestore';
import { Timestamp } from 'firebase-admin/firestore';
import { logger } from 'firebase-functions';
import { queryKPIs } from './queryKPIs';

export const analyticsAggregation = onSchedule(
  {
    schedule: 'every 60 minutes',    // Hourly
    timeoutSeconds: 540,              // Max for scheduled function
    region: 'southamerica-east1',
    memory: '512MB',                  // Memory for large queries
  },
  async (context) => {
    const db = getFirestore();
    const labs = await db.collection('labs').listDocuments();

    for (const labRef of labs) {
      const labId = labRef.id;

      try {
        // Query aggregation data (multi-collection)
        const kpis = await queryKPIs(db, labId);

        // Write aggregates atomically
        const batch = writeBatch(db);
        const timestamp = Timestamp.now();
        const aggId = timestamp.toDate().toISOString(); // 2026-05-04T14:00:00Z

        batch.set(
          db.doc(`labs/${labId}/analytics-agg/${aggId}`),
          {
            labId,
            timestamp,
            turnaroundMinAvg: kpis.turnaroundAvg,
            retrabalhoPercent: kpis.retrabalhoRate,
            compliancePercent: kpis.complianceRate,
            openNCCount: kpis.openNCs,
            // ... more fields
            computedAt: timestamp,
          }
        );

        // Update meta: latest aggregate pointer
        batch.set(
          db.doc(`labs/${labId}/analytics-meta/current`),
          {
            latestAggregateId: aggId,
            freshness: 'current',
            nextUpdateAt: new Timestamp((timestamp.seconds + 3600), 0),
          },
          { merge: true }
        );

        await batch.commit();
        logger.info(`[Analytics] Lab ${labId} aggregated successfully`);
      } catch (err) {
        logger.error(`[Analytics] Lab ${labId} failed:`, err);
        // Continue with next lab (don't cascade fail)
      }
    }
  }
);
```

**Firestore indices required:**

```json
// firestore.indexes.json (add to existing)
{
  "indexes": [
    {
      "collectionGroup": "items",
      "queryScope": "COLLECTION",
      "fields": [
        { "fieldPath": "labId", "order": "ASCENDING" },
        { "fieldPath": "status", "order": "ASCENDING" },
        { "fieldPath": "completedAt", "order": "DESCENDING" }
      ]
    }
  ]
}
```

**Key insights:**
- Scheduled function timeout is 540s; if query takes longer, split by lab-batches or shard data
- Use composite indices on (labId, status, dateField) for query speed
- Atomic batch.commit() ensures aggregates never partially written
- `labId` redundantly stored + filtered for RDC 978 auditability
- [CITED: Firebase docs on write-time aggregations + best practices]

### Pattern 3: Pub/Sub Job Queue for Export Processing

**What:** Client initiates export → callable creates job record + publishes Pub/Sub message → background worker processes async.

**When to use:** Long-running operations that might timeout (>540s for callable). Decouples initiation from processing.

**Example:**

```typescript
// functions/src/modules/exports/initiateExport.ts (callable)
import { onCall } from 'firebase-functions/v2/https';
import { getFirestore, writeBatch, serverTimestamp } from 'firebase-admin/firestore';
import { PubSub } from '@google-cloud/pubsub';
import { z } from 'zod';

const InitiateExportSchema = z.object({
  labId: z.string().min(1),
  format: z.enum(['XLSX_CIQ', 'XLSX_NC', 'PDF_REPORT', 'CSV_AUDIT']),
  dateRangeStart: z.string().datetime(), // ISO
  dateRangeEnd: z.string().datetime(),
});

export const initiateExport = onCall(
  {
    region: 'southamerica-east1',
    timeoutSeconds: 60, // Callable; no long processing here
  },
  async (request) => {
    const { labId, format, dateRangeStart, dateRangeEnd } = InitiateExportSchema.parse(
      request.data
    );

    const db = getFirestore();
    const pubsub = new PubSub();

    // 1. Validate claims (admin middleware would also check, but layer defense-in-depth)
    const memberDoc = await db.doc(`labs/${labId}/members/${request.auth?.uid}`).get();
    if (!memberDoc.exists) {
      throw new Error('User not member of lab');
    }

    // 2. Create job record
    const jobId = `${Date.now()}-${Math.random().toString(36).substr(2, 9)}`;
    const jobRef = db.doc(`labs/${labId}/export-jobs/${jobId}`);

    const batch = writeBatch(db);
    batch.set(jobRef, {
      labId,
      jobId,
      format,
      dateRangeStart,
      dateRangeEnd,
      status: 'pending',
      initiatedBy: request.auth.uid,
      initiatedAt: serverTimestamp(),
      completedAt: null,
      signedUrl: null,
      gsPath: null,
      error: null,
    });

    // Audit log
    batch.set(
      db.collection(`auditLogs/${labId}/items`).doc(),
      {
        labId,
        operationType: 'EXPORT_INITIATED',
        operatorId: request.auth.uid,
        format,
        jobId,
        ts: serverTimestamp(),
      }
    );

    await batch.commit();

    // 3. Publish Pub/Sub message
    const topic = pubsub.topic('export-jobs');
    const messageId = await topic.publish(
      Buffer.from(
        JSON.stringify({
          labId,
          jobId,
          format,
          dateRangeStart,
          dateRangeEnd,
        })
      )
    );

    logger.info(`[Export] Job ${jobId} enqueued, Pub/Sub msgId=${messageId}`);

    return { jobId }; // Return immediately; client polls for status
  }
);
```

**Key insights:**
- Callable handles request validation + job creation (fast, <60s)
- Pub/Sub decouples; worker can run 540s without blocking client
- Job record update is audit trail (who initiated, when completed, file size)
- [CITED: Firebase Pub/Sub docs]

### Anti-Patterns to Avoid

- **Client-side XLSX generation for >10k rows:** Browser memory exhausts. Always server-side. [VERIFIED: SheetJS community edition docs]
- **Real-time listeners for every CIQ run (analytics):** Costs skyrocket at >100k docs. Hourly aggregation is correct model.
- **No composite indices for (labId, status, date) queries:** Query times degrade. Pre-create indices before go-live.
- **Signed URLs without expiry:** Security risk. Always set 7-day ceiling; longer invites abuse. [VERIFIED: Firebase docs]
- **Ignoring Cloud Function timeout in Pub/Sub:** Subscriber functions have 540s limit. Test with 100k-row XLSX to validate fit.
- **Storing full PDF files in Firestore:** Use Cloud Storage only; store URL in Firestore metadata. Limits document sizes, improves reads.
- **Client-side auth token caching beyond Firebase SDK:** AsyncStorage via Firebase SDK is safe; manual token management = reauth failures.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Mobile auth session persistence | Custom localStorage wrapper | Firebase SDK + AsyncStorage | Handles token refresh, claim updates, sign-out cascades automatically |
| Firestore hourly scheduling | Cron job server + webhook | Firebase CloudSchedule + onSchedule | Autoscales, no ops burden, integrates with rules + audit |
| Background job queue | DIY callback loop in Cloud Function | Google Cloud Pub/Sub | Retry logic, dead-letter topics, decouples processing from request |
| XLSX generation server-side | Write raw XLSX bytes | SheetJS (xlsx npm package) | Handles formatting, formulas, large datasets; tested in production |
| Signed URL generation | Manual crypto + expiry logic | Cloud Storage SDK `getSignedUrl()` | Handles key rotation, expiry enforcement, edge cases |
| KPI aggregation formulas | Ad-hoc queries per dashboard | Scheduled pre-computed aggregates | Consistent results, RDC 978 reproducible, no re-calculation per user |
| Analytics state caching | Redux or Context | Zustand (2KB store) | Matches project convention, simpler than Redux, matches web stack |

**Key insight:** Every item above has hidden complexity (token expiry, retry exhaustion, format edge cases, timezone drift). Battle-tested libraries survive production better than custom code.

## Common Pitfalls

### Pitfall 1: Misconfiguring AsyncStorage Persistence on App Restart

**What goes wrong:** `initializeAuth` without explicit `persistence` option defaults to memory-only. User logs in, closes app, reopens → logged out. Terrible UX.

**Why it happens:** Firebase JS SDK docs show examples without persistence for web (browser auto-persists to IndexedDB). React Native requires explicit opt-in.

**How to avoid:** Always use `initializeAuth(app, { persistence: getReactNativePersistence(AsyncStorage) })` in RN projects. Test: log in, kill app, relaunch → user auto-logged in.

**Warning signs:** User reports "I have to log in every time I open the app."

---

### Pitfall 2: Cloud Function Timeout on Large Dataset Export

**What goes wrong:** CF starts generating 100k-row XLSX, hits 540s limit at row 50k, returns 504 Gateway Timeout. User sees error; export lost.

**Why it happens:** SheetJS loads full workbook in memory. Querying + generation + upload takes time. Under pressure, sequential operations cascade delays.

**How to avoid:**
1. **Test locally with 100k+ rows** before deploy. Time generation. If >450s, redesign.
2. **Use Pub/Sub** (Phase 3.1 does this) so subscriber function has 540s isolation, not callable's 60s.
3. **Stream if possible:** SheetJS has streaming API for CSV exports; use for audit log export.
4. **Monitor:** Log CF start/finish times. Alert if processing takes >400s (leave margin).

**Warning signs:** "Export sometimes fails for large labs"; CF logs show consistent timeouts.

---

### Pitfall 3: Missing Composite Indices for Analytics Queries

**What goes wrong:** Query for `WHERE labId='...' AND status='final' AND completedAt > now-24h` returns "requires composite index" error. Aggregation function fails hourly.

**Why it happens:** Firestore requires indices for multi-field filters. Auto-index only kicks in after first query error. Phase 3.1 won't catch this until production traffic.

**How to avoid:**
1. **Pre-create indices before deploy.** In `firestore.indexes.json`, add composite indices for (labId, status, dateField).
2. **Test in emulator first:** `firebase emulators:start --only firestore`. Run aggregation query locally.
3. **Index checklist:** If a query has `WHERE a AND b AND ORDER BY c` where a, b, c are distinct fields, declare composite index.

**Warning signs:** "Analytics aggregation failed to run last hour"; Firestore logs show "index not created" error.

---

### Pitfall 4: Unsigned Signed URLs or Expiry Miscalculation

**What goes wrong:** Signed URL generated with 24-hour expiry, but calculated as `Date.now() + 24` (milliseconds instead of 24 hours). URL expires in 24ms. User can't download.

**Why it happens:** JavaScript date arithmetic is error-prone. Easy to confuse ms / seconds / hours.

**How to avoid:** Use constants. Always specify expiry in clear units:
```typescript
const SIGNED_URL_EXPIRY_MS = 7 * 24 * 60 * 60 * 1000; // 7 days in milliseconds
const [signedUrl] = await file.getSignedUrl({
  expires: Date.now() + SIGNED_URL_EXPIRY_MS,
});
```

**Warning signs:** "Download link doesn't work" reported within hours of export completion.

---

### Pitfall 5: RDC 978 Compliance Gap: Not Logging Export Initiation

**What goes wrong:** Export initiated but no audit record. Auditor reviews `/auditLogs` and finds zero export activity. Compliance failure: "system doesn't log data exports."

**Why it happens:** Export callable focused on job orchestration; audit logging treated as optional "nice-to-have."

**How to avoid:** Every `initiateExport` callable writes to `/labs/{labId}/auditLogs/` with operatorId, timestamp, format, date range. Include in batch.commit() for atomicity.

**Warning signs:** Audit trail review finds no export records despite users confirming downloads.

## Code Examples

### Example 1: Mobile Hook — Offline-Aware Cache

```typescript
// src/mobile/hooks/useOfflineCache.ts
import { useEffect, useState } from 'react';
import { Firestore, collection, onSnapshot } from 'firebase/firestore';
import { useNetInfo } from '@react-native-community/netinfo';
import AsyncStorage from '@react-native-async-storage/async-storage';

export function useOfflineCache<T>(
  db: Firestore,
  path: string
) {
  const [data, setData] = useState<T | null>(null);
  const [isLoading, setIsLoading] = useState(true);
  const [isOnline, setIsOnline] = useState(true);
  const netInfo = useNetInfo();

  useEffect(() => {
    setIsOnline(netInfo.isConnected ?? true);
  }, [netInfo.isConnected]);

  useEffect(() => {
    if (!isOnline) {
      // Load from cache
      AsyncStorage.getItem(`cache:${path}`).then((cached) => {
        if (cached) {
          try {
            setData(JSON.parse(cached));
          } catch (e) {
            console.error('Cache parse error:', e);
          }
        }
        setIsLoading(false);
      });
      return;
    }

    // Online: subscribe to Firestore
    setIsLoading(true);
    const colRef = collection(db, ...path.split('/'));
    const unsubscribe = onSnapshot(
      colRef,
      (snap) => {
        const items = snap.docs.map((d) => d.data());
        setData(items as T);
        setIsLoading(false);

        // Cache result
        AsyncStorage.setItem(`cache:${path}`, JSON.stringify(items));
      },
      (err) => {
        console.error('Firestore error:', err);
        // Fallback to cache
        AsyncStorage.getItem(`cache:${path}`).then((cached) => {
          if (cached) setData(JSON.parse(cached));
          setIsLoading(false);
        });
      }
    );

    return unsubscribe;
  }, [isOnline, path, db]);

  return { data, isLoading, isOnline };
}
```

import { create } from 'zustand';
