# Phase 3.1 Plan 03: Export Foundation Summary

**Building async export infrastructure for XLSX/PDF/CSV compliance reports with 7-day signed URL access.**

---

## What Was Built

### 1. Export Type System
**File:** `src/features/export/types/index.ts`

Shared TypeScript definitions across client and Cloud Functions:
- `ExportFormat` — format union type (xlsx | pdf | csv)
- `ExportJobStatus` — lifecycle states (queued → processing → completed/failed)
- `ExportRequest` — client request contract (labId, format, startDate, endDate, operatorId)
- `ExportInitiateResponse` — immediate response from callable (jobId, status, estimatedMinutes)
- `ExportJob` — Firestore document schema with full audit metadata (operator, timestamps, file size, signed URL)
- `ExportJobMessage` — Pub/Sub message envelope (jobId, labId, format)
- `ExportWorkerResult` — worker return type (status, downloadUrl, expiresAt, processingDurationMs)

All types enforce multi-tenant isolation via `labId` field and soft-delete awareness.

### 2. Cloud Callable: initiateExport
**File:** `functions/src/modules/export/initiateExport.ts` (125 lines)

Validates and initiates export jobs:
- **Validation:** Auth token, format (xlsx|pdf|csv), date range (must be ordered, ≤365 days)
- **Lab check:** Verifies lab exists in Firestore (future: operator access control)
- **Job creation:** Creates document at `/labs/{labId}/export-jobs/{jobId}` with `status='queued'`
- **Pub/Sub publish:** Sends message to `exports` topic with job metadata
- **Response:** Returns `{ jobId, status: 'queued', estimatedMinutes: 2-5, createdAt }`
- **Error handling:** HttpsError for validation failures, internal errors logged to Cloud Logging

Region: `southamerica-east1` (same as web app)

### 3. XLSX Generator
**File:** `functions/src/modules/export/generators/xlsxGenerator.ts` (73 lines)

SheetJS wrapper for generating Excel files from CIQ run data:
- **Async generation:** `generateXlsx(data: Record<string, any>[])` → Promise<Buffer>
- **Flattening:** Converts nested objects to flat columns (max 50 columns per Excel spec)
- **Column widths:** Sets readable widths (ID:15, Status:12, Data:20, etc.)
- **Sample data helper:** `generateSampleData(count)` generates test data for Phase 3.1 validation
- **Error handling:** Catches and logs XLSX generation errors

Phase 3.2 will add multiple sheets, conditional formatting, and streaming API.

### 4. Pub/Sub Worker: exportWorker
**File:** `functions/src/modules/export/exportWorker.ts` (156 lines)

Cloud Function subscribing to Pub/Sub topic `exports`:
- **Listener:** `onMessagePublished` to topic `exports` with 540s timeout, 512MB memory
- **Job loading:** Reads `/labs/{labId}/export-jobs/{jobId}` from Firestore
- **Status transition:** Updates job to `status='processing'` before work begins
- **Data query:** Filters CIQ runs by date range + soft-delete safety (`deletadoEm === null`)
- **XLSX generation:** Calls `generateXlsx(data)` for Phase 3.1 (PDF/CSV Phase 3.2)
- **Cloud Storage upload:** Uploads to bucket at path `exports/{labId}/{jobId}/export.xlsx`
- **Signed URL generation:** Creates 7-day expiring read-only URL via v4 signature
- **Job completion:** Updates job with `status='completed'`, downloadUrl, expiresAt, fileSizeBytes, processingDurationMs
- **Error handling:** On failure, updates job to `status='failed'` with error message; Pub/Sub retries (5 attempts, exponential backoff)

Cache control header: `private, max-age=604800` (7 days)

### 5. Client-Side Export Service
**File:** `src/features/export/services/exportService.ts` (44 lines)

Typed service wrapper for both web and mobile:
- `initiateExport(request)` — calls Cloud Callable, returns `ExportInitiateResponse`, handles errors
- `getJobStatus()` — placeholder for Phase 3.2 (will fetch job document via Callable or Firestore)

Export syntax:
```typescript
const response = await ExportService.initiateExport({
  labId: 'lab-123',
  format: 'xlsx',
  startDate: new Date('2026-01-01'),
  endDate: new Date('2026-05-04'),
});
console.log(`Job ${response.jobId} ready in ~${response.estimatedMinutes} minutes`);
```

### 6. React Hook: useExportJob
**File:** `src/features/export/hooks/useExportJobs.ts` (135 lines)

Real-time job status subscription:
- `useExportJob(labId, jobId)` — subscribes to `/labs/{labId}/export-jobs/{jobId}` via `onSnapshot`
- Returns `{ job: ExportJob | null, loading: boolean, error: string | null }`
- Handles Firestore Timestamp → Date conversion
- Implements cleanup unsubscribe pattern

Helper functions:
- `isDownloadExpiringsome(job)` — returns true if <24 hours until expiry (warns user)
- `formatFileSize(bytes)` — "1234567" → "1.2 MB"
- `formatDuration(ms)` — "125000" → "2m 5s"

Usage pattern for UI:
```tsx
const { job, loading, error } = useExportJob(labId, jobId);
if (job?.status === 'completed') {
  return <DownloadButton url={job.downloadUrl} />;
}
```

## Job Lifecycle (End-to-End Flow)

```
1. Client calls ExportService.initiateExport()
   ↓
2. Cloud Callable validates + creates job in Firestore (status='queued')
   ↓
3. Callable publishes to Pub/Sub topic 'exports'
   ↓
4. exportWorker subscribes, receives message
   ↓
5. Worker queries CIQ runs for date range
   ↓
6. Worker generates XLSX via SheetJS
   ↓
7. Worker uploads to Cloud Storage at exports/{labId}/{jobId}/export.xlsx
   ↓
8. Worker generates signed URL (7-day expiry)
   ↓
9. Worker updates job: status='completed', downloadUrl, expiresAt, fileSizeBytes
   ↓
10. Client polls/subscribes via useExportJob hook
    ↓
11. Client downloads via signedUrl when job.status === 'completed'
```

**Job document path:** `/labs/{labId}/export-jobs/{jobId}`

**Cloud Storage path:** `exports/{labId}/{jobId}/export.xlsx`

**Signed URL expiry:** 7 days from generation (recommend user download immediately)

## Pub/Sub Infrastructure

**Topic:** `exports`
- Created via: `gcloud pubsub topics create exports --project=hmatologia2`
- Message format: JSON `{ jobId, labId, format }`

**Subscription:** `exports-worker`
- Created via: `gcloud pubsub subscriptions create exports-worker --topic=exports --push-endpoint=...`
- Push endpoint: `https://southamerica-east1-hmatologia2.cloudfunctions.net/exportWorker`
- Ack timeout: 10s (message must be processed or will retry)
- Max retries: 5 (exponential backoff: 1s, 2s, 4s, 8s, 16s)

**Error handling:**
- Failed processing throws error → Pub/Sub retries
- Max retries reached → message goes to dead-letter queue (optional Phase 3.2)
- Worker logs all errors to Cloud Logging for monitoring

## Multi-Tenant Isolation

All queries and documents enforce `labId`:
- Job creation: validates lab exists before creating job document
- Job query: `/labs/{labId}/export-jobs/{jobId}` path ensures isolation
- CIQ runs query: filters by `labId` explicitly
- Soft-delete filter: `deletadoEm === null` prevents exporting deleted runs
- Audit trail: `operatorId` captured from auth token (who initiated export)

Future Firestore rules will enforce read/write permissions at document level.

## Security & Compliance

### T-03.1-E01: Spoofing (mitigated)
- Cloud Callable validates `request.auth.uid`
- `operatorId` captured for audit trail
- Lab access verified before job creation

### T-03.1-E02: Tampering (mitigated)
- Job status updates only via Cloud Function (server-side)
- Firestore rules (Phase 3.2) prevent client-side tampering
- Signed URL is read-only (no write access)

### T-03.1-E03: Information Disclosure (mitigated)
- Signed URLs expire in 7 days (limited access window)
- Access logged in Cloud Audit Logs (Phase 3.2)
- User warned if download expires within 24h

### T-03.1-E04: Denial of Service (mitigated)
- Firebase default rate limiting on callables
- Pub/Sub max retries (5) prevent infinite loops
- Cloud Function timeout (540s) prevents runaway jobs

### T-03.1-E05: Cross-lab Leakage (mitigated)
- Export query filters by `labId` from job document
- Cloud Function cannot access other labs
- `deletadoEm` filter ensures soft-deleted runs not exported

### T-03.1-E06: File Overwrite (mitigated)
- `jobId` is unique (nanoid), prevents collisions
- Cloud Storage path includes both labId and jobId
- Concurrent exports from same lab use different jobIds
