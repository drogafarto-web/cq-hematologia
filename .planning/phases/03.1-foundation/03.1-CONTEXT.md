## Architecture Decisions (Locked)

### 1. Mobile Stack: React Native CLI (Native)

**Decision:** React Native with Expo-less native CLI. Full control over iOS/Android build pipeline.

**Why:**
- World-class systems require control, not vendor-managed platforms.
- No Expo limitations (app size, native module restrictions, push notifications).
- Investment in CI/CD (GitHub Actions → TestFlight/Play Store) is one-time, pays off for years.
- Biometric auth, custom modules, performance tuning — native control required.

**Constraints:**
- Requires iOS developer account (Apple) + Android Play Console account (Google).
- CI/CD pipeline (GitHub Actions) must handle signing, provisioning profiles, build versioning.
- Phase 3.1 deliverable: Project initialized, auth skeleton working, CI pipeline green light.
- Phase 3.3 gate: TestFlight + Play Store beta links ready.

### 2. Analytics: Scheduled Aggregation (Cloud Function, Hourly)

**Decision:** Pre-computed analytics data via Cloud Function running hourly. Firestore reads off cached aggregates, not live docs.

**Why:**
- Predictable costs (large labs with 100k+ docs would spike under real-time listeners).
- RDC 978 compliance: aggregated data is versionable, reproducible, auditable.
- Hourly lag is acceptable for compliance dashboards (not a real-time monitoring system).
- Foundation for AI predictions, anomaly detection (Phase 3 backlog).

**Architecture:**
```
CloudSchedule (hourly) 
  → Cloud Function: analyticsAggregation()
    → Query Firestore (CIQ runs, NCs, trainings)
    → Compute KPIs (turnaround, retrabalho%, compliance%)
    → Write to /labs/{labId}/analytics-aggregates/{hourTimestamp}
    → Write metadata to /labs/{labId}/analytics-meta (latest, freshness)
  → Client reads from aggregates + subscribes to analytics-meta for refresh trigger
```

**Caching layer:**
- Zustand for in-memory dashboard state.
- Redis cache (optional Phase 3.2, not Phase 3.1) for multi-lab queries.

**Manual refresh:**
- UI button: "Atualizar agora" → callable that triggers CF immediately, returns latest aggregate.

### 3. Export: Server-Side Cloud Function + Cloud Storage

**Decision:** All exports processed by Cloud Function, results stored in Cloud Storage with 7-day signed URLs.

**Why:**
- Scales to any dataset size (no browser memory limits).
- Async model is honest UX: user initiates, checks status, downloads via email/link.
- Each export is auditio: CF callable logs operation, result stored with lab + operator + timestamp.
- Timeout-safe: CF can run up to 540s for large labs (>100k docs).

**Architecture:**
```
Client UI: ExportWizard
  → Select format (XLSX CIQ | XLSX NC | PDF Compliance Report | CSV Audit Log)
  → Select date range
  → Submit → Cloud Callable: initiateExport()
    → Store job record in /labs/{labId}/export-jobs/{jobId}
    → Enqueue to Pub/Sub topic
    → Return jobId to UI
  
BackgroundWorker (Pub/Sub subscriber)
  → Download job spec
  → Generate file (SheetJS for XLSX, puppeteer for PDF, plain CSV for logs)
  → Upload to Cloud Storage /exports/{labId}/{jobId}/{filename}
  → Update job record: status=done, gsPath, signedUrl (7d expiry), completedAt
  → (Optional Phase 3.2) Send email with download link

Client polls job status or listens to job record
  → Download via signedUrl when ready
```

**Constraints:**
- XLSX generation: SheetJS (community edition) — no complex pivot tables.
- PDF generation: Puppeteer (Node.js in Cloud Function) or server-side template (Pug/EJS).
- Large dataset stress test: 100k+ runs must complete in <5min.

### 4. Mobile Offline: No Offline Writes (Read-Only Offline)

**Decision:** Phase 3.1 supports offline reads (cached data), but requires connectivity for writes.

**Why:**
- Phase 3.1 is foundation — avoid merge conflict complexity.
- CIQ workflows (submit run, sign training) are typically done on-site where Wi-Fi is available.
- Read-only offline covers 80% of use case (check past results, review POPs).
- Phase 3.2 can add smart offline queue + server-wins conflict resolution if needed.

**Implementation:**
- Service Worker caches API responses + static assets.
- Zustand + localStorage for session state.
- Firestore `onSnapshot` unsubscribes and caches on disconnect.
- Write buttons show "offline" state, remain disabled.

**Canonical refs:**
- `.planning/phases/03.1-foundation/mobile-offline-cache-strategy.md` (TBD) — Service Worker config, cache keys
