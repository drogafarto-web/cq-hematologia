# Phase 2 Batch 1: Sistema de Qualidade

**Scope:** 3 modules, >80% test coverage, production-ready deployment  
**Inputs:** ADR 0004 (POP spine live), ADR 0003 (NC spine live), ADR 0005 (HMAC live)  
**Outputs:** 3 modules live in prod, all UIs tested, rules deployed

---

## Tasks

### Task 1: POPs Module UI Implementation

**Duration:** 2-3 weeks  
**Type:** auto

Build complete POP management module:

1. Create `src/features/sgq/pops/types.ts`
   - POP, POPVersion, POPTraining interfaces
   - POPStatus enum (draft, vigente, expirada, revogada)

2. Create `src/features/sgq/pops/popsService.ts`
   - subscribePOPs(labId) - realtime list
   - createPOP(labId, input) - calls Cloud Function
   - createPOPVersion(labId, popId) - immutable versioning
   - assinaturaRT(labId, popId, versao) - RT approval
   - assignTraining(labId, popId, operadorId) - training link

3. Create UI components:
   - POPsList.tsx - table view, status, versions
   - CreatePOPModal.tsx - create + version UI
   - TrainingAssignmentUI.tsx - operator training matrix

4. Wire into CIQ modules:
   - Add popId + popVersaoId to run schema
   - Update criarRun callables to capture popId at write time
   - Validation: before saving run, check operador has required training

5. Create `src/features/sgq/pops/__tests__/pops.test.ts`
   - Test POP create → version → training workflow
   - Test version immutability
   - Test training validation gate
   - Target: 80%+ coverage

**Acceptance Criteria:**

- ✅ npm run build succeeds
- ✅ Create POP v1.0 → v1.1 → RT sign → training assigned
- ✅ Operator without training blocked from running CIQ
- ✅ All tests pass, coverage >80%
- ✅ No TypeScript errors

**Commit:** "Batch 1 Task 1: POPs Module UI complete"

### Task 2: Não-Conformidade + CAPA Module UI

**Duration:** 2-3 weeks  
**Type:** auto

Build complete NC lifecycle + CAPA workflow:

1. Create `src/features/sgq/naoConformidade/types.ts`
   - NaoConformidade interface (titulo, descricao, severidade, status, origem)
   - Severity enum (leve, moderada, grave, crítica)
   - Status enum (aberta, investigacao, acao, eficacia, fechada)
   - CAPA workflow state machine
   - Acao interface (investigacao text, acao text, dataEficacia)

2. Create `src/features/sgq/naoConformidade/ncService.ts`
   - subscribeNCs(labId) - realtime filtered by status
   - openNC(labId, input) - calls Cloud Function
   - updateNCStatus(labId, ncId, status) - state transitions
   - investigarNC, executarAcao, verificarEficacia - CAPA steps
   - Integration: getCriticalNCs(labId) returns severity=crítica for gate check

3. Create UI components:
   - NCList.tsx - filterable by status, severity
   - NCDetailModal.tsx - full lifecycle view
   - CAPAWorkflow.tsx - visual state machine (investigacao → acao → eficacia)
   - SeverityGateUI.tsx - shows blocking status

4. Wire into all 6 modules:
   - Before any create/update, check getCriticalNCs()
   - If severity=crítica exists for module, throw error with reason
   - UI shows banner: "Operações bloqueadas: NC crítica de equipamento aberta"

5. Create `src/features/sgq/naoConformidade/__tests__/nc.test.ts`
   - Test NC open → status transitions
   - Test blocking gate (grave/crítica)
   - Test CAPA workflow progression
   - Test severity escalation
   - Target: 80%+ coverage

**Acceptance Criteria:**

- ✅ Open grave NC → all 6 modules show blocking banner
- ✅ Cannot create run/insumo/etc while NC crítica open
- ✅ CAPA workflow: investigacao → acao → eficacia → fechada progresses
- ✅ All tests pass, coverage >80%
- ✅ Build succeeds

**Commit:** "Batch 1 Task 2: NC + CAPA Module UI complete"

### Task 3: Auditoria Interna Module UI

**Duration:** 1-2 weeks  
**Type:** auto

Build audit management with auto-NC creation:

1. Create `src/features/sgq/auditoria/types.ts`
   - Auditoria interface (titulo, data, responsavel, checklist)
   - Achado interface (item, constatacao, severidade, origem='auditoria')
   - Auditoria status (planejada, em_execucao, finalizando, fechada)

2. Create `src/features/sgq/auditoria/auditoriaService.ts`
   - subscribeAuditorias(labId)
   - criarAuditoria(labId, input) - calls CF
   - registrarAchado(labId, auditoriaId, achado) - auto-opens NC
   - aprovarPlanoAcao(labId, auditoriaId) - closes audit + verifies all NCs

3. Create UI components:
   - AuditoriaChecklist.tsx - template + item tracking
   - FindingsForm.tsx - constatacao entry (auto-creates NC on save)
   - PlanoAcaoUI.tsx - links to NCs, tracks CAPA progress
   - AuditoriaList.tsx - schedule + status view

4. Auto-NC logic:
   - When registrarAchado called, create NC with origem='auditoria', origemId=auditoriaId
   - Severity of NC = severity of achado
   - Link bidirectional: achado.ncId ↔ nc.origem='auditoria'

5. Create `src/features/sgq/auditoria/__tests__/auditoria.test.ts`
   - Test audit create → checklist → findings entry → auto-NC
   - Test severity mapping (achado grave → NC grave)
   - Test bidirectional linking
   - Target: 80%+ coverage

**Acceptance Criteria:**

- ✅ Create audit → add findings (grave) → auto-NC created with origem='auditoria'
- ✅ NC blocks operations (linked back to audit)
- ✅ Closing audit requires all linked NCs closed
- ✅ All tests pass, coverage >80%
- ✅ Build succeeds

**Commit:** "Batch 1 Task 3: Auditoria Interna Module UI complete"

### Task 4: Firestore Rules Deployment + Integration Tests

**Duration:** 1 week  
**Type:** auto

Apply rules, run cross-module integration tests:

1. Create/update Firestore rules patches:
   - `/labs/{labId}/pops/{popId}` - read/write by RT only (write via callable)
   - `/labs/{labId}/pops/{popId}/versoes/{versaoId}` - immutable after creation
   - `/labs/{labId}/naoConformidades/{ncId}` - read/write via callable, HMAC required
   - `/labs/{labId}/auditorias/{auditId}` - read/write via callable

2. Deploy rules:

   ```bash
   firebase deploy --only firestore:rules
   ```

3. Run integration smoke tests:
   - **Scenario A:** Create POP → assign training → run CIQ with popId → verify
   - **Scenario B:** Open grave NC → try to create run → blocked → investigate → close → unblocked
   - **Scenario C:** Create audit → add grave finding → auto-NC → blocks → CAPA workflow → close all

4. Verify no permission errors:
   - Test as operator (read NC, create runs)
   - Test as RT (approve POP, close NC)
   - Test as admin (view all)

**Acceptance Criteria:**

- ✅ firebase deploy succeeds
- ✅ All 3 smoke scenarios pass end-to-end
- ✅ No permission denied errors
- ✅ Blocking gates work (grave NC, missing training, etc)

**Commit:** "Batch 1 Task 4: Rules deployed + integration tests pass"
