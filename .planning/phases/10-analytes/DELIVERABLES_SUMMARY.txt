KEY ARCHITECTURAL DECISIONS
================================================================================

1. Multi-Equipment Validation
   • 1 control lot → N equipmentIds (array in ControlMaterial doc)
   • Validation via Firestore rules: equipmentIds must reference valid equipment
   • Lot archived when equipment retires (soft-delete)

2. Westgard Engine (Server-Only)
   • Cloud Function recordRunBioquimica computes violations
   • Client receives immutable result with status + violations array
   • Prevents operator manipulation of QC decisions (Threat T5)

3. Immutable Audit Trail
   • Runs become read-only after signature computed
   • Compliance overrides tracked in separate versioned collection
   • TraceabilityEvent append-only log with SHA-256 signature
   • ChainHash encadeamento validates historical integrity

4. Equipment Lifecycle
   • Status: ativo → manutencao → aposentado
   • Soft-delete only; retencaoAte = aposentadoEm + 5 years
   • EquipamentoSnapshot frozen in every run (survives master doc deletion)

5. Firestore Schema (Multi-Tenant)
   • All docs carry labId in path + payload (defense-in-depth)
   • equipmentIds as array allows cross-device queries
   • Separate stats/ & reports/ collections for future aggregation

WESTGARD IMPLEMENTATION SUMMARY
================================================================================

Rule                Severity    Trigger                          Client Impact
────────────────────────────────────────────────────────────────────────────
1-2s (1 result)     warn        |deviation| > 2σ from mean      Yellow alert
1-3s (1 result)     reject      |deviation| > 3σ from mean      Block "oficial"
2-2s (2 consecutive) reject     Both > 2σ, same side            Block "oficial"
R-4s (range)        reject      Range between 2 runs > 4σ       Block "oficial"

Extended Rules (Phase 11, disabled by default):
4-1s, 10x, 6T, 6X → all severity: reject

OUTSTANDING DECISIONS (For CTO Review)
================================================================================

1. Bula PDF parsing library
   Option A: Gemini Vision (current plan) — cloud-dependent, slow, accurate OCR
   Option B: PyPDF2 + local ML — no cloud dependency, faster, less accurate
   Decision needed for Phase 10.1 (applyBulaToLot implementation)

2. Method comparison threshold (acceptable bias)
   Option A: Fixed 10% bias tolerance (current)
   Option B: Dynamic per-analyte (e.g., glucose ±5%, lipids ±15%)
   Decision impacts how many instrument pairs are "comparable"

3. Extended Westgard rules enablement
   Option A: Disabled by default in v1.4, opt-in per analyte via UI
   Option B: Enabled by default in v1.4 with opt-out override
   Decision affects clinical oversight during rollout

4. Lot-equipment flexibility
   Current: Once a lot is used, equipmentIds list is frozen
   Issue: Cannot add new equipment mid-life (e.g., new analyzer installed)
   Solution: Allow supervised list mutations with TraceabilityEvent audit
