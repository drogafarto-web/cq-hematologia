**Why membership, not module claim?**

- Both analytics and export-jobs are feature-discovery modules (Phase 3.x) that audit cross-lab usage. They're NOT gated by `hasModuleAccess()` claims like CIQ modules.
- Membership is the only applicable gate: if you're not a member of the lab, you have no business reading its analytics.
- Export jobs are tenant-scoped (each lab's exports are private to that lab). Membership is the correct boundary.

**Backwards compatibility:**

- Existing docs in analytics/export-jobs do NOT require any data migration (no new `memberIds` field needed).
- Users who were reading via `isAuthenticated()` and are also lab members will continue to read (no regression).
- Users reading cross-tenant (audit scenario) will now be denied — this is the intended hardening.

        labId,
        lastRefreshAt: serverTimestamp(),

## Trust Boundaries

| Boundary                 | Description                                                                 |
| ------------------------ | --------------------------------------------------------------------------- |
| Client → Analytics API   | Untrusted client sends read request for /labs/{labId}/analytics/\*\*        |
| Client → Export-Jobs API | Untrusted client sends read request for /labs/{labId}/export-jobs/\*\*      |
| Lab membership           | Request.auth.uid must exist in /labs/{labId}/members/{uid} with active=true |

## STRIDE Threat Register

| Threat ID | Category               | Component                         | Disposition  | Mitigation Plan                                                                                                                               |
| --------- | ---------------------- | --------------------------------- | ------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| T-04-01   | Elevation of Privilege | /analytics read gate              | **mitigate** | Validate request.auth.uid in /labs/{labId}/members/ before allowing read. CLEAN-01 implements this.                                           |
| T-04-02   | Information Disclosure | /export-jobs read gate            | **mitigate** | Replace isAuthenticated() (line 1363) with isActiveMemberOfLab(labId). Blocks cross-tenant export job visibility.                             |
| T-04-03   | Tampering              | Analytics/export-jobs write       | **mitigate** | Both collections have `allow create: if false` + `allow update: if false` — writes only via Admin SDK (Cloud Functions). Client cannot forge. |
| T-04-04   | Repudiation            | Membership bypass via stale token | **mitigate** | isActiveMemberOfLab() checks live Firestore state at request time, not cached claims. Token revocation is respected immediately.              |

**Audit Trail Compliance:**

- DICQ 4.4 (controle de acesso): Firestore rules enforce multi-tenant isolation at the collection boundary.
- RDC 978 5.3 (acesso autorizado): Only active members of a lab can read that lab's analytics/exports.
- Audit verdict: After CLEAN-01, auditor reviewing rules will see no bypass markers in sensitive collections.
