  <name>Task 2: Implement NotivisaClient Class (Sandbox + Prod Mode)</name>
  <files>functions/src/modules/notivisa/NotivisaClient.ts</files>
  <action>
Create NotivisaClient class with sandbox/production mode toggle. Implementation spec:

1. **Constructor**:
   ```typescript
   constructor(
     mode: 'sandbox' | 'production',
     apiKey: string,  // from Firebase Secrets
     firestore: admin.firestore.Firestore  // for audit logging
   )
   ```
   - Set baseUrl = "https://notivisa-sandbox.anvisa.gov.br" (sandbox) or production URL
   - Store apiKey securely (never log)
   - Initialize retry queue with RetryConfig defaults

2. **submitPayload(payload: NotivisaPayload): Promise<NotivisaResponse>**:
   - Validate payload structure (zod schema)
   - POST to `/api/v1/events` with Bearer token
   - Request timeout: 10s (RetryConfig.timeoutMs)
   - Retry on network error, 5xx, or timeout (do NOT retry on 4xx validation errors)
   - Exponential backoff: 1m → 5m → 15m
   - Log attempt #, response code, error to audit service
   - Return response or throw NotivisaError
   - **Security**: CPF never logged; log only CPF hash

3. **submitBatch(payloads: NotivisaPayload[]): Promise<NotivisaResponse[]>**:
   - Map over payloads, call submitPayload for each
   - Aggregate responses + errors
   - Throttle: max 5 concurrent requests (prevent API flooding)
   - Return [response | error] array
   - Total batch time <10s for 5 events (typical)

4. **validateSignature(payload: NotivisaPayload, expectedSignature: string): boolean**:
   - Reconstruct HMAC-SHA256(JSON.stringify(payload), apiKey)
   - Constant-time comparison vs expectedSignature
   - Return boolean (never throw on signature mismatch—let caller decide)

5. **getHealth(): Promise<{ status: 'healthy' | 'unhealthy', latencyMs: number }>**:
   - Simple HEAD request to NOTIVISA base URL
   - Measure round-trip time
   - Return status + latency
   - Used by processQueue to decide if processing should proceed

6. **Private method _retryWithBackoff(fn, config): Promise<T>**:
   - Implements exponential backoff logic
   - On attempt N: await backoffMs[N], then call fn()
   - Catch + log error, check if retryable (network/5xx/timeout)
   - Max 3 attempts total
   - Return result or throw on final failure

7. **Error handling**:
   - Catch network errors → wrap in NotivisaError(API_UNREACHABLE)
   - Catch timeout → NotivisaError(TIMEOUT)
   - Catch 4xx response → log + return response (caller decides)
   - Catch 5xx → retryable
   - Log all attempts (attempt #, backoff time, error type)

Unit test coverage: 12+ tests covering happy path, retries, errors, signature validation, timeout, rate limiting.

  <files>functions/src/shared/notivisa.ts</files>
  <action>
Create shared utility module for NOTIVISA payload generation + signature validation. Exports:

1. **generateNotivisaPayload(laudo: Laudo, cpfHash: string, operatorId: string): NotivisaPayload**:
   - Input: Laudo object from firestore (must have: id, nomeExame, dataPublicacao, results[])
   - Extract: lab CNPJ from admin.firestore().collection('labs').doc(labId).get() (one-time call per laudo)
   - Set result_status:
     - "critical" if any result has flagCritico=true
     - "abnormal" if any result flags outside reference range
     - "normal" otherwise
   - Generate payload with Portaria 204 format
   - Do NOT include plaintext CPF in payload (cpfHash only)
   - Set sandbox_mode=true in v1.4
   - Return complete NotivisaPayload object

2. **signPayload(payload: NotivisaPayload, secretKey: string): string**:
   - Serialize payload to JSON (deterministic order: alphabetical keys)
   - HMAC-SHA256(jsonString, secretKey)
   - Return hex-encoded signature
   - **Important**: Use secretKey from Firebase Secrets, never hardcoded

3. **hashCPF(cpf: string, salt: string = ''): string**:
   - Remove formatting: "123.456.789-00" → "12345678900"
   - HMAC-SHA256(cpf, salt)
   - Return hex
   - Used to store CPF hash in queue (plaintext never in DB)

4. **validateWebhookSignature(payload: any, signature: string, secretKey: string): boolean**:
   - Reconstruct signature = signPayload(payload, secretKey)
   - Constant-time comparison vs provided signature
   - Return boolean (caller decides whether to reject)
   - Used by webhook receiver (Phase 4-03) to validate NOTIVISA callbacks

5. **enrichQueueEvent(event: QueueEvent, laudo: Laudo, operatorId: string): QueueEvent**:
   - Called when enqueuing event to notivisa-outbox
   - Add operator_signature: { hash, operatorId, ts }
   - Add createdAt, updatedAt timestamps
   - Return enriched event
   - Must be called server-side only (Cloud Function)

6. **formatErrorForQueue(error: Error | NotivisaError, httpStatus?: number): string**:
   - Truncate error message to 500 chars
   - Include HTTP status code if present
   - Return formatted string for QueueEvent.lastError field

All functions must:
- Never log plaintext CPF
- Use admin SDK (not client SDK) for firestore reads
- Include JSDoc comments with examples
- Be side-effect-free (pure functions where possible)

## Threat Model

### Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Cloud Function | Laudo publication trigger (client initiates, function enqueues) |
| Cloud Function → NOTIVISA API | Outbound HTTP requests with Bearer token (encrypted TLS) |
| NOTIVISA API → Webhook Receiver | Inbound callbacks, must validate HMAC signature |
| Firestore Rules → notivisa-outbox | Immutable queue, only admin SDK writes |

### STRIDE Threat Register

| ID | Category | Component | Disposition | Mitigation |
|----|----------|-----------|-------------|-----------|
| T-4-02-01 | **S**poofing | NOTIVISA API | Mitigate | Validate Bearer token in request headers, store API key in Secrets Manager (not code) |
| T-4-02-02 | **T**ampering | notivisa-outbox events | Mitigate | Firestore Rules enforce append-only, forbid updates to payload/signature fields |
| T-4-02-03 | **R**epudiation | Audit trail | Mitigate | registerAuditEntry logs all NOTIVISA actions (operator_signature immutable) per DICQ 4.4 |
| T-4-02-04 | **I**nformation Disclosure | CPF in queue | Mitigate | Hash CPF before writing to queue, plaintext only in API payload during transmission (TLS) |
| T-4-02-05 | **I**nformation Disclosure | API key in logs | Mitigate | Never log API key; log only "sandbox_mode=true" + error type + HTTP status |
| T-4-02-06 | **D**enial of Service | Retry loop | Accept | Max 3 attempts + exponential backoff; if API down, queue enters circuit-breaker state (5 min cooldown) |
| T-4-02-07 | **D**enial of Service | Webhook replay | Mitigate | HMAC signature + timestamp validation in webhook receiver (Phase 4-03) |
| T-4-02-08 | **E**levation of Privilege | Client direct queue write | Mitigate | Firestore Rules forbid client writes; only Cloud Functions via admin SDK |
