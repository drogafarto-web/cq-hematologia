# Task 04-01 — Patient Portal — Laudo Access & Download

**Objective:** Build read-only patient portal to access published laudos + download PDF + brand per lab (REQ-408: Paciente Portal Access + REQ-409: Laudo Download).

**Owner:** Stream B (Frontend engineer, patient-facing)
**Duration:** 2 weeks (Phase 4, Wave 2, Week 4)
**Compliance:** RDC 978 Art. 167 (patient notification), LGPD Art. 18 (data access rights)
**Wave:** 2 (CAPA + Portals parallel)

---

## Scope

### 1. Patient Authentication (Email-Link, No SSO in v1.4)

**Feature:** Patient accesses portal via email link sent post-laudo publication (via Phase 6 notifications).

**Requirements:**
- One-time email link (`token` + `tokenExpiry` stored in Firestore `laudos/{labId}/{laudoId}`)
- Token valid for 7 days, single-use (invalidated after first click)
- No password, no phone verification (per v1.4 DL-3 decision: patient consent via LGPD + audit trail)
- Email link format: `https://hmatologia2.web.app/portal/[labId]/auth?token=[uuid]`
- Upon click: create session token (JWT stored in localStorage), redirect to `/portal/[labId]/laudos`
- Session valid for 30 days or until logout
- Logout clears localStorage token + revokes in Firestore `patientSessions` collection

**Auth guards:**
- Route `/portal/[labId]/*` requires valid `patientToken` (JWT in localStorage)
- No requirement to be in `members` collection (patient is external to org)
- Rules allow read-only access to own laudo + portal config, no cross-patient access

**Edge cases:**
- Token already used: return 400 "Link already consumed"
- Token expired: return 403 "Link expired, request new access"
- Patient session expired: redirect to portal login screen (invalid token message)
- Malformed token: return 400 "Invalid link"

**Non-scope (v1.4.1):**
- SSO (Google Sign-In, AWS Cognito)
- SAML / multi-factor auth
- Patient account recovery / password reset

### 4. Firestore Rules & Security

**Rules additions to Phase 3 (schema extensions rules block for `portal-configuracao` + `laudos` read-only for patient):**

```firestore-rules
// Portal configuration — read-only for patients, write-only lab admin
match /labs/{labId}/portal-configuracao/{doc=**} {
  allow read: if
    request.auth != null &&
    (isMemberOfLab(labId) || isPatientWithValidToken(labId));
  allow write: if
    request.auth != null &&
    isAdminOrOwner(labId);
}

// Laudos — patient read filter: only published + visibilidadePaciente=true
match /labs/{labId}/laudos/{laudoId} {
  allow read: if
    request.auth != null &&
    (
      isMemberOfLab(labId) ||
      (isPatientWithValidToken(labId) && resource.data.status == 'published' && resource.data.visibilidadePaciente == true)
    );
}

// Patient sessions — create on auth link click, revoke on logout
match /labs/{labId}/patientSessions/{sessionId} {
  allow create: if
    request.auth == null &&  // unauthenticated token validation
    request.resource.data.tokenExpiry > now &&
    request.resource.data.cpfHash.size() == 64;  // HMAC-SHA256
  allow read: if
    request.auth != null && request.auth.uid == resource.data.operatorId;
  allow delete: if
    request.auth != null && request.auth.uid == resource.data.operatorId;
}
```

**RLS strategy:**
- `patientSessions` — created via Cloud Function callable (`POST /validatePatientToken`), no direct client write
- Validation: token hash (HMAC-SHA256) + CPF hash + expiry timestamp
- Audit logging: log reads of patient portal (flag `source: 'patient-portal'` in audit trail)

### 5. Cloud Function Callables (wired in Phase 4-03)

**1. `validatePatientToken(token: string, labId: string) → { cpf, firstName, sessionToken, expiresAt }`**
- Lookup token in `laudos` collection (field `patientAccessTokens[]`)
- Validate expiry, hash, single-use flag
- Generate JWT session token (signed with `HCQ_SESSION_SECRET`)
- Mark token as used (set `tokenUsedAt: now`)
- Create entry in `patientSessions` collection
- Return session token + patient CPF (hashed in response to client)
- Log: `{ action: 'patient-auth', labId, tokenHash: hash(token), cpf: hash(cpf), success: true }`

**2. `downloadLaudoPDF(laudoId: string, labId: string) → { pdfUrl }`**
- Validate caller is authenticated patient (has valid `patientToken` in request context)
- Fetch laudo doc, verify `status === 'published'` + `visibilidadePaciente === true` + CPF matches session
- Generate PDF (reuse existing Puppeteer callable from Phase 3-04)
- Upload to Firebase Storage at `patients/{labId}/{laudoId}/laudo.pdf` (temp URL valid 1h)
- Log: `{ action: 'patient-pdf-download', laudoId, labId, cpf: hash }`
- Return signed download URL (expires 1h)

**3. `updatePatientNotificationPreferences(preferences: { emailNotifications: bool, language: 'pt' | 'en' }, labId: string) → { success }`**
- Validate patient is authenticated
- Store in `patientSessions` doc (ephemeral per session, not persisted long-term)
- Return success + current settings

## Technical Approach

**Rationale:**
1. **Email-link auth** (not SSO): simpler UX for patients (no account creation), complies with v1.4 DL-3 (no consent flow needed yet, LGPD compliance via audit + SGQ), aligns with GDPR Art. 15 (right of access via direct link).
2. **Session tokens (JWT)** stored locally: fast, stateless (no Firestore listener needed for every page), secure with expiry.
3. **Reuse existing PDF generation** (Puppeteer): no new dependency, leverages Phase 3 infrastructure.
4. **Lab config via Firestore doc** (not app constants): allows per-lab branding + white-label at scale (multi-tenant v1.5).
5. **Read-only portal** (v1.4): no patient edits, no CAPA closure risk. Edit workflows (Phase 5) are RT-only.

## Risk Assessment

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|-----------|
| **Email delivery fail** (SMTP) | Medium (3/10) | High (7/10) | Resend fallback, retry queue, test with staging email |
| **Token collision/guessing** | Very Low (1/10) | Critical (9/10) | UUID v4 + HMAC-SHA256, rate-limit per IP on validate endpoint |
| **Patient session hijacking** | Low (2/10) | High (8/10) | JWT signed with secret, short expiry (30d), log all accesses |
| **Laudos leak cross-patient** | Low (2/10) | Critical (9/10) | Server-side CPF filter, Rules enforce, audit log every read |
| **Portal branding CSS injection** | Low (2/10) | Medium (5/10) | DOMPurify on rich text fields, CSP header |
| **Performance regression** (slow PDF gen) | Medium (4/10) | Medium (5/10) | Cache PDF for 7 days, monitor cold-start in Cloud Logs |
