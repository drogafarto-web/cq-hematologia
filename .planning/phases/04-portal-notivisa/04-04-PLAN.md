#### Flow 4: NOTIVISA Queue Processing (Sandbox)

**Setup:**
- Lab: Riopomba
- Laudo: publish with abnormal result (triggers queue enqueue)
- NOTIVISA sandbox API: mocked (testcontainers or WireMock)

**Test steps:**

```gherkin
Given a laudo with abnormal result (WBC > ref max)
When laudo is published (status → published)
Then:
  - Firestore trigger fires
  - notivisa-outbox/events/{newId} doc created
  - status: PENDING
  - payload contains: event_id, patient_cpf_hash, analyte, value, ref_range, result_status='abnormal'
  - operator_signature: { hash, operatorId, ts } recorded

When Cloud Scheduler cron runs (next hour or manual trigger via gcloud)
Then processNotiVisaQueue function executes:
  - Fetches PENDING events (batch ≤10)
  - For each event:
    * Calls mocked NOTIVISA API with POST to sandbox URL
    * Receives 202 Accepted response
    * Updates event status → DELIVERED
    * Sets sentAt, deliveredAt timestamps

After 1 cron cycle:
  - Cloud Logs show: "[NOTIVISA] Processed 1 events: 1 sent, 0 failed"
  - Event doc shows status: DELIVERED, response_code: 202
```

**Assertions:**
- [ ] Event payload valid against Portaria 204 schema (JSON schema validation)
- [ ] Patient CPF never in plaintext in Firestore (only hash in queue)
- [ ] Signature (HMAC-SHA256) verified by test
- [ ] Cloud Logs: 0 errors, 1 INFO "Processed" summary
- [ ] No data loss (event persisted before API call, idempotent)

**Retry scenario (bonus):**

```gherkin
Given API returns 503 Service Unavailable
Then:
  - Event status → PENDING (reverted)
  - nextRetry set to now + 5min
  - attempts incremented
  - Cloud Logs: "[NOTIVISA] Retrying event [id] in 5min"

When cron runs 5 min later:
  - Event re-processed
  - (If still 503, retry again with 10min backoff)

After 5 failed attempts:
  - Event status → FAILED
  - Cloud Logs alert: "[NOTIVISA] Event [id] failed after 5 attempts"
  - ops@lab email triggered (manual intervention required)
```

### 2. Cloud Logs Monitoring Setup

**Objective:** Continuous 24h monitoring post-deploy to catch P0 errors.

**Step 1: Create Log Sink (Cloud Logging)**

```bash
gcloud logging sinks create hcq-phase4-portal \
  pubsub.googleapis.com/projects/hmatologia2/topics/hcq-phase4-alerts \
  --log-filter='
    resource.type="cloud_function" OR resource.type="api" 
    AND (
      severity="ERROR" 
      OR severity="CRITICAL"
      OR text_payload=~"patient-auth.*error"
      OR text_payload=~"patient-pdf.*error"
      OR text_payload=~"notivisa.*error"
    )
  '
```

**Step 2: Alert Policy (Monitoring)**

```yaml
# Cloud Monitoring alert
displayName: "Phase 4 — Portal Errors (24h tail)"
conditions:
  - displayName: "Cloud Function ERROR/CRITICAL"
    conditionThreshold:
      filter: |
        resource.type="cloud_function"
        AND (severity="ERROR" OR severity="CRITICAL")
        AND function_name=~"(validatePatientToken|downloadLaudoPDF|processNotiVisaQueue|.*Portal.*)"
      comparison: COMPARISON_GT
      thresholdValue: 0
      duration: 300s  # 5 min window
  - displayName: "NOTIVISA Failure Rate >10%"
    conditionThreshold:
      filter: |
        resource.type="cloud_function"
        AND function_name="processNotiVisaQueue"
        AND (text_payload=~"totalFailed|failureRate")
      comparison: COMPARISON_GT
      thresholdValue: 0.1

notificationChannels:
  - "projects/hmatologia2/notificationChannels/[email-ops]"
  - "projects/hmatologia2/notificationChannels/[slack-channel]"
```

**Step 3: Manual Monitoring Script** (bash, run every 30min during 24h window)

```bash
#!/bin/bash
# scripts/monitor-phase4-logs.sh

LAB_ID="labclin-riopomba"
LOG_FILTER='
  resource.type="cloud_function"
  AND (
    function_name=~"(validatePatientToken|downloadLaudoPDF|processNotiVisaQueue|.*Portal.*)"
    AND timestamp>="'$(date -u -d "24 hours ago" '+%Y-%m-%dT%H:%M:%SZ')'"
  )
'

echo "=== Phase 4 Cloud Logs Summary (24h) ==="
echo ""
echo "ERROR/CRITICAL count:"
gcloud logging read "$LOG_FILTER AND severity>=ERROR" \
  --limit=100 --format=json | jq 'length'

echo ""
echo "Last 10 errors:"
gcloud logging read "$LOG_FILTER AND severity>=ERROR" \
  --limit=10 --format="table(timestamp, severity, textPayload)" \
  | head -20

echo ""
echo "Function call distribution:"
gcloud logging read "$LOG_FILTER" \
  --limit=1000 --format=json \
  | jq -r '.[] | .jsonPayload.function_name // .labels.function_name' \
  | sort | uniq -c

echo ""
echo "Report generated at $(date)"
```

**Step 4: Metrics Dashboard** (Cloud Monitoring)

```sql
-- BigQuery query to export logs for analysis
SELECT
  timestamp,
  severity,
  function_name,
  jsonPayload.action as action,
  jsonPayload.error as error,
  COUNT(*) as count
FROM `hmatologia2.project_logs.cloud_function_logs`
WHERE
  timestamp >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
  AND function_name IN ('validatePatientToken', 'downloadLaudoPDF', 'processNotiVisaQueue')
GROUP BY
  timestamp, severity, function_name, action, error
ORDER BY
  timestamp DESC,
  severity DESC
```
