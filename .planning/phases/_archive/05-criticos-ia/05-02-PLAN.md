# Task 05-02 — Critical Values Notification Cascade

**Objective:** Multi-channel escalation for critical values (SMS → Email → Cloud Logs) with SLA tracking, retry logic, and audit trail per RDC 978 Art. 99 (timely notification) + RDC 978 5.3 (audit trail).

**Owner:** Stream A (Backend/Cloud Functions, async messaging)  
**Duration:** 3 days (Phase 5, Wave 3, Week 6)  
**Compliance:** RDC 978 Arts. 99 (critical notification SLA), 167 (operator alerting), 5.3 (audit trail)

## Scope

### 1. Notification Callable & Dispatcher

**Callable:** `escalateCriticalValue(labId, escalacaoId)`

**Trigger:** Cloud Function fires on `criticos-escalacoes/{labId}/escalacoes` doc creation with status = FLAGGED

**Requirements:**

```typescript
interface EscalacaoPayload {
  labId: string;
  escalacaoId: string; // reference to CriticalFlag doc
  analitoId: string;
  resultado: number;
  threshold: { lowCritical; highCritical };
  direction: 'HIGH' | 'LOW';
  severity: 'CRITICAL' | 'WARN';
  patientId: string; // CPF or patient identifier
  patientName: string;
  equipmentName: string;
  unit: string;
}

// Callable response
interface EscalationResult {
  status: 'ESCALATED' | 'PARTIAL_FAILURE' | 'FAILED';
  smsMessageId?: string;
  emailMessageId?: string;
  logId: string; // audit log entry
  notificationsSent: {
    sms: boolean;
    email: boolean;
  };
  error?: string;
  timestamp: timestamp;
}
```

**Escalation flow:**

1. **Fetch critical flag** + operator info (who recorded run)
2. **Build notification payload:**

   ```
   SMS: "⚠️ CRÍTICO: {patient} {analito} {resultado} ({direction}).
         Contato: [ops phone]. ID: {escalacaoId}"

   Email:
     Subject: "RESULTADO CRÍTICO — {lab name} — {patient}"
     Body: Full details (analito, resultado, threshold, reference range,
                        quando detectado, run ID, operator, equipment)
   ```

3. **Parallel send** (both channels, no blocking):
   - SMS to clinician + lab ops (group list in config)
   - Email to clinician + supervisor + ops
   - Log to Cloud Logs + audit trail

4. **Update CriticalFlag status:** FLAGGED → ALERTED + set alertedAt timestamp

5. **Track SLA:** timestamp critical → timestamp alerted (<5 min target per RDC 978 99)

6. **Handle failures:**
   - SMS send fails: log warning, continue email
   - Email send fails: log warning, continue
   - Both fail: retry via scheduled job (Task 05-02-B)
   - Alert ops if retry threshold exceeded (3 consecutive failures)

### 3. Retry & Failure Handling

**Firestore collection:** `criticos-escalacoes/{labId}/retry-queue/{docId}` (new)

```typescript
interface RetryQueueEntry {
  escalacaoId: string; // reference to failed critical flag
  labId: string;
  attemptCount: number; // 0, 1, 2...
  lastAttemptAt: timestamp;
  nextRetryAt: timestamp; // exponential backoff
  channels: {
    sms: { attempted: boolean; succeeded: boolean; error?: string };
    email: { attempted: boolean; succeeded: boolean; error?: string };
  };
  status: 'PENDING' | 'RESOLVED' | 'ABANDONED'; // abandoned after 3 failures
  criadoEm: timestamp;
}
```

**Retry policy:**

- Attempt 1: immediate (Task 05-02-A escalateCriticalValue)
- Attempt 2: 2 min later (scheduler)
- Attempt 3: 5 min later (scheduler)
- Attempt 4+: 15 min later (scheduler)
- Abandon after 3 consecutive failures (status = ABANDONED) + alert ops

**Scheduler job:** `processCriticalRetryQueue` (cron: every 5 min)

```typescript
// Cloud Function triggered by Cloud Scheduler
export async function processCriticalRetryQueue(req: https.Request) {
  const labs = getAllActiveLabs(); // from admin config

  for (const lab of labs) {
    const pendingEntries = await getRetryQueueEntries(lab.id, {
      status: 'PENDING',
      nextRetryAt: { $lte: now() },
    });

    for (const entry of pendingEntries) {
      await escalateCriticalValue(lab.id, entry.escalacaoId);
      // Updates retry-queue doc with new attempt count
    }
  }
}
```
