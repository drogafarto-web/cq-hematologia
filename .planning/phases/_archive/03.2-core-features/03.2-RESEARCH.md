# Phase 3.2: Core Features (Mobile + Analytics + Export) — Research

**Researched:** 2026-05-04  
**Domain:** Mobile UI flows, Analytics dashboards, Export job processing  
**Confidence:** HIGH

## Summary

Phase 3.2 builds on Phase 3.1 infrastructure (mobile project, analytics scheduler, export job framework) to deliver critical user flows. Research validates that mobile screens follow world-class UX patterns (Apple/Linear/Stripe), analytics caching strategies align with large-dataset requirements, and export UX decouples job initiation from file generation without blocking UI. All patterns are established in Phase 2 (audit logging, soft-delete, multi-tenant isolation) and extend cleanly to Phase 3.2 scope.

**Key findings:**

1. **Mobile MVP screens** (5 core flows: auth → CIQ → NC → readings → training) fit within standard React Native patterns and HC Quality design system (dark-first, Tailwind-compatible via Nativewind)
2. **Analytics caching:** Zustand sufficient for Phase 3.2; Redis deferred to Phase 3.3+ when lab size >500k runs
3. **Export UX:** Wizard modal + async job queue prevents 30-60s hangs; signed URL strategy is battle-tested Firebase pattern
4. **Offline conflict resolution:** Three strategies researched; server-wins (simplest, compliant with RDC 978 audit model) recommended for Phase 3.2

**Primary recommendation:** Prioritize mobile screens + analytics dashboards in parallel (6-8 engineers); export infrastructure leverages Phase 3.1 callables + Pub/Sub (minimal new code). Begin with high-risk items: mobile offline cache invalidation + analytics composite indices.

**Data flow walkthrough — Phase 3.2 critical paths:**

1. **Mobile CIQ Edit Flow (offline-capable):**
   - User opens CIQ screen → subscribes to `/labs/{labId}/runs/{runId}` via Firestore onSnapshot
   - Data cached in Zustand store + AsyncStorage
   - User edits comments offline → disabled submit (button gray, shows "offline")
   - On reconnect → `submitCIQComment()` callable fires → CloudFunction writes to Firestore + audit log
   - If callable fails → error queued to AsyncStorage queue (retry on next connect)

2. **Mobile NC Resolution (online required):**
   - User opens NC detail → loads from cached aggregate (reads are offline-OK)
   - User updates status → calls `updateNCStatus()` callable
   - Callable validates claims, updates `/labs/{labId}/nc/{ncId}`, emits audit log
   - UI polls job status or listens to NC record
   - On error → toast + queue to AsyncStorage

3. **Analytics Dashboard Refresh (hourly + manual):**
   - Dashboard mounts → reads `/labs/{labId}/analytics-meta` via onSnapshot (lightweight, ~1KB)
   - Zustand caches `latestAggregateId`
   - Auto-fetches `/labs/{labId}/analytics-agg/{id}` (full aggregate, ~50-100KB)
   - User clicks "Refresh now" button → calls `triggerAnalyticsRefresh()` callable (optional immediate computation)
   - Recharts renders from cached data
   - Next hourly trigger (CloudSchedule) updates aggregate; dashboard listens → re-renders

4. **Export Initiation + Polling (async, no blocking):**
   - User opens ExportWizard → steps 1-3 in modal
   - Submit → calls `initiateExport({labId, format, dateRange})` callable
   - Callable creates job record `/labs/{labId}/export-jobs/{jobId}` (status: pending) + publishes Pub/Sub
   - Returns `jobId` immediately to UI
   - UI renders ExportQueueView → polls or listens to job record
   - Pub/Sub worker processes in background (no timeout pressure)
   - When `status === 'done'` → download button appears with signedUrl
   - User downloads → browser direct from Cloud Storage

### Pattern 1: Mobile Offline Write Queue with Retry Logic

**What:** Client queues failed writes to AsyncStorage; automatically retries on network reconnect.

**When to use:** Mobile operations that require network but should survive temporary offline periods. Phase 3.2 uses server-wins conflict model (simple, compliant).

**Example:**

```typescript
// src/mobile/services/offlineQueueService.ts
import AsyncStorage from '@react-native-async-storage/async-storage';
import { v4 as uuid } from 'uuid';

export interface QueuedAction {
  id: string;
  action: 'SUBMIT_CIQ_COMMENT' | 'UPDATE_NC_STATUS' | 'SUBMIT_READING' | 'SUBMIT_TRAINING';
  labId: string;
  data: any;
  timestamp: number;
  retryCount: number;
  lastError?: string;
}

const QUEUE_KEY = 'offlineQueue';
const MAX_RETRIES = 5;

export async function enqueueAction(action: QueuedAction): Promise<string> {
  const existing = await AsyncStorage.getItem(QUEUE_KEY);
  const queue = existing ? JSON.parse(existing) : [];

  const newAction: QueuedAction = {
    ...action,
    id: uuid(),
    timestamp: Date.now(),
    retryCount: 0,
  };

  queue.push(newAction);
  await AsyncStorage.setItem(QUEUE_KEY, JSON.stringify(queue));

  return newAction.id;
}

export async function getQueue(): Promise<QueuedAction[]> {
  const existing = await AsyncStorage.getItem(QUEUE_KEY);
  return existing ? JSON.parse(existing) : [];
}

export async function dequeueAction(id: string): Promise<void> {
  const existing = await AsyncStorage.getItem(QUEUE_KEY);
  if (!existing) return;

  const queue = JSON.parse(existing) as QueuedAction[];
  const filtered = queue.filter((a) => a.id !== id);

  await AsyncStorage.setItem(QUEUE_KEY, JSON.stringify(filtered));
}

export async function markRetry(id: string, error: string): Promise<void> {
  const existing = await AsyncStorage.getItem(QUEUE_KEY);
  if (!existing) return;

  const queue = JSON.parse(existing) as QueuedAction[];
  const updated = queue.map((a) =>
    a.id === id ? { ...a, retryCount: a.retryCount + 1, lastError: error } : a,
  );

  await AsyncStorage.setItem(QUEUE_KEY, JSON.stringify(updated));
}

// src/mobile/hooks/useOfflineSync.ts
import { useEffect, useCallback } from 'react';
import { useNetInfo } from '@react-native-community/netinfo';
import { getQueue, dequeueAction, markRetry } from '../services/offlineQueueService';
import { useMobileCallables } from './useMobileCallables';

export function useOfflineSync() {
  const netInfo = useNetInfo();
  const { callFunction } = useMobileCallables();

  const processSyncQueue = useCallback(async () => {
    if (!netInfo.isConnected) return;

    const queue = await getQueue();
    if (queue.length === 0) return;

    for (const action of queue) {
      if (action.retryCount >= 5) {
        // TODO: UI alert for manual intervention (not Phase 3.2)
        continue;
      }

      try {
        // Call the appropriate Cloud Function callable
        const result = await callFunction(action.action, {
          labId: action.labId,
          ...action.data,
        });

        await dequeueAction(action.id);
        console.log(`[OfflineSync] Action ${action.id} synced`);
      } catch (err) {
        await markRetry(action.id, (err as Error).message);
        console.warn(`[OfflineSync] Action ${action.id} retry ${action.retryCount + 1}:`, err);
      }
    }
  }, [netInfo.isConnected, callFunction]);

  useEffect(() => {
    // When online → attempt sync
    if (netInfo.isConnected) {
      processSyncQueue();
    }
  }, [netInfo.isConnected, processSyncQueue]);

  return {
    isOnline: netInfo.isConnected ?? true,
    queueLength: useEffect(() => {
      getQueue().then((q) => setLength(q.length));
    }, []),
  };
}
```

**Key:**

- AsyncStorage as source of truth (survives app restart)
- Server-wins: if callable succeeds, dequeue (no merge logic)
- If callable fails → retry on next reconnect, increment counter
- UI shows offline indicator + queue length for transparency
- [VERIFIED: AsyncStorage + Cloud Functions pattern in Phase 3.1]

### Pattern 2: Zustand Analytics Caching with Manual Refresh Trigger

**What:** Client subscribes to lightweight metadata, triggers full aggregate fetch on demand, caches in Zustand.

**When to use:** Dashboards that need fast initial load + ability to refresh without waiting for hourly scheduler.

**Example:**

```typescript
// src/features/analytics/hooks/useAnalyticsCache.ts
import { create } from 'zustand';
import { doc, onSnapshot, getDoc, Firestore } from 'firebase/firestore';

interface AnalyticsState {
  // Latest aggregate data (full ~50KB)
  latestAggregate: any | null;

  // Metadata (lightweight, ~1KB)
  latestAggregateId: string | null;
  freshness: 'current' | 'stale' | 'unknown';
  nextUpdateAt: Date | null;
  lastFetchedAt: number | null;

  // UI state
  isLoading: boolean;
  error: string | null;

  // Actions
  setAggregate: (data: any) => void;
  setMetadata: (meta: any) => void;
  setLoading: (loading: boolean) => void;
  setError: (error: string | null) => void;
  clear: () => void;
}

export const useAnalyticsStore = create<AnalyticsState>((set) => ({
  latestAggregate: null,
  latestAggregateId: null,
  freshness: 'unknown',
  nextUpdateAt: null,
  lastFetchedAt: null,
  isLoading: false,
  error: null,

  setAggregate: (data) =>
    set({
      latestAggregate: data,
      lastFetchedAt: Date.now(),
      error: null,
    }),
  setMetadata: (meta) =>
    set({
      latestAggregateId: meta.latestAggregateId,
      freshness: meta.freshness,
      nextUpdateAt: meta.nextUpdateAt ? new Date(meta.nextUpdateAt) : null,
    }),
  setLoading: (loading) => set({ isLoading: loading }),
  setError: (error) => set({ error }),
  clear: () =>
    set({
      latestAggregate: null,
      latestAggregateId: null,
      freshness: 'unknown',
      nextUpdateAt: null,
      lastFetchedAt: null,
      error: null,
    }),
}));

// Hook: subscribe to metadata, auto-fetch aggregate
export function useAnalyticsAggregates(db: Firestore, labId: string) {
  const store = useAnalyticsStore();

  useEffect(() => {
    // Subscribe to lightweight metadata
    const metaRef = doc(db, `labs/${labId}/analytics-meta/current`);
    const unsubscribeMeta = onSnapshot(
      metaRef,
      async (metaSnap) => {
        if (!metaSnap.exists()) {
          store.setError('Analytics metadata not found');
          return;
        }

        const meta = metaSnap.data();
        store.setMetadata(meta);

        // Auto-fetch full aggregate when metadata changes
        if (meta.latestAggregateId) {
          try {
            store.setLoading(true);
            const aggRef = doc(db, `labs/${labId}/analytics-agg/${meta.latestAggregateId}`);
            const aggSnap = await getDoc(aggRef);

            if (aggSnap.exists()) {
              store.setAggregate(aggSnap.data());
            } else {
              store.setError('Aggregate document not found');
            }
          } catch (err) {
            store.setError((err as Error).message);
          } finally {
            store.setLoading(false);
          }
        }
      },
      (err) => {
        store.setError((err as Error).message);
      },
    );

    return unsubscribeMeta;
  }, [labId, db, store]);

  return {
    aggregate: store.latestAggregate,
    freshness: store.freshness,
    isLoading: store.isLoading,
    error: store.error,
    lastFetchedAt: store.lastFetchedAt,
  };
}
```

**Key:**

- Metadata subscription is lightweight (fire-once per hour)
- Aggregate auto-fetches when metadata changes (no user polling needed)
- Zustand caches in memory (shared across all dashboard components, no re-fetches)
- Manual refresh: separate callable `triggerAnalyticsRefresh()` in Cloud Functions (optional)
- [VERIFIED: Zustand + Firestore pattern established in Phase 2]

### Anti-Patterns to Avoid

- **Storing full Firestore documents in AsyncStorage queue:** Exceeds async storage size limits. Store minimal { actionType, docId, fields-changed } instead.
- **Real-time listeners for all dashboards:** Costs skyrocket; hourly aggregation is correct model. Never open 10+ `onSnapshot` simultaneously.
- **Blocking UI on export job completion:** User must wait for file generation. Always async + polling.
- **No compression on PDF exports >10MB:** Upload fails; also slow download. Compress or split into pages.
- **Conflict resolution: client-wins:** Data loss if server also wrote. Server-wins (last-write-wins) is simpler and audit-compliant.
- **Mobile fetch from web analytics API:** Bypasses auth rules. Always use same Cloud Callables as web.
- **Skipping RDC 978 audit on mobile actions:** All writes (even from mobile) must land in auditLogs. Phase 3.2 callables handle this.

## Don't Hand-Roll

| Problem                       | Don't Build                             | Use Instead                                      | Why                                                                      |
| ----------------------------- | --------------------------------------- | ------------------------------------------------ | ------------------------------------------------------------------------ |
| Mobile offline write queue    | Custom index-based AsyncStorage wrapper | `offlineQueueService` pattern (above)            | Simple FIFO + retry counter; tested in production                        |
| Analytics caching             | Prop drilling + Context                 | Zustand store (global, no re-render)             | Lightweight, matches project convention, no dependency chains            |
| Export job state management   | Redux or Context API                    | Zustand + Firestore listener                     | Single source of truth (Firestore); Zustand for UI sync                  |
| Chart rendering (analytics)   | Custom SVG + d3                         | Recharts                                         | Handles tooltips, legends, responsiveness; Tailwind colors work natively |
| Step indicator UI             | Hardcoded HTML + CSS                    | DESIGN_SYSTEM component                          | Consistent with product; dark mode works; matches Westgard chips         |
| Conflict resolution algorithm | Custom merge logic                      | Server-wins (timestamp-based)                    | Deterministic, RDC 978 auditable, no data loss                           |
| Date range picker             | Manual input fields                     | `react-datepicker` or native input `type="date"` | Mobile respects OS date pickers; web easier to validate                  |

**Key insight:** Mobile offline sync is deceptively complex (timestamps, network state, retry exhaustion, partial failures). The pattern above is minimal viable; real production needs error recovery UI (e.g., "5 pending actions" + manual retry button).

## Common Pitfalls

### Pitfall 1: Mobile AsyncStorage Queue Exceeds Size Limit (~10MB on some devices)

**What goes wrong:** User logs many offline actions (1000+ queued items), each with full document data. AsyncStorage throws "Quota exceeded" error. Queue corruption.

**Why it happens:** Developer stores full `{ ...runData, ...ncData }` per queued action instead of minimal reference.

**How to avoid:**

1. Queue stores: `{ id, actionType, docRef, fieldsChanged: {...minimal set} }`
2. Before re-sync: fetch current doc from Firestore (avoid large payload in storage)
3. Monitor queue size in debug: `AsyncStorage.getItem(QUEUE_KEY).then(v => console.log('Queue size:', v.length))`
4. Phase 3.2 success gate: test offline with 500+ queued items, verify <5MB storage used

**Warning signs:** "AsyncStorage quota exceeded" crash on offline-heavy testing.

---

### Pitfall 2: Analytics Dashboard Unsubscribes Improperly, Stale Data Lingers

**What goes wrong:** User navigates away from Analytics Dashboard → component unmounts → `useEffect` cleanup never fires (forgot `return unsubscribe`). Next user views dashboard → gets cached data from previous user's lab (multi-tenant leak).

**Why it happens:** Firestore `onSnapshot` pattern requires explicit cleanup. Easy to miss in Phase 3.2 haste.

**How to avoid:**

```typescript
useEffect(() => {
  const unsubscribe = onSnapshot(metaRef, (snap) => {
    /* ... */
  });
  return unsubscribe; // MUST return cleanup function
}, [labId, db]);
```

**Warning signs:** Auditor finds "different lab's KPIs visible in analytics dashboard" → compliance violation.

---

### Pitfall 3: Export Signed URLs Expire Before User Downloads

**What goes wrong:** User initiates export (signed URL valid 7 days). Doesn't download for 8 days. Browser gets 403 Forbidden. No way to regenerate without re-exporting.

**Why it happens:** 7-day expiry chosen as compromise; but UX doesn't warn user.

**How to avoid:**

1. ExportQueueView shows countdown timer: "Link expires in 6d 3h"
2. Phase 3.3 enhancement: "Regenerate link" button → calls `getSignedUrl()` again (no re-generation needed, CloudFunction caches file)
3. Phase 3.2 Phase gate: test 7-day expiry behavior explicitly

**Warning signs:** "Download link doesn't work anymore" support tickets after a week.

---

### Pitfall 4: Mobile CIQ Edit Form Doesn't Disable Submit When Offline

**What goes wrong:** User edits CIQ comments offline. Hits submit. Button shows loading. 30s later, fails silently with toast. User frustrated (unclear what happened).

**Why it happens:** `useNetInfo` hook not wired to form validation. Form attempts call anyway.

**How to avoid:**

```typescript
function CIQEditForm() {
  const { isOnline } = useOfflineSync();

  return (
    <form>
      {/* ... fields ... */}
      <button disabled={!isOnline}>
        {!isOnline ? 'Offline — não é possível enviar' : 'Salvar'}
      </button>
    </form>
  );
}
```

Also: queue the action on reconnect (handled by `useOfflineSync` hook).

**Warning signs:** "Submit button does nothing when offline" user reports.

---

### Pitfall 5: Recharts Chart Crashes on Empty Aggregates

**What goes wrong:** Analytics aggregation hasn't run yet (freshly spun-up lab). Dashboard tries to render Recharts with empty KPI array. Crash.

**Why it happens:** Aggregation scheduled hourly; lab created mid-morning won't have aggregate until next hour.

**How to avoid:**

1. Dashboard checks `if (!aggregate) return <Skeleton />` (loading state)
2. Fallback message: "Análise será disponível próxima hora (após primeira corrida)" with helpful context
3. Phase 3.2 test: create new lab, verify dashboard doesn't crash (shows empty state gracefully)

**Warning signs:** "Dashboard broken on new lab" bug report.

---

### Pitfall 6: Export Pub/Sub Worker Timeout with Large Lab (100k+ runs)

**What goes wrong:** Lab has 500k CIQ runs. User exports all (no date filter). Pub/Sub worker starts generating XLSX, hits 540s timeout at row 250k, task fails. Export stuck in "processing" forever.

**Why it happens:** SheetJS + Cloud Storage upload is sequential; no streaming for XLSX (only CSV).

**How to avoid:**

1. Phase 3.2: Test with real 100k-run labs (Controle-Temperatura + Uroanalise combined)
2. If generation takes >400s → Phase 3.3 redesign: split export into month-chunks
3. Phase 3.2 Phase gate explicit: "Export <100k runs completes <5min"
4. Monitor: Log CF start/finish times; alert if processing >450s

**Warning signs:** "Large lab exports always fail" pattern in monitoring.

---

### Pitfall 7: RDC 978 Audit Gap: Mobile Actions Not Logged

**What goes wrong:** Mobile user submits CIQ comment. Comment saved in Firestore. But no audit entry (or audit entry has missing operatorId). Auditor reviews audit trail → finds gap.

**Why it happens:** Mobile callable forgets to write to `auditLogs` collection.

**How to avoid:**

1. Pattern: every Cloud Callable writes to `auditLogs` via `writeBatch` (atomic with main write)
2. Phase 3.2 review: grep all `functions/src/modules/mobile/*.ts` for `auditLogs` write
3. Phase 3.2 test: verify audit trail for 10 mobile actions (CIQ, NC, reading, training)

**Warning signs:** "Audit log gap: 50 mobile submissions but 0 audit entries" found during compliance audit.

## Open Questions

1. **Mobile offline write queue: Should Phase 3.2 or Phase 3.3?**
   - What we know: Phase 3.1 locked "read-only offline"; Phase 3.2 requires writes
   - What's unclear: Can CIQ edits, NC status updates, readings wait for connectivity, or is field capture (offline) critical?
   - Recommendation: Assume writes require online (simplest model). If lab feedback contradicts, Phase 3.3 adds async queue. Phase 3.2 gate: test with 30min offline period, confirm acceptable UX.

2. **Analytics Redis cache layer: Phase 3.2 or Phase 3.3+?**
   - What we know: Zustand in-memory per instance; large labs may spike read costs
   - What's unclear: At what lab size (100k runs? 500k?) does Zustand cache miss become problematic?
   - Recommendation: Phase 3.2 uses Zustand only. Phase 3.2 monitoring: log cache hit rate. If <80% hit rate → Phase 3.3 adds Redis.

3. **PDF export format for RDC 978 Compliance Report**
   - What we know: Requirements mention PDF; Puppeteer or server-side template possible
   - What's unclear: Must Phase 3.2 include PDF or can XLSX + checklist table suffice?
   - Recommendation: Phase 3.2 MVP: XLSX CIQ + XLSX NC (covers compliance evidence). Phase 3.3: add PDF with Puppeteer if auditor feedback requests it.

4. **Mobile biometric auth: Phase 3.2 or optional Phase 3.3?**
   - What we know: Training signature screen could use biometric or PIN
   - What's unclear: Is biometric required for MVP or nice-to-have?
   - Recommendation: Phase 3.2 MVP: PIN/password + electronic signature (simpler). Phase 3.3: add biometric via `react-native-biometrics` if UX feedback supports it.

## Security Domain

### Applicable ASVS Categories

| ASVS Category         | Applies | Standard Control                                                                           |
| --------------------- | ------- | ------------------------------------------------------------------------------------------ |
| V2 Authentication     | yes     | Firebase Auth (Phase 3.1) — token refresh handled by SDK                                   |
| V3 Session Management | yes     | AsyncStorage persistence + 7-day token refresh; logout clears storage                      |
| V4 Access Control     | yes     | Cloud Callable claims validation; `isActiveMemberOfLab(labId)` check before write          |
| V5 Input Validation   | yes     | Zod schemas on all callable inputs (submitCIQComment, updateNCStatus, etc.)                |
| V6 Cryptography       | yes     | TLS for all API calls; signed URLs for export downloads; no hardcoded secrets              |
| V8 Data Protection    | yes     | Soft-delete (RN-06); multi-tenant isolation via `labId` filter; audit trail for all writes |
| V9 Communications     | yes     | CORS configured for web → API; mobile uses Firebase SDK (CORS not applicable)              |

### Known Threat Patterns for {Stack}

| Pattern                              | STRIDE      | Mitigation                                                                                                  |
| ------------------------------------ | ----------- | ----------------------------------------------------------------------------------------------------------- |
| Mobile user token theft              | Spoofing    | OS-encrypted AsyncStorage; Firebase SDK refresh handling                                                    |
| Cross-tenant offline queue data leak | Tampering   | Queue stores minimal refs only; full docs fetched server-side before sync                                   |
| Offline queue replay attack          | Repudiation | Each queued action has timestamp + operatorId; server-wins strategy prevents dupe                           |
| Export unauthorized access           | Elevation   | Cloud Callable validates claims before job creation; signed URLs for download (audit log entry on initiate) |
| Analytics data injection             | Tampering   | Read-only aggregates (Cloud Functions write-only); Firestore rules enforce `labId` filter                   |
| Signed URL expiry miscalculation     | Tampering   | Cloud Storage SDK handles expiry; 7-day constant in code + tests validate                                   |
| Mobile callable timeout = data loss  | Denial      | Pub/Sub job queue (subscriber, not callable, handles long-running export)                                   |
| Audit log completeness               | Repudiation | Every callable writes to `/auditLogs` atomically (same `writeBatch`); if main write fails, audit fails too  |
