# Task 05-01 — Critical Values State Machine & Severity Rules

**Objective:** Build critical value detection engine with state machine (NORMAL → CRITICAL → ALERTED → RESOLVED) + configurable severity thresholds per analyte/equipment. RDC 978 Art. 115 + DICQ 4.6.3 compliance-first.

**Owner:** Stream A (Backend engineer, event-driven architecture)  
**Duration:** 3 days (Phase 5, Wave 3, Week 6 kickoff)  
**Compliance:** RDC 978 Art. 115 (critical result escalation), DICQ 4.6.3 (crítico workflow), RDC 978 Art. 99 (timely notification)

## Scope

### 1. Critical Value Detection Engine

**File:** `functions/src/modules/criticos/detectCriticalValue.ts` + service layer

**Requirements:**

- Callable: `recordRunWithCriticalDetection(labId, runId, analitos[])`
  - Triggered after run recorded in bioquímica module
  - Compares each analito value against thresholds (low + high per analyte)
  - Returns: `{ criticalFlagsDetected: boolean, flags: CriticalFlag[] }`

- Thresholds storage: `criticos-escalacoes/{labId}/config/thresholds/{docId}`
  ```typescript
  interface CriticalThreshold {
    analitoId: string;
    equipmentId: string; // per-equipment variation
    lowCritical: number;
    highCritical: number;
    lowWarn: number;    // optional: pre-alert level
    highWarn: number;
    unitId: string;     // e.g., "mmol/L", "mg/dL"
    enabledAt: timestamp;
    disabledAt?: timestamp; // soft-disable
  }
  ```

- Detection logic:
  ```typescript
  // Pseudocode
  const result = getValue(analito, equipmentId);
  const threshold = getThreshold(analito, equipmentId);
  
  if (result > threshold.highCritical) {
    return { isCritical: true, direction: "HIGH", severity: "CRITICAL" };
  } else if (result < threshold.lowCritical) {
    return { isCritical: true, direction: "LOW", severity: "CRITICAL" };
  } else if (result > threshold.highWarn) {
    return { isCritical: false, direction: "HIGH", severity: "WARN" };
  } else if (result < threshold.lowWarn) {
    return { isCritical: false, direction: "LOW", severity: "WARN" };
  }
  return { isCritical: false, severity: "NORMAL" };
  ```

- Trigger: Auto-enqueue critical flag to `criticos-escalacoes/{labId}/escalacoes/{docId}` if severity = "CRITICAL"
  
**Non-scope (v1.4.1):**
- Delta flags (change from last result)
- Automatic threshold learning from historical data
- Custom rule logic (only < > operators in v1.4)

### 2. Firestore Schema & Rules

**Collections created:**

```
/labs/{labId}/criticos-escalacoes/
  root/                                 (anchor: {labId, module, seededAt?})
  config/thresholds/{analitoId}         (CriticalThreshold, server-write-only)
  escalacoes/{docId}                    (CriticalFlag event, append-only)
  audit/{logId}                         (audit trail, immutable)
```

**CriticalFlag document (escalacoes):**
```typescript
interface CriticalFlag {
  id: string;                           // auto-generated
  labId: string;
  runId: string;                        // reference to bioquimica run
  analitoId: string;
  equipmentId: string;
  resultado: number;
  threshold: { lowCritical, highCritical };
  direction: "HIGH" | "LOW";
  severity: "CRITICAL" | "WARN";        // severity level
  status: "FLAGGED" | "ALERTED" | "RESOLVED"; // state machine
  alertedAt?: timestamp;                // when alert sent
  resolvedAt?: timestamp;               // when manually resolved
  operatorId: string;                   // who recorded run
  assinatura: LogicalSignature;         // server-generated
  criadoEm: timestamp;
  deletadoEm?: timestamp;
}
```

**Firestore Rules (escalacoes collection):**
- Client can read: `status == "ALERTED"` (for notification UI)
- Client can update status: "ALERTED" → "RESOLVED" (operator mark as read)
- Server-only create + assinatura validation (per security rule)

### 4. State Machine & Transitions

**State diagram:**
```
┌─────────┐
│ NORMAL  │  (no alert triggered)
└────┬────┘
     │ (critical flag detected)
     ↓
┌─────────┐
│ FLAGGED │  (critical value in queue, not yet escalated)
└────┬────┘
     │ (escalation callable triggered)
     ↓
┌─────────┐
│ ALERTED │  (SMS/email sent to clinician)
└────┬────┘
     │ (operator mark as read/resolved)
     ↓
┌─────────┐
│RESOLVED │  (no further action)
└─────────┘
```

**Transitions enforced via Firestore Rules:**
- FLAGGED → ALERTED: server-only (via callable)
- ALERTED → RESOLVED: client-write (via UI button, operator)
- No reverse transitions (immutable audit)

### 5. Tests & Validation

**Unit tests** (Jest, 25+ tests):
- `detectCriticalValue.test.ts`:
  - Value in range: no critical flag
  - Value above high critical: flag created (HIGH, CRITICAL)
  - Value below low critical: flag created (LOW, CRITICAL)
  - Value above high warn: flag created (HIGH, WARN)
  - Equipment-specific thresholds applied correctly
  - Soft-deleted thresholds ignored

- `stateTransitions.test.ts`:
  - FLAGGED → ALERTED succeeds (callable)
  - ALERTED → RESOLVED succeeds (client)
  - Invalid transitions rejected (e.g., RESOLVED → ALERTED)
  - State immutability enforced

- `rules.test.ts`:
  - Client cannot create escalacoes doc directly
  - Client cannot modify assinatura field
  - Client can read own labId's ALERTED flags
  - Client cannot read other lab's flags

**Integration test** (emulator + Firestore):
- Seed thresholds → create run → detect critical → verify escalacoes doc created with correct state
- Multi-analyte run: 2 criticos detected → 2 separate escalacoes docs
- Threshold disabled: same run value → no critical flag created
