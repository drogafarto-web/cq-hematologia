## Scheduling Architecture

Cloud Tasks API (GCP) is the scheduling system:
- User creates a ScheduledExport record in Firestore (`/schedules/{labId}/exports`)
- Cloud Function callable validates cron expression and creates a Cloud Task in the queue
- Cloud Task invokes a Cloud Function `exportJobWorker` at the scheduled time with the export config
- `exportJobWorker` initiates the export (same as immediate export) and emails the result
- Failures are retried via Cloud Tasks dead-letter handler (3x exponential backoff)

## Firestore Rules

```
/schedules/{labId}/exports/{scheduleId}
  - allow create: if isActiveMemberOfLab(labId) && validScheduleCron(d) && d.isActive == true
  - allow read: if isActiveMemberOfLab(labId)
  - allow update: if isActiveMemberOfLab(labId) && keepsLabId() && resource.data.deletedAt == null
  - allow delete: if false  // soft delete only
```

`validScheduleCron(d)` must validate the `d.cron` field matches standard 5-field format.

- **Cost**: Cloud Tasks + Cloud Functions invocations add to billing. Budget: ~100 tasks/month per lab (~$0.01/task = $1/month impact).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User Cron Input → Validation | User enters cron expression; must be validated server-side before Cloud Task creation |
| Timezone String → Cloud Tasks | User selects IANA timezone; passed to Cloud Tasks API and must match valid TZ database |
| Scheduled Job Payload → Cloud Function | Cloud Task invokes `exportJobWorker` with labId/scheduleId; must re-validate auth context |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-04 | Tampering | User modifies cron expression to run every second (DOS) | mitigate | Cron validator rejects high-frequency expressions (minimum 1-minute intervals). Server-side parse rejects `/1` or `* * * * *` patterns. |
| T-07-05 | Spoofing | User schedules export for another lab (labId mismatch) | mitigate | Cloud Function validates `request.auth.token.labClaims[labId]` before creating task. Firestore rules prevent cross-tenant write. |
| T-07-06 | Elevation of Privilege | User schedules export with non-member email recipient | mitigate | Email recipient must match lab's internal domain or registered contact list (validate in scheduleExportJob). Out-of-band email to external addresses blocked. |
| T-07-07 | Denial of Service | User creates 1000 scheduled exports, inflates Cloud Tasks quota | mitigate | Rate limit: max 50 scheduled exports per lab (enforce in callable). Audit log tracks all creates. Admin can disable runaway jobs. |

</threat_model>
