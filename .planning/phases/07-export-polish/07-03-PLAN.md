## Batch Export Architecture

**Workflow:**
1. User opens "Batch Export" in analytics or reports module
2. Selects documents to export (checkboxes, multi-select)
3. Chooses format(s) (XLSX, PDF, CSV — can select multiple)
4. Optionally applies a saved template (header/footer/branding)
5. Confirms → Cloud Callable initiates job
6. Job creates Cloud Task → background worker generates files
7. Worker compresses all files to zip, stores in Cloud Storage
8. Downloads/emails link to zip (7-day TTL)

**Template System:**
- Templates stored in `/templates/{labId}/exports/{templateId}`
- Fields: name, headerHtml, footerHtml, logoUrl, fontFamily, colors (optional)
- Templates owned by creating user (RN-12), editable by lab admins
- Header/footer are HTML snippets (max 500 chars each) applied to all generated documents
- Preview in editor shows mock header/footer in target format

export interface TemplateLayout {
  headerHtml: string;  // e.g., "<img src='{logoUrl}' /> <h1>{labName}</h1>"
  footerHtml: string;  // e.g., "<p>Generated on {date}</p>"
  pageNumbers: boolean;
  fontFamily: 'inter' | 'georgia';  // serif vs sans
  colors?: {
    accent?: string;
    text?: string;
  };
}

export interface ExportTemplate {
  templateId: string;
  labId: string;
  name: string;
  description?: string;
  layout: TemplateLayout;
  createdBy: string;  // userId of creator (RN-12 ownership)
  createdAt: Timestamp;
  updatedAt: Timestamp;
  deletedAt?: Timestamp;
  isDefault?: boolean;
}

export interface BatchExportJob {
  jobId: string;
  labId: string;
  formats: ExportFormat[];  // ['xlsx', 'pdf', 'csv']
  documentIds: string[];  // doc IDs to export (CIQ runs, audit records, etc.)
  templateId?: string;
  status: 'queued' | 'processing' | 'compressing' | 'completed' | 'failed';
  progress: {
    docsProcessed: number;
    docsTotal: number;
    currentFormat?: ExportFormat;
  };
  zipUrl?: string;  // Cloud Storage download link
  createdAt: Timestamp;
  completedAt?: Timestamp;
  estimatedSizeBytes?: number;
  errorMessage?: string;
}
```

## Firestore Rules

```
/templates/{labId}/exports/{templateId}
  - allow create: if isActiveMemberOfLab(labId) && resource.data.createdBy == request.auth.uid
  - allow read: if isActiveMemberOfLab(labId)
  - allow update: if isActiveMemberOfLab(labId) && (resource.data.createdBy == request.auth.uid || isSuperAdmin())
  - allow delete: if false  // soft delete only

/batch-export-jobs/{labId}/{jobId}
  - allow create: if isActiveMemberOfLab(labId)
  - allow read: if isActiveMemberOfLab(labId) && resource.data.createdBy == request.auth.uid
  - allow update: if request.auth.token.isAdmin == true  // only Cloud Function updates status
  - allow delete: if false
```

## Cost Analysis

Batch exports are I/O and compute intensive:
- **Cloud Functions:** 1 invocation per batch (small), N invocations for doc generation (one per format per doc)
- **Estimated:** 30 docs × 3 formats = 90 function calls (~$0.40/batch with free tier)
- **Cloud Storage:** zip file storage (7 days), ~10-100 MB per batch, ~$0.02/batch
- **Total per batch:** ~$0.50, easily offset by elimination of manual exports (saves 10-15 min operator time)

## Bundle Impact

New dependencies:
- `jszip` (npm, ~30KB gzip) — zip file creation
- `html2pdf` (npm, ~90KB gzip) — HTML → PDF conversion (already in use for single exports)
- No other new libs; use existing Zod, Zustand

Total bundle impact: ~30KB (jszip). HTML2PDF already bundled in Phase 3.3.

   ```
   for (let format of formats) {
     for (let docId of documentIds) {
       const doc = await getDoc(docId);
       const formatted = await renderInFormat(format, doc, template);
       await uploadToStorage(formatted, `.../{jobId}/{format}/{docId}`);
       updateJobProgress(jobId, docsProcessed++);

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Template HTML → Document Rendering | Template header/footer may contain malicious HTML/JS; must be sanitized before rendering |
| User Document Selection → Batch Job | User selects docs from list; must verify they own the lab and have read access |
| Batch Job Payload → Cloud Function | Cloud Task invokes worker with labId/jobId; must re-authenticate via admin SDK |
| Zip File Storage → Download Link | Signed URL grants temporary access; 7-day expiry prevents indefinite sharing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-08 | Tampering | User modifies template to inject malicious JS | mitigate | Template HTML sanitized with DOMPurify before rendering. Only safe tags allowed: p, h1-h6, img, br, em, strong. No script/iframe/onclick. |
| T-07-09 | Information Disclosure | User downloads zip containing docs from other labs | mitigate | Cloud Task validates `labId` in payload matches user's active lab. Firestore query filters docs by labId. Zip contains only queried docs. |
| T-07-10 | Denial of Service | User selects 10,000 docs, creates huge zip (100+ GB) | mitigate | Limit doc batch size: max 500 docs per batch (enforced in BatchExportWizard). Estimate size before creation; reject if > 1GB. |
| T-07-11 | Elevation of Privilege | Non-admin user creates template visible to entire lab | mitigate | Template CRUD enforces `isActiveMemberOfLab`. Templates read by all members but only creator/superadmin can edit/delete (RN-12). |

</threat_model>
