Manual refresh button triggers `analyticsAggregation` callable. `useAnalyticsMeta` subscribes to `/labs/{labId}/analytics-meta` which has `lastUpdated` timestamp.

**What does NOT exist yet:**

- Polling (30s automatic refresh checking meta lastUpdated)
- Date range — all dashboards show all-time data from latest aggregate
- Equipment / operator filter — no filtering at all
- PDF export — button stub only, no Cloud Function
- Tablet layout — fixed `max-w-4xl` desktop container, no responsive grid

**Architecture for date range + filtering:**
The aggregate schema stores pre-bucketed data (hourly slices). Filtering by date range = querying multiple aggregate docs and merging. Filtering by equipment/operator = post-process on the merged aggregate data. This means:

- `useDateRangeFilter` manages `startDate`/`endDate` state and fetches relevant aggregate docs
- `useEquipmentFilter` / `useOperatorFilter` manage selection state; filtering is done in `kpiCalculators.ts` (client-side, since aggregates are small)
- Performance target: <500ms for filter change — achievable because aggregates are already in Firestore, not raw runs

**Real-time polling strategy:**

- Do NOT re-fetch full aggregate on every poll tick
- Poll `/labs/{labId}/analytics-meta` every 30s
- Compare `lastUpdated` with cached `lastUpdated` in Zustand
- Only re-fetch aggregate if `lastUpdated` has changed
- This means most polls are 1 Firestore read (meta doc, ~500 bytes) — negligible cost

**PDF generation strategy:**

- Client calls `generateDashboardPDF` callable with `{ labId, dashboardType, dateRange, filters }`
- CF renders dashboard HTML server-side (pre-built snapshot template), converts to PDF via Puppeteer
- Stores in Cloud Storage: `/exports/{labId}/dashboards/{dashboardType}-{timestamp}.pdf`
- Returns signed URL (7-day expiry)
- Same job queue pattern as existing XLSX exports (initiate → process → status poll → download)

**Responsive grid:**

- Current: `grid-cols-2` fixed
- Target: `grid-cols-1 sm:grid-cols-2 lg:grid-cols-4` for KPI cards
- Charts: `w-full` with `aspect-[4/3]` container — scales correctly on tablet
- Filter bar: horizontal scroll on mobile (`overflow-x-auto`), wraps on tablet

**Create `src/features/analytics/hooks/useDateRangeFilter.ts`:**

```typescript
export type DatePreset = '30d' | '90d' | '1y' | 'custom';

export interface DateRange {
  start: Date;
  end: Date;
  preset: DatePreset;
}

export function useDateRangeFilter() {
  // Default: last 30 days
  const [range, setRange] = useState<DateRange>({
    start: subDays(new Date(), 30),
    end: new Date(),
    preset: '30d',
  });

  const setPreset = useCallback((preset: Exclude<DatePreset, 'custom'>) => {
    const end = new Date();
    const daysMap = { '30d': 30, '90d': 90, '1y': 365 };
    setRange({ start: subDays(end, daysMap[preset]), end, preset });
  }, []);

  const setCustomRange = useCallback((start: Date, end: Date) => {
    setRange({ start, end, preset: 'custom' });
  }, []);

  return { range, setPreset, setCustomRange };
}
```

Uses `date-fns` `subDays` (already in project dependencies). No new libs.

10. Log to `auditLogs` (callable side effect, per CLAUDE.md convention)

<threat_model>

## Trust Boundaries

| Boundary                           | Description                                                                               |
| ---------------------------------- | ----------------------------------------------------------------------------------------- |
| Analytics filter → Firestore reads | Equipment + operator lists fetched from lab's own subcollections; labId enforced in query |
| PDF export CF → Cloud Storage      | Signed URL generated server-side; client never writes to Storage directly                 |
| Polling tick → Firestore meta read | Point read (not listener); 30s cadence is ~2880 reads/day per user — acceptable cost      |

## STRIDE Threat Register

| Threat ID | Category               | Component                                 | Disposition | Mitigation Plan                                                                                                          |
| --------- | ---------------------- | ----------------------------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------ |
| T-03.3-05 | Information Disclosure | PDF contains dashboard data               | mitigate    | PDF generation CF validates auth + labId ownership before rendering; signed URL expires in 7 days                        |
| T-03.3-06 | Denial of Service      | 30s polling on 500+ concurrent users      | mitigate    | Meta doc is tiny (~200 bytes); Firestore scales to 1M reads/s; 500 users = 1k reads/min, negligible                      |
| T-03.3-07 | Tampering              | Filter params manipulated in PDF callable | mitigate    | CF validates `labId` against `request.auth.uid`; equipment/operator IDs validated against lab's collections before query |
| T-03.3-08 | Elevation              | PDF shows data outside user's date range  | mitigate    | Date range validated server-side; aggregate query uses Firestore `where` with timestamp bounds                           |

</threat_model>
