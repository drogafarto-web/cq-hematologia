1. Deterministic (same input → same output hash) for signature validation
2. Field-level granular (e.g., "field 'severidade' changed from 'alto' to 'crítico'")

4. Array-aware (detect item additions/removals in lists, not just "array changed")

Export functions:
- `buildDiff(before: any, after: any): DiffEntry[]` — returns array of field-level changes
- `DiffEntry { path: string; before: any; after: any; type: 'add'|'remove'|'update'|'array-item' }`

Implementation notes:
- Use recursive tree walk with dot-notation paths (e.g., "tratamento.status", "evidencias[0].arquivo")
- Ignore internal Firestore fields (__name__, timestamp mutations when using serverTimestamp())
- For arrays: deep-compare items by value, not reference
- Output must be JSON-serializable for signing (no circular refs, no functions)

Per RDC 978 5.3: diff must show exact before/after values for forensic audit.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → Cloud Function | Untrusted input: client may submit altered before/after payloads |
| Cloud Function → Firestore | Trusted: CF generates diffs + context server-side, Firestore rules validate signatures |
| Firestore → Audit Trail Collection | Immutable: rules prevent updates/deletes after creation |
| Operator → Audit Context | Partially trusted: operator identity from Firebase auth (reliable), but operator may misstate "intent" |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Spoofing | Operator identity in context | Mitigate | `operatorId` from `request.auth.uid` (Firebase guarantee); cannot be forged from client |
| T-07-02 | Tampering | Diff detection | Mitigate | Diffs computed server-side in Cloud Function; client cannot inject altered diffs. Signature covers diffs. |
| T-07-03 | Repudiation | Operator denies action | Mitigate | Audit entry signed with `operatorId + hash + ts`; operator cannot claim someone else signed |
| T-07-04 | Info Disclosure | Audit context leakage | Mitigate | Read access to audit trail restricted to AUDITOR/RT/admin roles via Firestore rules |
| T-07-05 | Denial of Service | Audit write floods | Accept | Cloud Function quota limits apply; audit writes are synchronous, so volume is naturally throttled by operation frequency. Monitor function invocations if spike detected. |
| T-07-06 | Elevation of Privilege | Non-auditor reads audit | Mitigate | Firestore rules enforce `isActiveMemberOfLab() && (role == 'AUDITOR' or admin)` on read |

</threat_model>
