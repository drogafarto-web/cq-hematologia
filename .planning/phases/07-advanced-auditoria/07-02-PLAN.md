## Baseline Model (Phase 7 design)

A "baseline" captures expected behavior for an operator/module/time combination:
```typescript
interface OperatorBaseline {
  operatorId: string;
  role: string;
  module: string;
  timeOfDay: 'night'|'morning'|'afternoon'|'evening';  // binned from 24h
  avgOperationsPerDay: number;
  commonOperations: { [operation: string]: number };  // frequency counts
  typicalResultDistribution: { sucesso: %age, falha: %age, aviso: %age };
  lastUpdated: Timestamp;
  trainingDataPoints: number;  // how many audit entries contributed
}
```

During detection, compare incoming audit entry against baseline:
- If operation type is new to baseline → low anomaly score
- If result is atypical (e.g., unusual fail rate) → medium-high score
- If operation at unusual time + operation type uncommon for role → high score
- Use Gemini to explain *why* the combination is unusual

  <name>Task 2: Implement anomalyDetector.ts (pattern deviation scoring)</name>
  <files>functions/src/modules/qualidade/anomalyDetector.ts</files>
  <action>
Create the anomaly detection engine that scores incoming audit entries against the baseline. The scorer must:

1. Accept an audit entry + its context (from Phase 7-01 enhancements)
2. Fetch the operator's baseline (via `getOperatorBaseline()`)
3. Compute anomaly score across multiple dimensions:
   - **Operation rarity** (0–1 scale): how often has this operator done this operation? (rare = higher score)
   - **Time-of-day anomaly** (0–1): is this operation happening at an unusual time for this operator? (night-time release by day-shift operator = high)
   - **Result rarity** (0–1): is the result (sucesso/falha/aviso) unusual for this operation type? (rare result = higher score)
   - **Velocity anomaly** (0–1): how many operations in last 5 minutes? (spike = higher score)
   - **Module jump** (0–1): did operator jump between distant modules? (analyzer → sgq instantly = suspicious)

4. Combine dimensions with weighted scoring:
   ```
   overall_score = 0.3 * operation_rarity 
                 + 0.2 * time_anomaly 
                 + 0.2 * result_rarity 
                 + 0.15 * velocity_anomaly 
                 + 0.15 * module_jump
   ```

5. Export structure:
   ```typescript
   interface AnomalyScore {
     operatorId: string;
     entryId: string;
     overall: number;  // 0.0–1.0
     dimensions: {
       operationRarity: number;
       timeOfDayAnomaly: number;
       resultRarity: number;
       velocityAnomaly: number;
       moduleJump: number;
     };
     reasoning: string;  // e.g., "High time-of-day anomaly (night operation for day-shift role)"
     requiresAIAnalysis: boolean;  // true if overall > 0.75
   }
   ```

6. Export function:
   - `detectAnomalies(entry: QualidadeAuditEntry, context: AuditContext): Promise<AnomalyScore>`

Implementation notes:
- For new operators (baseline unknown): skip anomaly detection, return score 0 with reasoning "baseline not yet established"
- Velocity check: query audit trail for this operator's entries in the last 5 minutes. If >10, flag as velocity anomaly.
- Module jump: if previous operation was in module X and current is in module Y (and X != Y), compute distance as anomaly (close modules like analyzer/coagulacao = less anomalous than analyzer/sgq)
- For result rarity: if operation has no historical failures but current entry shows failure, score 0.7 (moderate concern, not extreme)

Per RDC 978 5.3: anomaly detection must not create false positives (new operators, legitimate operational patterns must not flag).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Historical Data → Baseline | Baseline assumes historical audit entries are trustworthy (already validated by Phase 7-01) |
| Operator → Anomaly Score | Operator identity from Firebase auth; baseline is per-operator (cannot spoof another operator's baseline) |
| Gemini API → Insight | Untrusted: Gemini response is advisory only, not used for automatic actions (auditor reviews before action) |
| Cloud Function → Firestore | Trusted: anomaly detection runs server-side, results persisted immutably |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-07 | Spoofing | Baseline for wrong operator | Mitigate | Baseline keyed by `operatorId` from `request.auth.uid`; cannot be spoofed |
| T-07-08 | Tampering | Anomaly score injection | Mitigate | Scores computed server-side in CF trigger, not modifiable by client. Stored immutably in Firestore. |
| T-07-09 | Repudiation | "I didn't authorize this anomaly flag" | Accept | Anomaly flags are advisory, require auditor review before action. Auditor dismissal is logged separately. |
| T-07-10 | Info Disclosure | Baseline leakage to unauthorized users | Mitigate | Firestore rules restrict reads to AUDITOR + admin. Baseline includes aggregated stats (no individual values). |
| T-07-11 | Denial of Service | Baseline recomputation overload | Mitigate | Daily cron job (not per-entry). Queries limited to 30-day window. Timeout: 5 minutes. Monitor function duration. |
| T-07-12 | Elevation | Non-admin dismisses anomaly | Mitigate | Firestore rules check `isActiveMemberOfLab() && role == 'AUDITOR'` before allowing dismissal |

</threat_model>
