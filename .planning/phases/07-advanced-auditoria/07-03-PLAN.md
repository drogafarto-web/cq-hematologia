## Alert Routing Rules

**RT (Técnico):**
- Gets alerts for their assigned modules only
- Example: RT assigned to `analyzer` + `coagulacao` → only sees anomalies in those modules

**AUDITOR:**
- Sees all alerts across all modules in the lab
- Can dismiss/acknowledge any alert

**ADMIN:**
- Sees all alerts, can dismiss/reassign

**OWNER:**
- Dashboard view of all labs, aggregated alert count

## Alert Lifecycle

1. **Anomaly detected** (07-02 trigger) → AnomalyScore with score > 0.85
2. **Alert generated** (this plan) → Persisted to `labs/{labId}/audit-alerts/{alertId}`
3. **User notified** → Real-time listener triggers UI update
4. **Auditor reviews** → Clicks drill-down, sees operator profile + related audits + AI insight
5. **Auditor dismisses** → Logs dismissal with reason (e.g., "Planned emergency release", "False positive")
6. **Archive** → Alert moved to `audit-alerts-archived` after 90 days

## Compliance Mapping

- **RDC 978 5.3** — Alert system ensures timely response to irregular patterns
- **DICQ 4.4** — Audit trail must be actively monitored and unusual events investigated
- **AUD-03-ALERTS** — User requirement: real-time alerts with drill-down context

Create the alert generation and routing engine that converts high-confidence anomalies into actionable alerts.

1. Export function: `generateAlert(anomalyScore: AnomalyScore, entry: QualidadeAuditEntry, insight: AnomalyInsight): AuditAlert`
   - Builds alert object with: anomalyScore, entry summary, insight summary, timestamp, lab/operator context
   - Assigns alert severity: "critical" (score > 0.95), "high" (0.85–0.95), "medium" (0.75–0.85)
   - Includes recommended actions from AI insight (e.g., "Contact operator", "Review related entries")

2. Export function: `routeAlert(alert: AuditAlert, labId: string, operatorId: string): Promise<{ recipientRoles: string[]; recipientIds: string[] }>`
   - Queries lab members to find RTs assigned to the operator's module (if RT) or all AUDITORs
   - Returns list of user IDs to notify
   - Default: send to all AUDITORs in the lab

3. Persist alert to Firestore:
   ```typescript
   interface AuditAlert {
     id: string;  // auto-generated
     labId: string;
     entryId: string;  // link to source audit entry
     operatorId: string;
     anomalyScoreId: string;
     severity: 'critical' | 'high' | 'medium';
     score: number;  // 0.0–1.0
     anomalyReasons: string[];  // from AnomalyScore.dimensions
     aiInsight?: AnomalyInsight;
     relatedEntries: string[];  // IDs of 5 most recent entries from this operator
     dismissedAt?: Timestamp;
     dismissedBy?: string;
     dismissalReason?: string;
     createdAt: Timestamp;
     recipientRoles: string[];  // who gets notified
   }
   ```

4. Integrate with onAuditTrailEntry trigger (from 07-02):
   ```typescript
   if (anomaly.overall > 0.85) {
     const alert = await generateAlert(anomaly, entry, insight);
     await routeAlert(alert, labId, entry.operatorId);
     await db.collection(`labs/${labId}/audit-alerts`).doc(alert.id).set(alert);
   }
   ```

5. Support dismissal:
   - Export function: `dismissAlert(labId: string, alertId: string, userId: string, reason: string): Promise<void>`
   - Verifies user is AUDITOR/admin
   - Updates alert: `dismissedAt = now()`, `dismissedBy = userId`, `dismissalReason = reason`
   - Log dismissal in a separate `audit-alert-dismissals` collection for compliance review

Implementation notes:
- Related entries: query last 10 audit entries for this operator, pick 5 most relevant (same module preferred)
- Severity thresholds tuned from Phase 7-02 (can be adjusted post-deployment based on false positive rate)
- Dismissal reason is required (enforced at API level) — free-form text, auditor-chosen

Per DICQ 4.4: alert dismissal must be logged for compliance audit.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Cloud Function → Alert Generation | Trusted: CF computes alerts, client cannot forge alerts |
| User → Alert Dismissal | Partially trusted: user submits reason, must be logged for audit |
| AUDITOR → Alert View | Trusted: AUDITOR role verified by Firestore rules |
| RT → Module-filtered Alerts | Trusted: RT can only see alerts for assigned modules (rules enforced) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-13 | Spoofing | User dismisses alert as another user | Mitigate | `dismissedBy` field captured from `request.auth.uid`; cannot be spoofed |
| T-07-14 | Tampering | Non-AUDITOR dismisses alert | Mitigate | Firestore rules enforce role check before update |
| T-07-15 | Repudiation | AUDITOR denies dismissal | Mitigate | Dismissal logged in separate collection with timestamp + user ID |
| T-07-16 | Info Disclosure | Non-AUDITOR reads alert | Mitigate | Firestore rules restrict reads to AUDITOR + admin + assigned RT |
| T-07-17 | Denial of Service | Alert deletion spam | Mitigate | Delete is forbidden in rules; only dismissal allowed (immutable create) |
| T-07-18 | Elevation | Non-admin modifies alert fields beyond dismissal | Mitigate | Rules allow update only for specific dismissal fields (`dismissedAt`, `dismissedBy`, `dismissalReason`) |

</threat_model>
