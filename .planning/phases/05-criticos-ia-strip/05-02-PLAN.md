
### REQ-02.2: Escalation Callable + State Machine

**File:** `functions/src/modules/criticos/escalacaoCriticos.ts`

**Callable signature:**

```typescript
callable: escalacaoCriticos(data: {
  laudoId: string;
  labId: string;
  criticalResults: Array<{
    analitoId: string;
    valor: number;
    threshold: CriticoThreshold;
  }>;
  patientPhone?: string;
  physicianEmail: string;
  operadorId: string; // Who created the laudo
})
```

**Transactional operations:**

1. Create escalacao document in `/labs/{labId}/criticos-escalacoes/{id}` with state `NEW`
2. Extract recipients from routing rules (Task 05-01)
3. For each recipient:
   - Create SMS job (if telefone)
   - Create email job (if email)
   - Create in-app notification (firestore message)
4. Set initial SLA deadline (current time + slaMinutos)
5. Return escalacao ID + status

**State machine:**

```
NEW → [SENDING_SMS | SENDING_EMAIL] → [SENT | FAILED] → ACKNOWLEDGED
```

**Escalacao document schema:**

```
/labs/{labId}/criticos-escalacoes/{id}
  ├─ laudoId: string
  ├─ criticalValues: Array<{ analitoId, valor, severidade }>
  ├─ escaladoEm: Timestamp
  ├─ recipients: {
  │   ├─ operadorIds: string[]
  │   ├─ telefones: string[]
  │   └─ emails: string[]
  │ }
  ├─ status: 'NEW' | 'SENDING' | 'SENT' | 'FAILED' | 'ACKNOWLEDGED' | 'EXPIRED'
  ├─ slaMinutos: number
  ├─ slaDeadline: Timestamp
  ├─ acknowledgedAt?: Timestamp
  ├─ acknowledgedBy?: string (operadorId)
  ├─ logs: Array<{
  │   ├─ timestamp: Timestamp
  │   ├─ action: 'SMS_SENT' | 'SMS_FAILED' | 'EMAIL_SENT' | 'EMAIL_FAILED' | 'ACKNOWLEDGED'
  │   ├─ detail: string
  │   └─ provider?: string (e.g., 'twilio', 'nodemailer')
  │ }>
  └─ deletadoEm?: Timestamp (soft-delete)
```

**Error handling:**

- SMS fails → log in `logs`, try email fallback
- Email fails → log, alert on dashboard (manual follow-up)
- Escalacao doc fails → wrap in try-catch, return error + escalacao ID (client retries)

**Acceptance criteria:**

- [ ] Transactional write: all-or-nothing escalacao creation
- [ ] State machine transitions logged
- [ ] Error logs captured (recipient, action, reason)
- [ ] SLA deadline calculated correctly
- [ ] E2E test passing (create escalacao, verify doc structure)

```typescript

  escalacaoId: string;
  labId: string;
  recipientePhone: string;
  criticoData: { analito: string; valor: number; referencia: string };
```

**Flow:**

```typescript

  body: smsText,
});
```

**Fallback (if Twilio fails):**

- Log error in escalacao doc
- Return error to caller
- Client retries via escalacao status dashboard

**Acceptance criteria:**

- [ ] SMS sends within 10 seconds

- [ ] Delivery rate >99% (quota alert at 98%)

### REQ-02.4: SLA Monitoring Cron Job

**File:** `functions/src/modules/criticos/escalacao-sla-monitor.ts`

**Trigger:** Cloud Scheduler, every 5 minutes

**Logic:**

```typescript
async function monitorEscalacaoSLA(context: ScheduledEvent) {
  const now = Timestamp.now();

  // Get all escalacoes with status != ACKNOWLEDGED and slaDeadline < now
  const overdue = await getAllLabs().then((labs) =>
    Promise.all(
      labs.map((lab) =>
        getFirestore()
          .collection('labs')
          .doc(lab.id)
          .collection('criticos-escalacoes')
          .where('status', '!=', 'ACKNOWLEDGED')
          .where('slaDeadline', '<', now)
          .get(),
      ),
    ),
  );

  // For each overdue escalacao:
  // 1. Mark status as 'EXPIRED'
  // 2. Send alert to lab admin (in-app notification)
  // 3. Log escalation failure in ciq-audit

  for (const doc of overdue) {
    await updateEscalacao(doc.id, {
      status: 'EXPIRED',
      logs: [
        ...doc.logs,
        {
          timestamp: now,
          action: 'SLA_EXPIRED',
          detail: `SLA exceeded by ${now.seconds - doc.slaDeadline.seconds}s`,
        },
      ],
    });

    // Send admin alert
    await notifyAdminSLABreach(doc.ref.parent.parent.id, doc.id);
  }
}
```

**Alert conditions:**

- Red: SLA expired (status = EXPIRED)
- Amber: SLA deadline within 5 minutes
- Green: SLA met

**Acceptance criteria:**

- [ ] Cron runs every 5 minutes (no missed runs)
- [ ] Overdue escalacoes detected correctly
- [ ] Admin alerts sent (in-app notification)
- [ ] Performance: <10s for 100 labs (batch query optimization)

### REQ-02.5: SLA Dashboard Component

**Features:**

1. **Status summary:** Count by status (NEW, SENDING, SENT, ACKNOWLEDGED, EXPIRED)
2. **SLA table:** List escalacoes with:

**Design:**

- Dark-first Tailwind

- Pagination (load 50 at a time, lazy-load on scroll)

**Acceptance criteria:**

- [ ] Dashboard loads escalacoes in real-time
- [ ] SLA countdown updates every second
- [ ] Status badges update instantly
- [ ] WCAG AA contrast + keyboard nav
- [ ] Performance: <1s initial load for 200 escalacoes
