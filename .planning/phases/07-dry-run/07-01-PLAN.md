**Execution Flow:**

**Block A (Organização e Responsabilidade)** — ~20 items, 1.5 hours
- Walk through organizational structure, contracts, hierarchy
- RT/Auditor assess each item: Conforme / Não-conforme / N.A.
- For each "Não-conforme" item:
  - Auditor describes what was observed (evidence: photo or note)
  - Operator enters finding in Achado form (description, severity: maior/menor/observação)
  - Upload photo if applicable
  - Save → if severity ≥ maior, auto-creates NC in main system
- Rest period: 10 min (every 30 items)

**Blocks B-J (Pessoal, Qualificação, Equipamento, Métodos, CIQ, Comunicação, Confidencialidade)** — ~95 items, 3.5-4.5 hours
- Same pattern as Block A
- Track time: should maintain ~2 min per item average (115 items / 6 hours = 1.9 min/item)
- Auditor leads discussion; Operator enters responses; RT observes and validates

**Evidence Capture Strategy:**
- For each critical non-conformance: photo of area/document/process involved
- For observational findings: typed notes in Achado form
- Evidence automatically stored with LogicalSignature (hash + operatorId + timestamp)
- All evidence linked to sessao + achado record (immutable chain-hash per RDC 978)

**Real-Time System Integration:**
- Each registered Achado (severity ≥ maior) triggers Cloud Function registerAchado callable
- Callable auto-creates Non-Conformidade in /labs/{labId}/naoConformidades/
- Auditor verifies NC appeared in main system (lag <2 sec)
- If N.A. item: recorded but no NC created (expected behavior)

**Severity Mapping (during execution):**
- "Conforme": Item meets requirement (no NC created)
- "Não-conforme Menor": Weakness found, low risk (NC created, priority: medium)
- "Não-conforme Maior": Gap requiring action, audit finding (NC created, priority: high, requires CAPA)
- "N.A.": Not applicable (e.g., lab doesn't perform this test type) — recorded but no action

**Breaks & Debrief:**
- Break after Block D (halfway point), ~10 min
- Running tally: update docs/AUDIT_DRY_RUN_2026-05.md with interim totals (every 2 blocks)
  - Items assessed so far: X / 115
  - Conforme count, Não-conforme count, N.A. count
  - Critical findings count
- Final debrief: 30 min after all items reviewed
  - Auditor & RT discuss findings trends
  - Identify top 3-5 priority areas for CAPA
  - Confirm all critical NCs auto-created

**Sign-off During Execution:**

- Session marked complete when all 115 items responded

- In NC detail, navigate to "Plano de Ação" tab (or "CAPA" section)
- Fill out:
  - **Ação Corretiva (Corrective Action):** What will be done to correct the gap
    - Example: "Implement change log for POP v2.3 → document rationale + approval date"
    - Example: "Schedule calibration of thermometer Z-100 with external lab"
    - Must be specific (not "fix it")
  - **Responsável (Owner):** Who will execute (person name, role)
    - Should be section lead or manager (not coordinator)
    - Required to sign CAPA commitment
  - **Prazo (Deadline):** When action will be complete
    - Logic: Critical gaps → 30 days (by June 5, 2026)
    - Moderate gaps → 60-90 days (Phase v1.3 remediation window)
  - **Critério de Eficácia (Effectiveness Criteria):** How will success be measured
    - Example: "New POP version documented + approved by RT + communicated to team"
    - Example: "Calibration certificate dated + uploaded to equipment file"
    - Must be verifiable (not subjective)
  - **Data de Conclusão Planejada (Planned Close Date):** Auto-filled from Prazo
  
- Auditor reviews CAPA entry against finding description (ensure alignment)
- RT validates CAPA is achievable (consults with Responsável if needed)
- CAPA saved → links to NC record

|----------|-----------|------|
| Auditor ↔ Firestore | Audit findings entered on-site via tablet UI | Tampering: auditor maliciously false-flags items; integrity: LogicalSignature immutable, all changes timestamped |
| RT ↔ Audit Report | RT signs final PDF with cryptographic signature | Repudiation: RT claims audit was not authorized; mitigation: LogicalSignature + email confirmation before signing |
| External Auditor ↔ Evidence | Third-party reviews audit evidence (photos, notes) | Disclosure: sensitive internal process images; mitigation: signed URLs (7-day expiry), lab must control distribution |
| Cloud Storage ↔ Retention | Audit records stored 5 years | Availability: storage bucket deleted by accident; mitigation: GCP backup policy, no deletion allowed for audits folder |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Spoofing (Operatorid) | registerAchado callable | Mitigate | Only signed-in users (request.auth.uid) can call. LogicalSignature captures actual operatorId at server (not trusting client). |
| T-07-02 | Tampering (Achado modification) | Firestore document | Mitigate | No client direct write (callable-only). Firestore rules deny update/delete. Soft-delete only via callable. Chain-hash immutable. |
| T-07-03 | Repudiation (Auditor claims not authorized) | RT signature on PDF | Mitigate | LogicalSignature tied to request.auth.uid (signed in user). Email log sent on report generation. |
| T-07-04 | Information Disclosure (Evidence photos) | Cloud Storage signed URLs | Mitigate | 7-day expiry on shareable links. Lab controls who receives URL. Access logs in Cloud Storage. |
| T-07-05 | Denial of Service (Audit data deleted) | Firestore/Storage | Mitigate | Soft-delete enforced (no hard delete). GCP retention policies (5-year minimum). Backup available via Cloud SQL. |
| T-07-06 | Elevation of Privilege (Non-auditor modifying) | Firestore rules | Mitigate | `isActiveMemberOfLab(labId) && hasRole('auditor')` gate on registerAchado callable. Role checked at token claim (JWT). |
