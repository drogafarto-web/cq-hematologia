KEY FEATURES
================================================================================

Feature Flag System
-------------------
• Gradual rollout: 0% → 25% (May 24) → 50% (May 29) → 100% (June 6)
• Deterministic bucketing: labs assigned via SHA-256(labId) hash
• Whitelist override: internal labs enabled first
• Server-side + client-side evaluation
• Fast disable: set enabled=false in Firestore (no code rollback needed)
• Time to disable: <1 minute

Monitoring (4 Dashboards, 5 Alerts)
------------------------------------
Dashboard 1: Portal Auth Health
  • Auth success rate, login latency, email delivery, failed attempts

Dashboard 2: NOTIVISA Queue Health
  • Queue status, processing latency, success rate, stuck entries, feature flag state

Dashboard 3: Firestore Access
  • Patient/laudo read latency, rule rejections, index usage

Dashboard 4: System Health Overview
  • Error rate, function execution time, quota usage, error budget, uptime

Alert Policies (5 Total)
  P1: Portal Auth Failures (>5 in 5 min) — SMS page + Slack
  P1: NOTIVISA Queue Stuck (>0 entries >15 min) — SMS page + Slack
  P2: Rule Rejections (>10 in 5 min) — Email + Slack
  P2: Email Delivery Failure (>5% failure rate) — Email + Slack
  P3: Function Latency (p90 >3s) — Email only

Rollback Strategy
------------------
Fast path (feature flag disable):
  • Time: <1 minute
  • Effect: All submitNotivisa calls return "FEATURE_DISABLED" immediately
  • No code rollback needed

Code rollback path (if flag disable insufficient):
  • Hosting rollback: 1 min
  • Rules rollback: 2 min
  • Functions rollback: 5 min
  • Full rollback: 10 min
