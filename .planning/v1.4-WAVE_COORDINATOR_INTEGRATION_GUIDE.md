# v1.4 Wave Coordinator Integration Guide

**For:** Wave 1–4 coordinators (Agents leading each wave)  
**Purpose:** Your role in real-time integration validation during Phases 4–15  
**Read time:** 10 min  
**Reference:** `.planning/v1.4-INTEGRATION_VALIDATOR.md` (master spec)

---

## Your Role

You are responsible for:

1. **Reporting integration status daily** (standup 09:00 BRT)
2. **Flagging integration blockers immediately** (not weekly)
3. **Validating schema/callables/rules before merging** (with validator help)
4. **Coordinating cross-wave dependencies** (with other wave leads)
5. **Supporting integration test execution** (your phase owns the tests)

## Integration Blocker Escalation (Immediate)

**If you discover a blocker:** Don't wait for standup. Escalate immediately.

### Blocker Definition

- Schema conflict (two phases modifying same collection)
- Callable signature regression (callable changes break consuming phase test)
- Permission denial (callable blocked by Firestore rules, not in spec)
- Test data incompatibility (fixture breaks your phase's tests)
- Auditor feedback contradicts code (Phase 11 meeting reveals compliance issue)
- Cascading delay (your phase blocked by upstream, blocking downstream, critical path at risk)

### Escalation Steps

1. **Slack message:** Post to `#hc-quality-integration` with:
   - Wave + Phase
   - 1-line blocker title
   - Est. impact (which phases affected, hours of delay)
   - Your proposed mitigation
2. **Tag:** Wave coordinator tag (yourself), Validator tag, CTO tag (if critical)
3. **Wait:** Validator will create BLOCK-\* entry and route to Wave coordinators + CTO
4. **Follow-up:** Escalation meeting scheduled within 2h (critical) or 24h (medium)

### Escalation Template (Slack)

```
🔴 CRITICAL BLOCKER — Wave [1/2/3/4] Phase [X]

[1-line title]

Impact: [which phases blocked + hours]
Root cause: [schema | callable | rules | fixture | auditor feedback]
Evidence: [PR link | error log | meeting notes]
Proposed fix: [option A | option B]
Timeline: [unblock by YYYY-MM-DD]
```

## Schema/Callable/Rules Changes (How to Coordinate)

### Before merging any code that changes:

- Firestore collections (new or modified)
- Callable signatures (parameters or return types)
- Firestore rules (new blocks or modified permissions)
- Composite indexes

**Follow this process:**

1. **Check callable signatures JSON** (`.planning/v1.4-CALLABLE_SIGNATURES.json`)
   - Is your callable in the spec?
   - Does your code match the spec signature?
   - If no: discuss with Validator, get CTO approval, update spec

2. **Run schema diff check** (before opening PR)

   ```bash
   git diff main -- firestore.rules  # Are rule changes expected?
   git diff main -- functions/src/**/*Callable.ts  # Any signature regressions?
   ```

3. **Notify Validator** (in PR description)

   ```markdown
   ## Integration Changes

   - [ ] New callable: `myNewCallable(params) → returns` — consuming phases: [list]
   - [ ] Modified rule: Block `my-collection` — permission impact: [describe]
   - [ ] New index: `my-collection`, fields: [list] — query optimization
   - Integration test: Scenario [N] validates this
   ```

4. **Integration test coverage**
   - Which E2E scenario validates this change?
   - Is that scenario test updated/added?
   - Tag Validator to review test coverage

5. **Merge gate:** Validator sign-off on integration test coverage
   - Validator checks: new callable/rule covered by Scenario N? Yes → approve
   - If no coverage: request test addition before merge

6. **Post-merge:** Validator runs Scenario N integration test to validate

## Callable Signature Stability (Non-Negotiable)

**Once your phase ships to production, callable signatures are FROZEN.**

### What locked means:

- ✅ Can add optional parameters (with defaults)
- ✅ Can add new fields to return object
- ✅ Can add new error codes (consumer must handle gracefully)
- ❌ Cannot remove parameters (breaking change)
- ❌ Cannot change return type (breaking change)
- ❌ Cannot rename fields in return (breaking change)

### Example: Phase 4 ships `signLaudo` callable

```json
{
  "params": { "laudoId", "labId", "rtId", "signatureHash" },
  "returns": { "success", "laudoId", "estado", "assinatura" }
}
```

**Phase 6 (3 weeks later) wants to call it:**

- ✅ Adding optional param to `signLaudo` in Phase 6? OK (backward compatible)
- ❌ Renaming `signatureHash` → `hash` in Phase 4? BLOCK — breaks Phase 6

### If you need a breaking change:

1. Create new callable version: `signLaudoV2` (keep original)
2. Deprecate original (Phase 4-02 UI switches to V2)
3. Give downstream phases 1 sprint to migrate
4. Remove original after deprecation window

## Cross-Wave Dependency Coordination

### Critical dependencies (Wave 1 → Wave 2 → Wave 3):

**Wave 1 → Wave 2:**

- Phase 4 laudo signature format must be stable (Phase 6 PDF depends on it)
- Phase 5 critical escalation SLA contract must be locked (Phase 9 KPI depends on it)
- **Action:** Wave 1 coordinator: confirm signatures locked before Phase 6 kickoff (Jul 1)

**Wave 2 → Wave 3:**

- Phase 6–7 must complete before Phase 8 Plans 05–07 (auditor ceremony needs evidence)
- **Action:** Wave 2 coordinator: confirm Phases 6–7 smoke tests pass before Jul 12 (Phase 8 Plan 05 start)

**Wave 3 ← Waves 1–2:**

- CAPA closure (Phase 8) must wait for Phase 4–7 to complete (findings require working implementations)
- **Action:** Wave 3 coordinator: don't start Phase 8 Plans 05–07 until Phases 4–7 are production-ready

**Wave 4 → All waves:**

- Phase 10 pen-test findings (Jun 21) trigger remediation in Phases 4–8
- **Action:** Wave 4 coordinator: notify all wave coordinators immediately when findings arrive. Create remediation checklist per phase.

- [ ] Wave 2: Waiting on Wave 1? Unblock date confirmed?

## CTO Sign-Offs You May Need

**When to request CTO review:**

1. **Callable signature change** (any breaking change)
2. **New Firestore rule block** (permission policy change)
3. **New composite index** (data model change)
4. **Phase timeline slip** (affects critical path)
5. **Auditor feedback contradiction** (phase 11 feedback contradicts your design)
6. **Security finding (critical)** (Phase 10 pen-test critical finding in your phase)

**CTO review SLA:** 2–4 hours (critical), 24 hours (medium)

**Request format (Slack #hc-quality-integration):**

```
@CTO — Review needed

[Category]: [Title]
Impact: [which phases, which dates]
Evidence: [PR link | meeting notes | test result]
Options: [A | B]
Recommendation: [your take]
Decision deadline: [YYYY-MM-DD HH:MM BRT]
```
