48 requirements across 4 parallel streams (A–D), mapped to 15 phases.

**Milestone:** v1.4 — Compliance Closure + Portal Expansion + IA Strip OCR Foundation  
**Target Duration:** 22 weeks (2026-05-07 → 2026-09-30)  
**Deadline:** 2026-09-30 (6 weeks before external audit 2026-10-15)

**Key metrics:**
- Target DICQ: 88–92% (baseline 78–82%)
- Target RDC 978: 100% Arts. 117, 167, 179-191, 204
- Target Web Vitals: LCP <2.5s, INP <200ms, CLS <0.1
- New modules: Patient Portal, IA Strip OCR (foundation), NOTIVISA integration
- New Cloud Functions: 50+

## Section 1: User Stories (v1.4 Must-Have Features)

### REQ-401: Advanced KPI Dashboarding (Department + Equipment Slice)

**User story:** As a QA manager, I want to slice KPI data by department, equipment, operator, and reagent lot, so that I can identify systemic quality trends, correlate failures to root causes, and target training interventions.

**Acceptance criteria:**
- [ ] Dashboard accepts ≥3 independent filter dimensions (equipment, operator, lot, analyte)
- [ ] Filtering applies in real-time (<500ms response, <3KB network round-trip)
- [ ] Results persist filter state in URL (shareable deep links)
- [ ] Each slice shows turnaround%, retrabalho%, NC rate, trend sparklines (30d + 90d)
- [ ] Export to PDF preserves all dimensions with timestamp
- [ ] WCAG AA accessible (keyboard nav, screen reader compatible)
- [ ] Performance: LCP <1.5s for dashboard load, CLS <0.05

**Effort:** Medium (16–20 pts) | **Phase assignment:** v1.4 Phase 1  
**DICQ mapping:** 4.12.1 (Plano de Melhoria), 4.14.7 (Indicadores de Qualidade)  
**RDC mapping:** Art. 86 (Programa de Garantia da Qualidade — GCQ)

- [ ] Audit trail: who accessed, when, signature on approval

**Effort:** Large (24–28 pts) | **Phase assignment:** v1.4 Phase 1

## Section 2: v2 Parking Lot (Out of Scope v1.4, Strategic Vision)

These features extend beyond v1.4 scope but chart direction toward market leadership.

### REQ-501: OCR para Imunologia (Strip Reading IA)

**Vision:** Integração com Google Gemini 2.5 vision para leitura automática de strips de teste rápido (HIV, dengue, hCG, sífilis, COVID), com classificação ≥95% acurácia, TTL <3s, armazenamento de foto + resultado no audit chain.

**Rationale for deferral:** Requer QA/validação do modelo em Riopomba (3–4 semanas de testes clínicos), integração de câmera no mobile PWA, e aprovação de regulação (Anvisa pode exigir validação clínica). v1.4 foca consolidação compliance; v2 leva diferencial comercial.

**Dependencies:** Mobile PWA hardening (done v1.3), Gemini integration (done), test dataset (Riopomba to provide).

**Estimated effort:** 3–4 weeks engineering + 4 weeks QA validation

---

### REQ-502: Multi-Tenant Real (N Labs, 1 Deploy)

**Vision:** Escalar de single-tenant `labclin-riopomba` para N labs independentes no mesmo deploy, com isolamento por `labId` em rules, claims, e UX (logo, tema, configurações por lab).

**Rationale for deferral:** v1.4 foca fechar compliance gaps do laboratório atual. Multi-tenant exige audit de rules (2–3 semanas), refactor de seeding/setup, e pricing model definition (SaaS recurring). v2 torna-se modelo de receita.

**Dependencies:** v1.4 compliance solidified (audit-proof rules), billing system (not yet designed).

**Estimated effort:** 3–4 weeks engineering (rules audit + UX), 2 weeks QA

---

### REQ-503: NOTIVISA Auto-Submission

**Vision:** Completar REQ-410 com native API call ao Anvisa NOTIVISA portal, eliminating manual submission e returning receipt code directly to app.

**Rationale for deferral:** Anvisa NOTIVISA API documentação é limitada + requer certificado digital (e-CNPJ). v1.4 implementa offline form geração (audit-safe); v2 automatiza ponta-a-ponta.

**Dependencies:** NOTIVISA form generation (v1.4 REQ-410), digital certificate provisioning (legal decision pending).

**Estimated effort:** 2–3 weeks engineering (API integration + testing)

---

### REQ-504: Multi-Equipamento Hematologia (Mindray, Sysmex)

**Vision:** Estender suporte além do Yumizen H550 para Mindray BC-6800 e Sysmex XN-L, com mapping automático de canais, validação de intervalo referência por analisador, e CIQ/CEQ Westgard polimórfico por fabricante.

**Rationale for deferral:** Cada analisador tem interface proprietária (protocolo, formato CSV/HL7 distinto). v1.4 consolida Yumizen como referência; v2 expande mercado via portabilidade de equipamento.

**Dependencies:** Yumizen module hardened (done), equipment abstraction layer (not yet designed), test datasets for Mindray/Sysmex (partner to provide).

**Estimated effort:** 4–5 weeks engineering per equipment (protocol + validation)

---

### REQ-505: Integrações LIS (Middleware)

**Vision:** Middleware (serverless Cloud Function) consumindo HL7 v2.4 / FHIR de LIS terceiro (ex. Sysmex SH/D, Bionexo), auto-populando order entry, correlacionando resultados, e bidirecionais sincronizando laudos de volta ao LIS.

**Rationale for deferral:** Design de integração exige conversa com clientes sobre LIS target + formato dados. v1.4 assume entry manual; v2 torna-se diferencial operacional.

**Dependencies:** Order entry baseline (out-of-scope v1.4 per current DICQ), HL7 parser library, LIS vendor APIs.

**Estimated effort:** 3–4 weeks engineering per LIS target (partner collaboration heavy)

---

### REQ-506: Formação de Consórcio Laboratorial (Multi-Site)

**Vision:** Suportar 1 grupo controlando N laboratórios (ex. Riopomba + filiais), com consolidação de KPIs, conformance comparativo (lab A vs lab B), e plano de auditoria inter-sites.

**Rationale for deferral:** Exige hierarquia de usuários (global admin, lab admin, operator), schema replicado por site, e decisão de billing (por lab vs grupo). v1.4 é single-lab; v2 abre modelo franchise.

**Dependencies:** Multi-tenant (REQ-502), RBAC refinement, billing architecture.

**Estimated effort:** 4–5 weeks engineering (hierarchy + KPI aggregation)

---

### REQ-507: App Auditor (Read-Only Portal)

**Vision:** Aplicação dedicada (PWA ou nativo) pra auditor da Vigilância — acesso read-only a seleção de docs (audit trail, CIQ, CEQ, NC, CAPA, pessoal), compartilhado via link temporário (24h), sem acesso a dados sensíveis (email, endereco, dados clínicos).

**Rationale for deferral:** Exige design de permissioning granular, portal separado, e policy de compartilhamento. v1.4 assume auditor lê sistema full (com NDA); v2 democratiza acesso.

**Estimated effort:** 3–4 weeks engineering (portal + access control)

## Appendix: Glossary & Abbreviations

- **DICQ 4.3** = Diretrizes Integradas de Conformidade da Qualidade, Bloco 4 (Gestão da Qualidade). Norma de acreditação via SBAC.
- **RDC 978/2025** = Resolução Anvisa em vigor desde 2026-06-10 (com alterações RDC 986 em 2025-08).
- **PGQ** = Programa de Garantia da Qualidade (6 componentes obrigatórios RDC 978 Art. 86)
- **CIQ** = Controle Interno da Qualidade (Westgard rules, Levey-Jennings)
- **CEQ** = Controle Externo da Qualidade (ensaio de proficiência)
- **NC** = Não-Conformidade (desvio de especificação)
- **CAPA** = Ação Corretiva e Preventiva (RCA + action plan)
- **RT** = Responsável Técnico (qualificação legal obrigatória RDC 978)
- **NR-32** = Norma Regulamentadora (segurança ocupacional, perfurocortantes, etc.)
- **PGRSS** = Plano de Gerenciamento de Resíduos Sólidos de Saúde (RDC 222/2018)
- **DPIA** = Data Protection Impact Assessment (LGPD Art. 38)
- **TTI** = Time to Interactive (Web Vitals performance metric)
- **Westgard Rules** = Statistical process control rules for QC (4-1s, 2-2s, R-4s, etc.)
