## Schema

| Field               | Description                                            |
| ------------------- | ------------------------------------------------------ |
| Risk ID             | RISK-401 onwards (immutable identifier)                |
| Title               | Short descriptor (≤8 words)                            |
| Description         | Full context — what, why, how it surfaces              |
| Probability         | Low (1) / Medium (2) / High (3)                        |
| Impact              | Low (1) / Medium (2) / High (3)                        |
| Risk Score          | P × I (1–9)                                            |
| Trigger             | Observable event that surfaces the risk                |
| Mitigation          | Active steps to reduce probability/impact (preventive) |
| Contingency         | If risk materializes, what we do (reactive)            |
| Owner               | Single accountable person                              |
| Escalation Criteria | Threshold to escalate to CTO                           |
| SLA                 | Time-to-resolution target                              |
| Status              | Active / Mitigated / Materialized / Closed             |
| Last Review         | ISO date                                               |

**Severity bands:**

- 🟢 **1–3 (Low):** Phase Lead handles; review weekly.
- 🟡 **4–6 (Medium):** QA Lead engaged within 48h.
- 🔴 **7–9 (High):** Immediate CTO escalation; daily review until score reduced.

## Weekly Review Process

**Cadence:** Every Monday, 09:30 BRT, 30 min
**Attendees:** CTO, QA Lead, Stream Leads (A–D)
**Agenda:**

1. (5 min) Status updates on all 🔴 + top 5 🟡 risks
2. (10 min) Trigger events fired this week (review evidence)
3. (10 min) New risks identified (add to register)
4. (5 min) Closure proposals (move risks to Closed status)

**Status Transitions:**

- **Active → Mitigated:** Score reduced by ≥50% via mitigation; trigger no longer imminent
- **Active → Materialized:** Trigger fired; contingency activated; track as incident
- **Mitigated → Closed:** Mitigation complete + 2-week observation window with no recurrence
- **Materialized → Closed:** Incident resolved + post-mortem documented + preventive action deployed

**Status update format (per risk):**

```
RISK-XXX (current score) → (proposed score) | Owner | Last action: [date + summary] | Next action: [date + plan]
```

---

## Closure Criteria

A risk is **CLOSED** when ANY of the following:

1. **Trigger condition no longer applicable** — phase complete, dependency removed, scope changed
2. **Mitigation reduced score to 1–2** — sustained for 2-week observation window
3. **Phase completion makes risk obsolete** — e.g., RISK-402 closes when Phase 4 (CAPA) signed off
4. **Contingency executed successfully** — risk materialized but resolved per plan; lessons learned documented
5. **Risk superseded** — replaced by more specific risk via decomposition

**Closure procedure:**

1. Owner proposes closure in weekly review
2. CTO + QA Lead approve (joint sign-off for 🔴; QA Lead alone for 🟡; Phase Lead alone for 🟢)
3. Last review date stamped
4. Move to "Closed Risks" section (preserved for audit trail; never deleted)
5. Lesson learned added to `v1.4-CLOSURE-MEMO.md` if instructive
