## ADR-0009: React 19 + TypeScript 5.8 — Version Lock Strategy

**Context:** No forced version upgrades in v1.4 to maintain timeline stability and reduce context-switching.

**Decision:**

- React 19 locked (no upgrade to React 20, even if available).
- TypeScript 5.8 locked (no TS 5.9 / 6.0).
- Vite 6.0, Zustand 5.0, Firebase v12, Node 22 all pinned.
- `package.json` uses exact versions (no `~` or `^`).

**Rationale:** v1.4 timeline is fixed (auditoria 2026-10-15). Breaking changes in major library upgrades would derail DICQ compliance + feature delivery.

**Consequence:** Tech debt accumulates; v1.5 will have "upgrade week" (~2-3 weeks) to stage React 20, TS 6.0, Node 26. Planned + budgeted.

**Gate:** 2026-06-07 (1-month checkpoint: confirm v1.3 + early v1.4 stack stability).

## ADR-0010: Gemini Vision API as IA Baseline

**Context:** v1.4 needs IA foundation for imuno strip OCR. Question: fine-tuned model now or Gemini 2.5 Flash baseline?

**Decision:**

- Use Gemini 2.5 Flash (baseline) for v1.4 Phase 11.
- Collect 500+ strip images (labeled dataset).
- Fine-tuning deferred to v1.5 (when dataset is mature + model ROI is clear).

**Rationale:** Gemini is production-ready (0.5–2s latency). Fine-tuning would consume 6+ weeks of v1.4 runway; riskier. Gemini baseline achieves ~85% accuracy (literature); v1.5 fine-tuning can target 95%+.

**Consequence:** v1.4 IA baseline is API-dependent (Google); latency/cost trade-offs accepted. v1.5 can evaluate fine-tuned model (proprietary vs Vertex AI vs open-source).

**Gate:** 2026-07-07 (Phase 11 mid-point: confirm Gemini API stability + dataset collection pace).

## ADR-0011: Single-Lab Deployment Model (v1.4)

**Context:** HC Quality is architectured multi-tenant-ready but deployed single-lab (Riopomba). Expand to multi-tenant now or defer?

**Decision:**

- Maintain single-lab model v1.4 (hardcoded `labId=Riopomba`).
- Firestore schema stays multi-tenant-compatible (for v1.5).
- Multi-tenant full architecture (RBAC, billing, onboarding) deferred to v1.5+.

**Rationale:** Multi-tenant requires 10–14 weeks (Rules audit, billing, E2E testing). v1.4 timeline doesn't accommodate. Single-lab is market-viable (case study: Riopomba DICQ accredited). v1.5 multi-tenant unblocks market scale (10+ labs).

**Consequence:** v1.4.1+ customer acquisition requires new lab instance or v1.5 multi-tenant readiness. Acceptable; v1.5 has clear roadmap.

**Gate:** 2026-07-07 (Phase 6 mid-point: confirm single-lab stability; no cross-tenant data leaks). v1.5 Planning Gate: 2026-11-01.

## ADR-0012: RDC 978 Audit Trail via LogicalSignature

**Context:** RDC 978 Art. 117 exige rastreabilidade ponta-a-ponta. How to prove audit trail integrity?

**Decision:**

- Extend v1.3 LogicalSignature (HMAC + SHA-256 + chain-hash) to all audit events in v1.4 (CAPA, NOTIVISA, personnel, calibração, liberação).
- Universal pattern: client creates `pending`, CF seals with HMAC, rules enforce immutability.
- Rules template + shared factory function for reuse across modules.

**Rationale:** Cryptographic audit chain is mathematically rigorous (auditor can verify integrity independently). Competitors use weak logging; HC Quality is world-class. Compute cost is negligible (~$1.20/day).

**Consequence:** Latency adds 200–500ms per sealed event (CF cold start + HMAC calc). Mitigated: UX shows spinner. Debugging complexity increases (stale `pending` docs need recovery procedure).

**Gate:** 2026-06-21 (Phase 6 completion: confirm all v1.4 modules seal events correctly).

## ADR-0013: DICQ 85% Target Sequencing (Waves 1–4)

**Context:** DICQ baseline is 78–82%. Target 85%+ by v1.4 EOL. How to sequence 15 phases across 22 weeks?

**Decision:**

- **Wave 1 (Weeks 1–3):** Foundation (v1.3 stabilization + v1.4 planning + schema extensions). Phases 1–3.
- **Wave 2 (Weeks 4–8):** Mandatory DICQ blocks (CAPA closure, patient portal, critical values, satisfação). Phases 4–7. **Target: 80–82%** (intermediate).
- **Wave 3 (Weeks 9–15):** Stretch DICQ blocks (NOTIVISA, documentation, multi-equipment, IA foundation). Phases 8–11. **Target: 85–87%** (achieved).
- **Wave 4 (Weeks 16–22):** Final audit + launch prep (performance, security, smoke tests, production deploy). Phases 12–15. **Target: ≥85%** (locked).

**Rationale:** Wave structure balances parallelism (Streams B–D independent in Waves 2–3) with critical-path protection (Stream A CAPA blocks Waves 1–2). DICQ incremental milestones (80% → 85% → ≥85%) provide checkpoint visibility.

**Consequence:** Limited flexibility if Wave 2 slips (Wave 3 is dependent). Escalation protocol handles slips (Phase 14 extension if needed).

**Gate:** Week 8 (Wave 2 EOD): Verify Blocks A, C, G, H at target %. Week 15 (Wave 3 EOD): Verify overall DICQ ≥85%.

## ADR-0014: NOTIVISA Integration (Sandbox → Production)

**Context:** RDC 978 Art. 66 mandates notification of adverse events to MS via NOTIVISA. Implement full API now or defer?

**Decision:**

- v1.4 Phase 8: Form generation (Sandbox mode). Zod schema validation, RT approval workflow, audit trail, PDF export.
- v1.5+ (when certificate is provisioned): Real Anvisa API submission (requires e-CNPJ certificate; legal/fiscal process ongoing).
- Code is written sandbox-compatible; v1.5 swap is localized (mock function → real HTTP call).

**Rationale:** Certificate provisioning is blocker (4+ weeks legal/fiscal work outside engineering). Form generation achieves Art. 66 intent (captured + audited). v1.5 production submission is clean upgrade (1–2 weeks).

**Consequence:** Art. 66 is "preparatory compliant" (form + approval) not "fully submitting" until v1.5. Auditor may require manual workaround (RT exports PDF → submits to Anvisa portal manually). Acceptable interim.

**Gate:** 2026-07-23 (Phase 8 mid-point: confirm form generation + audit export working). v1.5 Gate: 2026-11-01 (assess certificate status).

## ADR-0015: Patient Portal — Email-Link Auth (LIS Deferred)

**Context:** v1.4 patient portal needs authentication. Requires LIS integration (patient data sync) or can fallback to email-link?

**Decision:**

- v1.4 Phase 5: Email-link auth (simple, timeline-safe). Patients enter CPF → receive email link (72h expiry) → download laudo.
- Patient data: Manual entry by RT or CSV import (labor-intensive but sufficient for v1.4).
- v1.4.1 (2–3 weeks post-launch): LIS integration (patient sync, self-service onboarding). Deferred.

**Rationale:** LIS integration is 4–6 weeks (design + middleware + testing). Blocks Phase 5 launch (late Week 8 instead of Week 5). Email-link auth is proven + simple; portal ships on-time.

**Consequence:** Patient experience is slightly less polished (manual patient add, email link per download). v1.4.1 fixes UX (auto-sync from LIS). Backward-compat: email-link continues to work post-LIS-sync.

**Gate:** 2026-06-18 (Phase 5 mid-point: confirm email-link auth working + patient CSV imported). v1.4.1 Planning: 2026-10-15.
