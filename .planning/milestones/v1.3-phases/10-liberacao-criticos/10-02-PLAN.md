### Task 2: criarLaudo callable (2 dias)

- `functions/src/liberacao/criarLaudo.ts`:
  - Auth: `isActiveMemberOfLab` + claim `tecnico` ou `rt`
  - Input Zod:
    ```typescript
    {
      labId, runIds: string[], pacienteId, medicoSolicitanteId, exames: ExameLaudo[]
    }
    ```
  - Pipeline (Firestore transaction):
    1. Lê runs aprovadas + médico solicitante (cache local) + exameConfig
    2. Detecta criticoFlag (server-side: passa thresholds + resultados)
    3. Classifica laudo (rotina/revisao-rt/bloqueio-critico)
    4. Roda `shouldAutoRelease`:
       - Se TRUE: cria Laudo em status `Auto-Liberado` + cria LaudoVersion v1 com signature `Sistema` + audit log
       - Se FALSE: cria Laudo em status `Pendente` (espera RT)
    5. Retorna `{ laudoId, status, autoReleased: bool, version: number? }`
  - Idempotência: hash do payload (labId+runIds+pacienteId) — se já existe, retorna existente
  - Region: `southamerica-east1`, memory: `512MiB`, timeout: `60s`

**Acceptance:** 100 invocações concorrentes não duplicam laudo; chainHash sequencial sem buracos.

### Task 3: liberarLaudo callable (2 dias)

- `functions/src/liberacao/liberarLaudo.ts`:
  - Auth: `isActiveMemberOfLab` + claim `rt` ou `rtSubstituto`
  - Input Zod: `{ labId, laudoId, signaturePayload: { hash, operatorId, timestamp }, observacao? }`
  - Pipeline (Firestore transaction):
    1. Lê laudo + última versão
    2. Valida transição: `validateTransition(currentStatus, 'Liberado', role)`
    3. Recalcula chainHash server-side (não confia no client)
    4. Cria LaudoVersion v(N+1) com snapshot atual + signature server-side
    5. Update laudo: `status: 'Liberado'`, `currentVersion: N+1`
    6. Audit log
  - Retorna `{ laudoId, version, status, chainHash }`

**Acceptance:** Race condition test: 2 RTs liberando o mesmo laudo simultaneamente → 1 sucesso + 1 erro `already-liberated`.
