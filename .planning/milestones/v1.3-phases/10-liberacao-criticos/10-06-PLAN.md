### Task 2: liberacao.spec.ts — 8 fluxos críticos (2 dias)

**Fluxo 1 — Criar laudo a partir de runs aprovadas (técnico):**

- Login técnico → /liberacao/criar → seleciona runs aprovadas → médico solicitante → submit
- Asserir: laudo criado, status `Pendente`, classificação detectada

**Fluxo 2 — Auto-libera rotina sem crítico:**

- Seed laudo classificado `rotina`, sem Westgard reject, sem crítico
- Trigger criarLaudo → asserir auto-released, status `Auto-Liberado`, version 1 criada, signature operatorRole = 'Sistema'

**Fluxo 3 — RT libera manualmente exame revisao-rt:**

- Seed laudo classificado `revisao-rt`
- Login RT → /liberacao → abre ReviewLaudoModal → revisar → liberar com PIN
- Asserir: status `Liberado`, version 1 criada com signature RT

**Fluxo 4 — RT rejeita laudo:**

- Login RT → ReviewLaudoModal → rejeitar com motivo → laudo volta para `Pendente` com flag `revisao-pendente`
- Asserir: estado correto, audit log presente

**Fluxo 5 — Liberar com crítico bloqueia até comunicar:**

- Seed laudo crítico (severidade alta)
- RT libera → asserir: status `Liberado`, mas `criticoBloqueador = true` impede transição para `Comunicado` até registrar comunicação
- Após comunicação registrada → status `Comunicado`

**Fluxo 6 — Retificação de laudo comunicado (cria v2):**

- Seed laudo `Comunicado` v1
- RT abre LaudoDetail → "Retificar" → motivo → submit
- Asserir: v1 marcada `Superado`, v2 criada com `motivoRetificacao`, supersededBy correto

**Fluxo 7 — Histórico de versões + bloqueio download v1:**

- Seed laudo com v1, v2, v3
- LaudoDetail mostra dropdown histórico
- Click "Baixar v1" → confirmação warning "Esta versão foi superada"
- Após confirmar → download permitido com banner

**Fluxo 8 — Busca + filtros:**

- Dashboard com 50 laudos seed (status mistos)
- Filter por status `Comunicado` → asserir só Comunicados visíveis
- Filter por crítico → asserir só com criticoFlag
- Filter por médico solicitante específico → asserir laudos correspondentes

**Acceptance:** 8/8 verde em CI; tempo total <4min.

### Task 6: Edge cases (1.5 dia)

- `e2e/edge-cases/retificacao.spec.ts`:
  - Retificar v1 já comunicado: v1 → Superado, v2 → Pendente (precisa nova liberação)
  - Tentar retificar laudo Pendente → erro (precisa estar liberado primeiro)
  - Retificar 5x → asserir v6 funciona; v21 rejeitado (limite)
- `e2e/edge-cases/comunicacao-failures.spec.ts`:
  - Email bounce → webhook → flag médico `email-problemático`
  - Email complaint (spam) → flag médico + alerta UI
  - Resend timeout → retry 3x → fallback "tentar manual"
- `e2e/edge-cases/concurrency.spec.ts`:
  - 2 RTs simultaneously call liberarLaudo → 1 sucesso, 1 erro `already-liberated`
  - 2 RTs simultaneously retificar → 1 sucesso, 1 erro `version-conflict`
  - PDF generation race: criar laudo + retificar imediatamente → ambas versões geram PDFs sem corromper

**Acceptance:** 8 edge cases verdes; sem flakiness em 5 runs.
