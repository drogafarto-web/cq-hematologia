## REQ-401: Advanced KPI Dashboarding (Department + Equipment Slice)

**Phase assignment:** v1.4 Phase 1  
**Priority:** Should-Have (Stretch)  
**Effort:** 16–20 story points  
**Risk if deferred:** Medium (limits root-cause analysis; auditor expects slice capability)

**Description:**  
Enhance KPI dashboard to filter by 3+ independent dimensions (equipment, operator, lot, analyte). Support real-time filtering with sub-500ms response, shareable deep links via URL state, and comparative sparklines (30d/90d trends). Export to PDF with all dimensions preserved.

**Acceptance criteria:**
- [ ] Dashboard UI accepts ≥3 independent filter dimensions (equipment, operator, lot, analyte)
- [ ] Filtering applies in real-time, <500ms response, <3KB network payload per filter change
- [ ] Filter state persists in URL query params; shareable deep links fully functional
- [ ] Results show turnaround%, retrabalho%, NC rate, trend sparklines (30d + 90d), NC origin pie chart
- [ ] PDF export preserves all dimensions + timestamp + filter summary on cover page
- [ ] WCAG AA accessibility: keyboard navigation, screen reader compatible labels, focus indicators
- [ ] Web Vitals: LCP <1.5s for dashboard load, CLS <0.05

**Dependencies:**
- REQ-412 (Risk Register) — if identifying root causes, must link NC back to risks
- v1.3 analytics module fully operational

**Related DICQ blocks:**
- 4.12.1 (Plano de Melhoria — improvement planning requires trend analysis)
- 4.14.7 (Indicadores de Qualidade — KPI tracking mandatory)

**Related RDC articles:**
- Art. 86 (Programa de Garantia da Qualidade — componente 1: governance via metrics)

## REQ-404: Coleta + Transporte (Pré-Analítico)

**Phase assignment:** v1.4 Phase 2  
**Priority:** Should-Have (core compliance)  
**Effort:** 18–22 story points  
**Risk if deferred:** Medium (pre-analytical gaps reduce DICQ 5.4 score; RDC 978 Art. 128–131 traceability required)

**Description:**  
Collection form capturing patient ID, collector, date/time, material type, volume, and specimen integrity criteria (clot, hemolysis, etc.). Integration with controle-temperatura to auto-log transport conditions (temp range, duration). Rejection criteria UI with automatic NC creation. Barcode/QR scanning. Mobile-first offline-capable design.

**Acceptance criteria:**
- [ ] Collection form: patient ID, collector name, date/time, material type, volume (mL), integrity code (hemolyzed, clotted, low volume, etc.)
- [ ] Integration with controle-temperatura: transport conditions (temp range, duration) auto-logged as environmental context
- [ ] Rejection criteria: user selects from standardized list (hemolyzed, clotted, insufficient volume, etc.)
- [ ] Auto-create NC with origin="pre-analítico" when rejection logged; NC links back to collection record
- [ ] Barcode/QR scan for sample tracking; sample ID auto-populated
- [ ] Mobile-first form: responsive <320px, offline-capable (sync on reconnect), minimal data entry
- [ ] Rules: collectors can only log own collections; lab manager + quality manager can review/override all

**Dependencies:**
- controle-temperatura module (v1.3, extend with transport logging)
- QR code library (zxing or similar)
- Mobile testing on iPad/tablet

**Related DICQ blocks:**
- 5.4.4 (Coleta — instruções, identificação)
- 5.4.5 (Transporte — prazo, temperatura, integridade)

**Related RDC articles:**

**Dependencies:**

## REQ-406: Gestão de Áreas de Risco + Biossegurança (ISO 14644)

**Phase assignment:** v1.4 Phase 2  
**Priority:** Should-Have (core compliance)  
**Effort:** 15–19 story points  
**Risk if deferred:** Medium (DICQ 4.2 (environment) + RDC 222 (biohazard) directly audited)

**Description:**  
Classify lab areas by biohazard level (NB1–NB4), assign environmental protection equipment (EPE), track annual ISO 14644 inspections with particle counting, and link to training requirements. Auto-create NC if inspection fails. PDF audit-ready report.

**Acceptance criteria:**
- [ ] Area registry: name, NB level (1–4), square meters, equipment list, EPE requirements, last inspection date
- [ ] ISO 14644 particle counting form (annual mandatory): equipment, standards used, result (particles/m³), pass/fail
- [ ] Inspection checklist: biosafety cabinet certifications (annual cert link), pressurization test result, warning signage, PPE stations, hand hygiene
- [ ] Non-conformance trigger: if inspection fails ISO 14644 or checklist, auto-create NC with inspection link
- [ ] Training linkage: new employees assigned to area must complete biossegurança training linked to that NB level
- [ ] PDF report: NB classification, inspection records (3-year history), training roster, emergency contacts (auditor-ready)
- [ ] Rules: only biossegurança manager + admin can modify area definitions; anyone can read

**Dependencies:**
- Biossegurança module data (v1.3)
- Trainings module (v1.3, extend with area-linked courses)
- External calibration certificates (for biosafety cabinet)

**Related DICQ blocks:**

## REQ-412: Plano de Gerenciamento de Riscos (Risk Register)

**Phase assignment:** v1.4 Phase 1  
**Priority:** Should-Have (governance)  
**Effort:** 14–16 story points  
**Risk if deferred:** Medium (DICQ 4.14.6 risk management mandatory; RDC 978 Art. 86 componente 2 requires demonstrated risk ownership)

**Description:**  
Living risk register mapping hazards to controls. Risk matrix (hazard → likelihood × severity → score). Risk categories (pre-analytic, analytic, post-analytic, equipment, personnel, environment). Control registry per risk. Annual revalidation + NC linkage. Heatmap visualization. Action plan for high-risk items. PDF audit-ready report.

**Acceptance criteria:**
- [ ] Risk matrix UI: hazard description → likelihood (1–5) × severity (1–5) → risk score (1–25, color-coded)
- [ ] Risk categories (DICQ-aligned): pre-analytic, analytic, post-analytic, equipment, personnel, environment (6 categories)
- [ ] Control registry per risk: list preventive + detective controls, effectiveness rating (high/medium/low), owner, test frequency
- [ ] Revalidation trigger: annually (calendar reminder) + when NC opens in that risk category (auto-flag)
- [ ] NC linkage: if NC created with origin="analitic", flag corresponding risk for review; vice versa
- [ ] Heatmap visualization: all risks plotted (likelihood × severity), color zones (red=high, yellow=medium, green=low)
- [ ] Action plan: for each risk score ≥15 (high), define owner + deadline + mitigation actions, track completion
- [ ] PDF report: all risks table + heatmap + action status + effectiveness evidence + approval signatures (auditor-ready)
- [ ] Rules: quality manager + director can edit; all can read heatmap; revalidation immutable once signed

**Dependencies:**
- NC module (v1.3, link integration)
- Heatmap visualization library (Chart.js or Plotly)

**Related DICQ blocks:**

## REQ-415: Reconciliação de Laudos + Arquivo (Laudo Lifecycle + Retention)

**Phase assignment:** v1.4 Phase 3  
**Priority:** Should-Have (compliance)  
**Effort:** 13–16 story points  
**Risk if deferred:** Medium (RDC 978 Art. 115 5-year retention mandatory; non-compliance = audit finding)

**Description:**  
Laudo lifecycle management (draft → RT-reviewed → released → final → archived). Retention rules (5-year minimum per RDC 978 Art. 115). Physical sample + digital laudo barcode link. Archival reminders. Disposal workflow. Digital cold storage backup (GCS Nearline). Compliance audit-ready report.

**Acceptance criteria:**
- [ ] Laudo state machine: draft → RT-reviewed → released → final → archived (workflow enforced, immutable state transitions)
- [ ] Retention rules: 5 years minimum (RDC 978 Art. 115), configurable per lab region (São Paulo vs. federal standard)
- [ ] Storage tracking: physical sample barcode + digital laudo linked, storage location (freezer A, shelf 3, etc.), retention expiry date
- [ ] Archival reminder: email + in-app notification at 30d, 7d before retention expires (idempotent, no duplicate alerts)
- [ ] Disposal workflow: mark for destruction, manager approval, contractor pickup confirmation, receipt signed (immutable once destroyed)
- [ ] Digital archival: PDF backup in GCS Nearline (cold storage, <$1/100 docs/month), immutable, accessible for audit
- [ ] Compliance report: audit-ready list of laudos by retention status (active/expiring-30d/expiring-7d/expired/destroyed), export to CSV/PDF
- [ ] Rules: RT can release, lab manager can archive, contractor can confirm pickup, auditor can read all

**Dependencies:**
- Laudos module (v1.3, extend with lifecycle)
- GCS bucket setup (standard Firebase project)
- Barcode/QR library (already used in REQ-404)

**Related DICQ blocks:**
- 4.13 (Controle de Registros — record retention & control)
- 5.7.4 (Armazenamento, Retenção, Descarte de Amostras)

**Related RDC articles:**
