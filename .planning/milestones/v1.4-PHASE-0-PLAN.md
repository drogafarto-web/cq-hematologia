## ⚠️ Decisions Locked (2026-05-07) — supersede original spec

These 3 decisions resolve ambiguities surfaced during pre-execution review by CTO. The Tasks below remain in original form for traceability but **executors MUST follow these overrides**:

### DL-1: Callable from day 1 (not client-direct)

**Override:** Tasks 1, 3, 4 — all `create*` operations on `turnos`, `lab-apoio`, `risks` ship as **Cloud Function callables** in Phase 0 (not "Phase 1 migrates to callable" as original Plan stated).
**Rationale:** CLAUDE.md governance rule "Escrita regulatória via Cloud Function callable (Fase 0b+)" applies to Phase 0 (= Fase 0b+ semantically). RDC 978 audit trail must be tamper-evident from launch — accepting client-direct write as "transitional debt" exposes the project to auditor finding (RT supervision, lab apoio contracts, risk register are the highest-criticality writes).
**Impact:** +0.5d per module (~+1.5d total) for callable scaffold; Phase 0 may stretch from 7d to 8.5d. Still inside the 9d hard stop documented in P0-R4.
**Reference:** ADR-0012 (universal audit trail), CLAUDE.md "Convenções invioláveis", `firestore-security.md` rule.

### DL-2: SGQ (not SGD) for POL-LGPD-001

**Override:** Task 2 deliverable #1 reads "SGD document" → it must be **SGQ document** (`/labs/{labId}/sgq-documentos/`, `tipo: 'POL'`).
**Rationale:** Code reality — the `sgd` module (`src/features/sgd/`) is a Drive importer for external documents (`sgd-externos`); its schema does not have `tipo`, `versao`, or `status` (em_revisao/vigente/obsoleto). The `sgq` module (`src/features/sgq/`, in production since 2026-04-26) is the document hierarchy module with `TipoDocumento = MQ | PQ | IT | FR | POL | DC | LM | EXT` and the full em_revisao→vigente→obsoleto state machine, audit chain (`sgq-documentos-audit`), and RN-SGQ-01..07 invariants.
**Impact:** Zero — SGQ already supports POL natively. Task 2 effort unchanged (4h). DPIA template ships as `tipo: 'IT'` (Instrução de Trabalho) per `IT-LGPD-DPIA-001` codigo.
**Acceptance gate adjustment:** "Lista mestra obrigatória" badge integrates with `SGQView` (not `/sgd` route).
**Reference:** `src/features/sgq/types/Documento.ts` lines 40-48 (TipoDocumento), `src/features/sgq/CLAUDE.md` (RN-SGQ-01..07).

### DL-3: No patient consent flow in Phase 0 (deferred to Phase 5)

**Override:** Task 2 deliverable #3 (Patient registration UI consent checkbox) — **REMOVED from Phase 0 scope.**
**Rationale:** In HC Quality, patients are NOT entities of the application. The system is a CIQ laboratory quality control tool — `auth` users are operators (RT, supervisors, admins), not patients. The original Plan inferred a patient-facing onboarding that does not exist (and `src/app/(auth)/registro/page.tsx` is not the project's path convention — there is no Next.js App Router here; project is React 19 + Vite SPA). The LGPD compliance for Phase 0 is satisfied by:

1. POL-LGPD-001 published in SGQ as `vigente` with `autoridadeEmitente` = RT signature (organizational acceptance representing the lab as data controller).
2. DPIA template (`IT-LGPD-DPIA-001`) published in SGQ as `vigente`.
3. Audit-trail evidence via `sgq-documentos-audit` (created/status-changed events).
4. UI badge "Documentos Obrigatórios" in `SGQView` showing POL-LGPD-001 status.
   **Patient consent is out of scope for Phase 0** and will be addressed in **Phase 5 (Patient Portal)** when patients become entities (email-link auth per ADR-0015). Until then, patient-side LGPD is governed by the lab's offline contract with the patient (paper consent at sample collection — pre-existing Riopomba practice).
   **Impact:** Task 2 effort drops from 0.5d to 0.25d (just the 2 SGQ docs + badge). Acceptance criterion "Patient registration shows policy link + consent checkbox" → **DROPPED** (not in v1.4 Phase 0 scope; deferred to v1.4 Phase 5).
   **Reference:** ADR-0015 (patient portal email-link auth, Phase 5), CLAUDE.md "Reads abertos a qualquer user autenticado em 2026-05" (no patient as user).

### Task 1: Supervisor Shift Registry (RDC 978 Art. 122 + RDC 786)

**Goal**: Track which credentialed professional is supervising each shift (manhã/tarde/noite/plantão). Auditor must be able to query "quem supervisionou em [data] [turno]" with one click.

**Effort**: 1 day (small) — single-collection CRUD module

**Compliance citation**:

- **RDC 978/2025 Art. 122**: "Todo laboratório deve ter responsável técnico legalmente habilitado, com supervisão presencial documentada por turno."
- **DICQ 4.1.2.7**: "Designação formal do responsável pela qualidade."
- **RDC 786/2023**: Habilitação profissional (RT) vs capacitação (operador).

**Deliverables**:

1. **Firestore collection**: `/labs/{labId}/turnos/{turnoId}`
   - Fields:
     ```ts
     {
       labId: string;              // redundant per multi-tenant convention
       data: Timestamp;            // shift date
       periodo: 'manha' | 'tarde' | 'noite' | 'plantao';
       supervisorId: string;       // reference to /labs/{labId}/colaboradores/{id}
       supervisorName: string;     // denormalized for fast list rendering
       supervisorCRBM?: string;    // habilitação (RT) ou cargo
       certificatesActive: string[]; // active habilitações IDs at shift time (snapshot)
       observacoes?: string;       // free text (max 500 chars)
       criadoEm: Timestamp;
       criadoPor: string;          // operatorId
       deletadoEm?: Timestamp;     // RN-06 soft delete only
       logicalSignature: { hash: string; operatorId: string; ts: Timestamp };
     }
     ```

2. **Service**: `src/features/turnos/services/turnosService.ts`
   - `listTurnos(labId, { from, to })` — onSnapshot subscription
   - `createTurno(labId, dto: CreateTurnoDTO)` — DTO via `Omit<Turno, 'id'|'labId'|'criadoEm'|'deletadoEm'|'logicalSignature'>`
   - `softDeleteTurno(labId, turnoId, operatorId)` — RN-06 only
   - **Note**: Phase 0 ships client-direct create; Phase 1 migrates to callable per `firestore-security.md` rule.

3. **Hook**: `src/features/turnos/hooks/useTurnos.ts`
   - Pattern: mirror `useColaboradores.ts` (canonical reference).
   - Validates `supervisorId` exists in active `colaboradores`.
   - Validates `certificatesActive` snapshot has at least one habilitação ativa.
   - Generates `logicalSignature` (SHA-256 canonical via `hcq-ciq-audit-trail` skill).

4. **UI**:
   - `src/features/turnos/components/TurnoForm.tsx` — create/edit form with date picker + period selector + supervisor combobox.
   - `src/features/turnos/components/TurnosList.tsx` — table view with filters (data range, periodo, supervisor).
   - `src/features/turnos/components/CoberturaReport.tsx` — auditor view: "shift coverage last 90 days", missing-shift heatmap.
   - Route: `/turnos` (lazy-loaded; manualChunks entry).
   - Design: dark-first per `DESIGN_SYSTEM.md`, `tabular-nums` on data column.

5. **Firestore rules** (use `hcq-firestore-rules-generator` skill):
   - `match /labs/{labId}/turnos/{turnoId}` — read: `isActiveMemberOfLab(labId)`; create/update: `isAdminOrOwner(labId)` + `isValidTurno(request.resource.data)`; delete: blocked (soft delete via update of `deletadoEm`).
   - `isValidTurno`: validates required fields, `data is timestamp`, `periodo in ['manha','tarde','noite','plantao']`, `logicalSignature.hash.size() == 64`.

6. **Audit trail integration**:
   - Subcoleção `/labs/{labId}/turnos/{turnoId}/events/{eventId}` — append-only, chainHash trigger via Cloud Function.
   - Pattern: mirror `hcq-ciq-audit-trail` skill scaffold.

7. **Backfill** (production data):
   - Cloud Function callable `backfillTurnos90Days(labId)` (admin-only).
   - Uses existing `colaboradores` + `equipamentos` lab settings to infer "default supervisor" and creates `inferred: true` turnos for last 90 days.
   - Manager must confirm/correct each inferred row in week 1 via dedicated UI banner.

**Acceptance criteria**:

- [ ] Manager can register supervisor per shift in <30s (single-screen form).
- [ ] Auditor can query shift coverage history (last 90d, custom range, by supervisor).
- [ ] All shifts in last 90 days have a registered supervisor (post-backfill, with `inferred: true` flag for manager review).
- [ ] Logical signature validates (`verifyChain` script passes).
- [ ] Firestore rules emulator test passes (read/write/delete cases).
- [ ] List view loads <200ms with 90 days of data (≈270 rows).

**Integration points**:

- Reads from `/labs/{labId}/colaboradores/{id}` (existing `educacao-continuada` module).
- Read by future Phase 9 (Documentation Hardening) personnel dossier.
- Read by future Phase 13 (Final DICQ Audit) coverage report.

### Task 2: LGPD Formal Policy (RDC 978 Art. 77)

**Goal**: Written privacy policy stored as versioned SGD document, linked from patient registration and onboarding flows.

**Effort**: 0.5 day (small) — mostly documentation, leverages existing SGD module

**Compliance citation**:

- **RDC 978/2025 Art. 77**: "Política de privacidade documentada e disponível ao titular dos dados."
- **LGPD (Lei 13.709/2018) Art. 8**: Consentimento livre, informado e inequívoco.
- **DICQ 4.3**: Documentos da qualidade (POL = Política).

**Deliverables**:

1. **SGD document**: "Política de Privacidade e Proteção de Dados (LGPD)"
   - Tipo: `POL` (política)
   - Código: `POL-LGPD-001`
   - Versão: `1.0`
   - Created via existing `src/features/sgd/hooks/useSGDDocumentos.ts` + form UI.
   - Stored at `/labs/{labId}/sgd-documentos/{docId}` (existing collection).

2. **Boilerplate content** (template, ~3 pages, pt-BR):
   - **1. Coleta de dados**: que dados coletamos (cadastrais, clínicos, biométricos quando aplicável)
   - **2. Finalidade**: realização de exames, emissão de laudos, faturamento, obrigações legais (RDC 978, RDC 222)
   - **3. Base legal**: consentimento (Art. 7º LGPD) + obrigação legal (Art. 7º, II) + tutela da saúde (Art. 11º, II, "f")
   - **4. Retenção**: laudos 5 anos (RDC 978 Art. 115); dados cadastrais conforme política de retenção interna
   - **5. Compartilhamento**: auditores credenciados (DICQ), órgãos regulatórios (ANVISA via NOTIVISA), laboratórios de apoio mediante contrato (Arts. 36–39)
   - **6. Direitos do titular**: acesso, correção, eliminação, portabilidade, revogação de consentimento (Art. 18 LGPD)
   - **7. DPO/Encarregado**: nome, email, canal de contato
   - **8. Medidas de segurança**: criptografia em trânsito + repouso (Firebase), MFA, audit trail tamper-evident, soft delete
   - **9. Revisão**: revisão obrigatória anual (review cycle automation in Task 4 risk module)
   - **10. Contato regulador**: ANPD (autoridadenacional)

3. **Patient consent flow integration**:
   - Update `src/app/(auth)/registro/page.tsx` (or current onboarding flow) to display LGPD policy link via SGD document URL.
   - Add checkbox: "Li e concordo com a Política de Privacidade (POL-LGPD-001 v1.0)" — captured in patient record (`consentLGPD: { version: '1.0', timestamp, ip }`).
   - **Note**: Patient registration UI already exists in v1.3; this is a copy + checkbox addition only.

4. **DPIA template** (Data Protection Impact Assessment):
   - New SGD document: `IT-LGPD-DPIA-001` (Instrução de Trabalho).
   - Template covering: descrição do tratamento, finalidade, riscos identificados, medidas de mitigação, owner, revisão.
   - Used for any future processing change (e.g., new IA feature, new export channel).

5. **SGD UI badge**:
   - Update `/sgd` route to highlight "Documentos Obrigatórios" section listing POL-LGPD-001 + future obligatórios (POP-RT, POL-Biossegurança, etc.).
   - Visual: emerald-500 check badge if v1.0+ published; red-400 if missing.

**Acceptance criteria**:

- [ ] Policy published in SGD with version 1.0, status `aprovado`, RT signature captured.
- [ ] Linked from `/sgd` UI under "Documentos Obrigatórios" with status badge.
- [ ] DPIA template (IT-LGPD-DPIA-001) exists and approved.
- [ ] Patient registration shows policy link + consent checkbox; consent persisted with version + timestamp.
- [ ] Annual review reminder scheduled (Cloud Scheduler, fires at v1.0 + 365d).
- [ ] Auditor can demonstrate: published policy + patient consent records + DPIA template in <5 min.

**Integration points**:

- Existing `src/features/sgd/services/sgdService.ts` (no schema changes).
- Existing patient registration flow (copy + 1 checkbox).
- Annual review reminder consumed by Task 4 risk register `reviewDate` cadence.

### Task 4: Risk Management Skeleton (DICQ 4.14.6)

**Goal**: FMEA-lite framework for risk identification + treatment + review. Foundation for Phase 4 (CAPA) and Phase 13 (Final Audit).

**Effort**: 3 days (medium-large) — new module with matrix visualization + automated review cycle

**Compliance citation**:

- **DICQ 4.14.6**: "Gestão de risco — identificação, análise, avaliação, tratamento, monitoramento."
- **ISO 15189:2022 § 8.5**: Actions to address risks and opportunities.
- **RDC 978/2025 Art. 86**: PGQ (Programa de Gestão da Qualidade) — risco é componente obrigatório.

**Deliverables**:

1. **Firestore collection**: `/labs/{labId}/risks/{riskId}`
   - Fields:
     ```ts
     {
       labId: string;
       codigo: string;                 // R-001, R-002, ... auto-incrementing per lab
       descricao: string;              // descrição do risco
       processo: string;               // pré-analítico | analítico | pós-analítico | gestão | infraestrutura | pessoal
       categoria: string;              // segurança | qualidade | regulatório | operacional | financeiro
       probabilidade: 1 | 2 | 3 | 4 | 5;   // 1=raro, 5=quase certo
       severidade: 1 | 2 | 3 | 4 | 5;       // 1=insignificante, 5=catastrófico
       deteccao: 1 | 2 | 3 | 4 | 5;         // 1=facilmente detectável, 5=indetectável
       npr: number;                    // auto-calc: probabilidade * severidade * deteccao (1–125)
       nivel: 'baixo' | 'medio' | 'alto' | 'critico'; // derived from NPR thresholds
       tratamento: {
         estrategia: 'evitar' | 'mitigar' | 'transferir' | 'aceitar';
         acoes: Array<{ descricao, prazo: Timestamp, owner: string, status: 'pendente' | 'em_andamento' | 'concluido' }>;
       };
       owner: string;                  // colaborador responsável
       status: 'aberto' | 'mitigando' | 'monitorado' | 'fechado';
       reviewDate: Timestamp;          // próxima revisão (default: criadoEm + 12 meses)
       reviewHistory: Array<{          // histórico de revisões
         data: Timestamp;
         revisor: string;
         resultado: 'mantido' | 'reduzido' | 'reclassificado' | 'fechado';
         observacoes: string;
         nprPrevio?: number;
         nprNovo?: number;
       }>;
       linkedCAPAs?: string[];         // future Phase 4 CAPA IDs
       linkedNCs?: string[];           // não conformidades relacionadas
       criadoEm: Timestamp;
       criadoPor: string;
       deletadoEm?: Timestamp;
       logicalSignature: { hash, operatorId, ts };
     }
     ```

2. **NPR thresholds** (configurable per lab in `labSettings`):
   - `baixo`: NPR ≤ 24
   - `medio`: 25–60
   - `alto`: 61–99
   - `critico`: NPR ≥ 100

3. **Service + hook + UI**:
   - `src/features/risks/services/risksService.ts` — CRUD + softDelete + NPR auto-calc.
   - `src/features/risks/hooks/useRisks.ts` — list/create/update/review + signature.
   - `src/features/risks/components/RiskRegister.tsx` — sortable/filterable table (by NPR, processo, status).
   - `src/features/risks/components/RiskForm.tsx` — multi-step (descrição → análise FMEA → tratamento → owner/review).
   - `src/features/risks/components/RiskMatrix.tsx` — 5×5 heatmap (probabilidade × severidade), color-coded by NPR (emerald/amber/orange/red).
   - `src/features/risks/components/RiskReviewModal.tsx` — periodic review form.
   - `src/features/risks/components/Top5RisksWidget.tsx` — dashboard widget showing top 5 risks by NPR.
   - Route: `/risks` (lazy-loaded; manualChunks entry).

4. **Firestore rules** (`hcq-firestore-rules-generator`):
   - read: lab member; create/update: admin/owner + `isValidRisk`; delete: blocked.
   - `isValidRisk`: validates `probabilidade in 1..5`, `severidade in 1..5`, `deteccao in 1..5`, `npr == probabilidade * severidade * deteccao`.

5. **Audit trail**: subcoleção events with chainHash (mirror prior tasks).

6. **Annual review automation** (Cloud Function):
   - `functions/src/modules/risks/scheduledReview.ts` — daily scheduled at 07:00 BRT.
   - Queries risks where `reviewDate <= today` and `status != 'fechado'`.
   - Notifies `owner` + lab admins via email + in-app.
   - Top 5 critical risks (NPR ≥ 100) flagged for monthly review even if `reviewDate` not reached.

7. **Initial risk seed** (CTO-defined, post-Phase 0):
   - Optional CSV import for initial 10–15 known risks (data leak, CIQ failure, equipment downtime, RT absence, contract expiry, etc.) to give the lab a starting register.

**Acceptance criteria**:

- [ ] Manager can register risk with FMEA scoring (probabilidade × severidade × detecção).
- [ ] NPR auto-calculated; nivel derived; visual badge (color-coded).
- [ ] Heat map (5×5 matrix) shows risk distribution; click cell to filter.
- [ ] Top 5 critical risks (NPR ≥ 100) flagged for monthly review.
- [ ] Treatment status tracked (ações + prazos + owner + status).
- [ ] Periodic review form captures resultado + NPR transition (previous/new).
- [ ] Annual review reminder fires automatically.
- [ ] Auditor can demonstrate active risk register + treatment evidence + review history.
- [ ] Logical signature + audit trail validate.

**Integration points**:

- Future Phase 4 (CAPA): `linkedCAPAs` field cross-links risks to CAPA records.
- Future Phase 12 (Audit Dashboard): top risks widget on dashboard.
- Future Phase 13 (Final DICQ Audit): risk register evidence for DICQ 4.14.6 closure.
