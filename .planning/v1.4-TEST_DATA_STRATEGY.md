# v1.4 Test Data Strategy — Phases 4–9 Execution

**Scope:** Complete test data ecosystem for phases 4–9 (Portal, NOTIVISA, Critical Values, CAPA, Bioquímica, OCR, Patient Feedback).

**Success criteria:**
- All test data scripts executable and idempotent
- Test labs isolated from production via `isTestLab: true`
- Data fixtures cover happy path + error scenarios
- Ready for QA team to use in E2E testing across all phases
- Reusable template for Phases 10–15

---

## Part 1: Test Environment Setup (~250 words)

### 1.1 Staging Firestore Instance

**Production project:** `hmatologia2`  
**Staging project:** `hmatologia2-staging`  
**Local emulator:** `localhost:8080` (for dev/unit test loops)

**Setup steps:**
1. Create Cloud Firestore instance in `hmatologia2-staging` project (region: `southamerica-east1`)
2. Deploy production security rules to staging (step one before any data writes)
3. Enable Firebase Authentication + custom claims support
4. Create service account key for admin SDK (stored in `.env.staging`, gitignored)

**Firestore emulator for local dev:**
```bash
# Start emulator (persists data to .emulator/firestore_export/ by default)
firebase emulators:start --only firestore

# Reset emulator between test runs (clean slate)
rm -rf .emulator/firestore_export
```

### 1.2 Cloud Functions in Staging Region

Deploy a staging-only function bundle to `southamerica-east1` (same region as production):
- All 50+ callables must be deployable to staging
- Use environment variable `FIREBASE_PROJECT_ID` to route between staging/prod
- `sendgrid`, `twilio`, `gemini` mocked in staging (no real SMS/emails)
- Test metrics collection (error rate, latency) enabled

```bash
firebase deploy --only functions --project hmatologia2-staging
```

### 1.3 Test User Accounts per Role

**Roles required (RBAC via custom claims + member doc):**

| Role | labIds | Module Access | Notes |
|------|--------|---|---|
| **CTO** | all | all modules | Full admin, deployment gate reviewer |
| **Operador** | assigned | analyzer, runs, chart, reports | Data entry, CIQ workflows |
| **Supervisor Clínico** | assigned | analyzer, bioquimica, reports, ceq, kpis | Approvals, critical values, trend analysis |
| **Patient** | — | portal (own laudos) | External portal, via phone/email auth |
| **Auditor** | all (read-only) | auditoria, sgq, pops, risks, capa-tracking | Compliance review, no write access |
| **Admin Lab** | assigned | all except admin | Lab configuration, user management |

## Part 7: Data Isolation Strategy

### 7.1 Test Lab Tagging

**Firestore rule (in `firestore.rules`):**
```
// Prevent production from reading test data
match /labs/{labId} {
  allow read: if resource.data.isTestLab != true || request.auth.customClaims.role == 'test-runner';
  allow write: if resource.data.isTestLab == true;
}
```

**Schema addition to lab doc:**
```typescript
interface Lab {
  // ... existing fields
  isTestLab: boolean;  // true only in staging
  testLabMarker?: string;  // "v1.4-phase-4-9" for tracking
  testLabCreatedAt?: Timestamp;
  testLabMaintainedBy?: string;  // "automation" or operator email
}
```

### 7.2 Test User Tagging

```typescript
interface Usuario {
  // ... existing fields
  isTestUser: boolean;  // true only for seed-created users

  testUserPassword?: string;  // encrypted, for test automation only
}
```

**CronCreate trigger (in `.claude/` memory or Cloud Scheduler):**
```bash
# Every Sunday at 02:00 UTC (staging time)
```

### 8.1 Emulator Setup for Local E2E Tests

**File:** `scripts/firestore-emulator-setup.sh` (already exists)

```bash
#!/bin/bash
set -e

# 1. Check prerequisites
./scripts/check-emulator-prerequisites.sh

# 2. Start emulator in background
firebase emulators:start --only firestore &
EMULATOR_PID=$!

# 3. Wait for readiness
until curl -s http://localhost:8080 > /dev/null; do
  sleep 1
done

# 4. Seed test data to emulator
export FIREBASE_EMULATOR_HOST=localhost:8080
export FIREBASE_PROJECT_ID=hmatologia2-staging
npx ts-node scripts/seed-test-labs.ts
npx ts-node scripts/seed-test-users.ts
npx ts-node scripts/seed-test-laudos.ts

# 5. Run E2E tests
npm run test:e2e

# 6. Cleanup
kill $EMULATOR_PID
```
