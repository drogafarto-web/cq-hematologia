## 1. Severity Classification Matrix

Define incident severity by **impact scope** (user %), **SLA impact**, and **operational constraints**. Severity drives **paging rules**, **stakeholder notification**, and **response timelines**.

### 🟢 GREEN — Service Degraded

**Definition:** Feature partial/slow, <1% users affected, SLA impact <15 min, no data loss, audit trail intact.

**Examples:**
- Single module slow (analytics 5s instead of 2s)
- Specific equipment type unavailable (1 of 5 analyzers down)
- Single lab locked out (auth token rotation issue)
- Non-critical job delayed (nightly export pending)

**Response SLA:** 1 hour to acknowledge; initial investigation started  
**Paging:** Primary on-call only  
**Escalation:** Auto-escalate to Yellow if issue not triaged in 30 min  
**Notification:** Slack #production-alerts (summary line); no customer proactive call  
**Authority:** Primary on-call decides: monitor vs. hotfix vs. scheduled rollback

**Success Criteria:**
- User can work around the issue (alternate path available)
- <15 min SLA impact = no RDC 978 violation
- No data loss or signature corruption

---

### 🟡 YELLOW — Service Down (Partial)

**Definition:** Feature completely unavailable OR cascading degradation, 1–10% users affected, SLA impact 15 min–1 hour, no data loss, audit trail intact.

**Examples:**
- Single module down (all users cannot access CQ runs)
- Regional outage (South America region slow/timeout)
- Batch job failure (daily QC report not generated)
- Auth degradation (2-factor SMS delays >5 min)
- Firestore index missing (complex queries timeout)
- Specific tenant unavailable (multi-lab scenario)

**Response SLA:** 15 min to acknowledge; mitigation started  
**Paging:** Primary + Secondary on-call (parallel); CTO notified (Slack @cto)  
**Escalation:** Auto-escalate to Red if not mitigated in 45 min  
**Notification:** Slack #production-alerts (severity + ETA); CTO decision on customer proactive notification  
**Authority:** Incident Commander (Primary or CTO) decides: hotfix vs. rollback vs. scaling

**Success Criteria:**
- Mitigation started within 15 min (deploy, scale, or rollback initiated)
- Data integrity verified (no partial writes, no audit trail gaps)
- ETA provided to stakeholders every 5 min if incident extends beyond 15 min

---

### 🔴 RED — Service Down (Critical)

**Definition:** Core service unavailable, >10% users affected OR audit trail affected/suspicious, SLA impact >1 hour, potential data loss or compliance violation.

**Examples:**
- Entire platform down (Hosting 5xx, Firestore quota exceeded, Auth service unavailable)
- Audit trail not recording (Cloud Functions not triggering event capture)
- Batch signature corruption (multiple chain-hash mismatches detected)
- Data loss detected (documents deleted, critical field null unexpectedly)
- Regulatory deadline missed due to system outage (e.g., NOTIVISA report window)
- Security event (unauthorized access, token leakage, rules bypass)

**Response SLA:** 5 min to acknowledge; incident commander assigned immediately  
**Paging:** All hands — Primary + Secondary + CTO + incident commander (via Slack @channel #production-emergency)  
**Escalation:** CEO notified if incident >30 min without mitigation plan  
**Notification:** Slack #production-emergency + #production-alerts (Slack post every 5 min); Customer proactive notification (email + phone); Auditor notified if compliance-related  
**Authority:** Incident Commander (CTO if primary unavailable) — sole decision authority; no parallel approval required; deploy or rollback at will

**Success Criteria:**
- Incident declared in Slack within 5 min (timestamp + severity + IC assigned)
- Severity confirmed (is it really 10%+? audit trail really down?)
- Mitigation plan articulated (rollback commit, hotfix branch, or scaling strategy)
- Status updates every 5 min (no longer than 30 min between updates)
- Data integrity verified post-resolution (no audit trail gaps, no signature corruption)
- Customer communication sent at resolution + ETA for post-mortem (24h)

---

### ⚫ BLACK — Critical Infrastructure/Compliance Failure

**Definition:** Data loss confirmed, security breach, audit trail corruption, legal/regulatory hold triggered, or infrastructure catastrophe (backup also down).

**Examples:**
- Firestore backup corruption (restore point also compromised)
- Unauthorized access to audit trail (attacker modified event logs)
- Encryption key compromised (all encrypted fields at risk)
- ANVISA/auditor discovery of compliance violation during live support
- Patient data breach (LGPD trigger: unencrypted download, unauthorized access)
- Ransomware or DDoS attack
- Critical mass of documents deleted or corrupted (>1% of data)

**Response SLA:** 2 min acknowledgment; all leadership mobilized  
**Paging:** All hands emergency — CTO + CEO + incident commander + auditor (legal review may be required)  
**Escalation:** Legal + Auditor + ANVISA contacts (if data/compliance)  
**Notification:** Slack #incident-command (not #production-alerts); CEO email; Legal briefing; external escalation (auditor/regulator) within 1h  
**Authority:** Incident Commander (CTO/CEO) with legal advisory; deploy/rollback only post-legal clearance (exception: if rollback itself is the remediation)

**Success Criteria:**
- Incident declared in Slack within 2 min with Black severity label
- Legal informed within 1 min of declaration (phone call, not email)
- Auditor notified within 1 hour
- Severity verified (is data actually lost? is it customer-facing?)
- Forensic preservation: freeze logs, snapshot backup, preserve evidence
- RCA held within 48h (legal may gate publication)
- Customer/regulator notification per legal guidance

## 2. On-Call Rotation Template

### 2.1 Schedule Overview

**Model:** 4-week rolling cycle, 2 people overlapping (Primary + Secondary), 24/7 coverage, weekly handoff.

**Coverage Hours:**
- **Week 1 (Mon 00:00 – Sun 23:59 BRT):** Primary A + Secondary A
- **Week 2 (Mon 00:00 – Sun 23:59 BRT):** Primary B + Secondary B
- **Week 3 (Mon 00:00 – Sun 23:59 BRT):** Primary C + Secondary C
- **Week 4 (Mon 00:00 – Sun 23:59 BRT):** Primary D + Secondary D
- **Repeat (Week 5 = Week 1 rotation)**

**Handoff Day:** Every Friday 18:00 BRT (60 min)

**Weekend Coverage:** Mandatory (same people as weekday).

---

### 2.2 SLA & Escalation Rules

| Stage | SLA | Action | Trigger |
|-------|-----|--------|---------|
| **Page Primary** | 15 min response | Primary acknowledges + begins investigation | Any incident detected (Green+) |
| **Escalate to Secondary** | +10 min (25 min total) | Page Secondary; escalation decision made | Primary not ACK'd in 10 min OR severity assessed as Yellow+ |
| **Escalate to CTO** | +5 min (30 min total) | Page CTO + assign Incident Commander | Primary + Secondary both unavailable OR severity = Red/Black |
| **Escalate to CEO** | +30 min (60 min total) | CEO briefed | Red severity + no mitigation plan articulated OR Black severity |

### 2.5 Handoff Protocol (Friday 18:00 BRT)

**Duration:** 60 min (18:00–19:00 BRT)  
**Attendees:** Outgoing Primary + Secondary, Incoming Primary + Secondary, CTO (optional, attends Q1/Q3 handoffs)  
**Location:** Zoom [LINK TBD]

**Agenda:**

1. **Open Incidents (10 min)**
   - List all open incidents (if any) with current state, ETA, and action owner
   - Incoming on-call reviews mitigation plan, agrees on handoff strategy

2. **Critical Alerts & Threshold Changes (5 min)**
   - Firestore quota utilization (if >60%, incoming must monitor)
   - Function error rate trend (increasing/stable/decreasing)
   - Any deploy or config change that affects alerting

3. **Baseline Metrics & Last 7 Days (5 min)**
   - Uptime % (target: >99.5%)
   - Incident count (Green/Yellow/Red/Black split)
   - MTTR (Mean Time To Resolution) for Yellow+ incidents
   - Any patterns or concerns

4. **Contact Tree Validation (5 min)**
   - Verify all phone numbers and emails are current
   - CTO availability for incoming week (confirm vacation/blocked time)
   - Any escalation contact changes

5. **Runbook Review & Questions (10 min)**
   - Incoming on-call asks questions about any runbook
   - Outgoing on-call reviews recent incidents: "Here's what we learned about the auth issue last week…"

6. **Slack Channel Setup (5 min)**
   - Verify `#on-call-paging` is in incoming person's Slack workspace
   - Test paging via mock page: "@incoming-primary, testing 1-2-3" → incoming ACK's with ✅

7. **Close & Document (5 min)**
   - Outgoing on-call posts in Slack #on-call-paging: "Handoff to [Name] complete. Open incidents: [list]. CTO on-call this week: [Name]. —[Outgoing Primary]"
   - Incoming on-call posts acknowledgment: "Incoming. Briefed on [count] incidents. Ready. —[Incoming Primary]"

### 2.6 Response Expectations & Unavailability Protocol

**Primary On-Call Responsibility:**
- Sleep with phone nearby (ringer ON, not vibrate)
- Respond to Slack page within 15 min any time of day/night
- If unavailable (shower, driving, out of range), post in #on-call-paging ASAP: "Running late, ETA [time], paging [Secondary] now"
- Do not ignore pages; if you know you cannot respond, tell Secondary immediately

**Secondary On-Call Responsibility:**
- Monitor `#on-call-paging` (may work, but must be available)
- If Primary not ACK'd in 10 min, reply in thread: "@[Primary], escalating to you as backup" + acknowledge the incident
- Cover for Primary if Primary flagged as unavailable
- Do not sleep if expecting Yellow+ (check Slack before bed)

**Escalation Protocol If Primary Unavailable:**
1. Page Secondary in thread (mention @secondary-on-call)
2. Wait 5 min for Secondary ACK
3. If no Secondary ACK, page CTO: "@cto — both primary + secondary unavailable, incident [severity] ongoing"

## 3. Incident Commander Role

### 3.1 Weekly Assignment & Rotation

**Assigned:** Same person as Primary on-call (exception: CTO always serves as Incident Commander for Red/Black).

**Duration:** 1 week (Mon 00:00 – Sun 23:59 BRT)

**Handoff:** Friday 19:00 BRT, documented in Slack #on-call-paging.

---

### 3.2 Responsibilities (Decision Authority)

**Authority Scope:**
- **Green:** Incident Commander may handle alone (monitor, investigate, post findings)
- **Yellow:** Incident Commander may hotfix/rollback alone; primary engineer must be looped for 2-person code review (async OK if urgent)
- **Red:** Incident Commander may hotfix/rollback alone (2-person rule suspended in emergency); must brief CTO within 10 min of decision
- **Black:** Incident Commander must consult CTO/CEO before any destructive op; legal clearance may be required

**Weekly Checklist (completed by Friday EOD):**
- [ ] 1. Incident log reviewed (all incidents logged in `#production-alerts`)
- [ ] 2. Incident severity calibrated (is any Green mislabeled as Yellow?)
- [ ] 3. RCA scheduled for all Yellow+ incidents (within 48h of resolution)
- [ ] 4. Post-mortem action items tracked in `v1.4-INCIDENT_ACTIONS.md` (updated weekly)
- [ ] 5. Runbooks updated if any incident revealed gaps
- [ ] 6. Alerting rules reviewed (did we catch this earlier?)
- [ ] 7. No open critical bugs from incidents (escalated or assigned)
- [ ] 8. Contact tree validated (phone numbers current?)
- [ ] 9. Team sentiment checked (any burnout from on-call? rotate out next week if needed)
- [ ] 10. Handoff document posted (open incidents, upcoming risks, CTO notes)

---

### 3.3 Decision Matrix

**Scenario: Code deployment decision**

| Severity | Decision | Authority | 2-Person Rule | Deadline |
|----------|----------|-----------|---------------|----------|
| **Green** | Deploy hotfix, monitor, auto-rollback on fail | IC alone | No | 2 hours |
| **Yellow** | Deploy hotfix after 1 code review (async OK) | IC | Yes (async code review by 1 engineer) | 15 min |
| **Red** | Deploy/rollback — IC sole authority; brief CTO post-decision | IC | No (emergency exception) | 5 min |
| **Black** | Consult CTO/legal before deploy; preserve forensics first | IC + CTO | Yes | 2 min → decision |

**Scenario: Rollback decision**

| Scenario | Decision | Authority | Can Roll Back Alone? | 2-Person Witness |
|----------|----------|-----------|----------------------|------------------|
| **Hotfix failed (code defect)** | Rollback to last stable | IC | Yes | IC must screenshot decision + reason in Slack |
| **Firestore rule broke (config)** | Revert rules.json, redeploy | IC | Yes | IC must tag @cto (async review) |
| **Rogue data deletion detected** | Restore from backup | CTO + IC (joint) | **No** — requires 2 people | Both must sign off before restore |
| **Security breach (auth bypass)** | Isolate + preserve forensics, escalate to legal | CTO + IC | **No** — legal gates restore | Legal review first |

### 4.2 Runbook Template (Fill in details, one per incident type)

```markdown
# Runbook: [Incident Type]

## Decision Tree

START → Is [condition]? 
  YES → [Severity Level] → [Primary Action]
  NO → [Alternative Path]

## Severity Assessment (30 seconds)

- [ ] Check Cloud Logs (error count, affected users, SLA impact)
- [ ] Check Firestore quota + index status (if relevant)
- [ ] Check function cold-start latency
- [ ] Declare severity (🟢/🟡/🔴/⚫)

## Action Steps

1. **Confirm Scope** — Is it 1% or 50% of users? (check by tenant/region/module)
2. **Identify Root** — Use Cloud Trace, Cloud Logs, or error message + stack trace
3. **Communicate** — Post status to #production-alerts with ETA (5 min max from incident start)
4. **Mitigate** — Hotfix, scale, or rollback (decide within 10 min of incident start)
5. **Verify** — Confirm metric return to baseline (error rate <0.1%, latency <2s, etc.)

## Rollback Procedure

If mitigation fails or uncertainty, rollback:
```
git revert -m 1 <commit-hash>
firebase deploy --only functions --project hmatologia2
firebase deploy --only hosting --project hmatologia2
```

## Post-Incident

- [ ] Schedule RCA within 48h
- [ ] Tag incident in Slack with `#incident-[number]`
- [ ] Notify auditor if compliance-related (Yellow+)
```

## 2. Root Cause (Single Sentence)

Missing composite index on (labId, createdAt) caused Firestore to fall back to linear scan, timing out the `createRun` Cloud Function after 30 seconds.

---

## 3. Contributing Factors

- **Factor 1:** Index creation checklist in deploy-protocol.md was not explicitly linked to pre-deploy validation (humans skipped it).
- **Factor 2:** Firestore index latency monitoring was not automated (manual spot-check only).
- **Factor 3:** Load test pre-deploy did not simulate 1000+ docs/lab (we tested <100).

---

## 4. Immediate Mitigation (What We Did)

1. Deployed composite index (labId, createdAt) via Firebase Console
2. Increased Cloud Function timeout from 30s to 60s (temporary while index built)
3. Paginated query results to 500 docs/batch (prevents future linear scans)
4. Sent customer notification email + status update every 5 min

**Actions Taken During Incident:** 3 (index, timeout, pagination)  
**Data Loss:** 0 (audit trail intact; failed runs auto-retried)  
**Customer Impact:** 10 min read-only for run creation; all retries succeeded post-resolution

## 7. Lessons Learned

**What Surprised Us:**
1. Firestore degradation felt like a function timeout (not a data layer issue) — Cloud Logs initially showed only function metrics, not query latency breakdown.
2. Index recommendations in Firebase Console were easy to miss (buried in Performance tab) — added to daily monitoring checklist.

**What Went Well:**
1. IC decision was fast (5 min); team was well-coordinated.
2. Customer communication template worked (customers replied with supportive messages, not complaints).
3. Rollback plan was clear, but not needed (hotfix worked).

**What We'll Do Differently:**
1. Load test pre-deploy includes 10x expected peak (1000+ docs/lab).
2. Pre-deploy gate now validates index coverage (no more manual checklists).
3. Firestore query performance is monitored continuously (alert at P50 >500ms).
