## Architectural Decisions (ADR-0022–0026)

### ADR-0022 — CAPA Closure Workflow: 5-State Machine

**Decision:** 5-state CAPA machine enforced server-side via Cloud Function callables with append-only audit trail per transition. Evidence chain-of-custody via content hash (SHA-256).

**RDC 978 Coverage:** Arts. 147 (CAPA lifecycle), 166–180 (audit trail), 24 (records immutability)

**STRIDE Mitigations:**
- **Spoofing:** Evidence hash (SHA-256) at submission; mismatch = rejection
- **Tampering:** Evidence immutable post-submission (Firebase Storage rules deny deletion)
- **Repudiation:** `auditorIdAprovador` field + chainHash prevents editing
- **Information Disclosure:** Firebase Storage rules restrict access to lab members + auditors
- **Denial of Service:** Implicit SLA: auditor dashboard tracks pending >7d
- **Elevation of Privilege:** Only `isAuditorMasterCIQ` can approve via callable

---

### ADR-0023 — Critical Values Escalation: 4-State Machine + Twilio SMS

**Decision:** 4-state machine (NORMAL → CRITICO → ALERTADO → RESOLVIDO) with Twilio SMS escalation. Physician ACK within 15 min; escalation auto-triggers at 10 min mark.

**RDC 978 Coverage:** Arts. 184–191 (notification + SLA), 161 (audit trail)

**STRIDE Mitigations:**
- **Spoofing:** Twilio webhook includes `From` phone; verify against physician phone record
- **Tampering:** SMS inherently insecure; mitigate by never including patient name or lab name in SMS
- **Repudiation:** Twilio message ID logged; webhook ACK proves delivery
- **Information Disclosure:** SMS template anonymized ("Lab Alert: Glucose 35 [CRITICAL]")
- **Denial of Service:** Rate limit max 5 ACK replies per minute; duplicate ignored
- **Elevation of Privilege:** Only `isAdminOrOwner` role can escalate via callable

---

### ADR-0024 — Patient Portal Authentication: HMAC-Signed Email-Link Tokens

**Decision:** Stateless HMAC-signed email-link tokens (7-day TTL), no password storage, no OAuth delegation. Token verification via server-side HMAC recompute; custom Firebase token issued on verification (30-day TTL).

**RDC 978 Coverage:** Arts. 166–180 (delivery proof), 115 (LGPD consent), 24 (records)

**STRIDE Mitigations:**
- **Spoofing:** Email authentication via SPF/DKIM; token includes lab ID (mismatch rejected)
- **Tampering:** HMAC-SHA256 over token; any byte change invalidates signature
- **Repudiation:** Sendgrid logs all sends with timestamp; Firestore audit trail captured
- **Information Disclosure:** 7-day token TTL (short window); email includes "if unsolicited, contact lab" warning
- **Denial of Service:** Rate limit max 3 link requests per patient per 24h
- **Elevation of Privilege:** Custom token includes `patientId`; Firestore rule enforces `request.auth.uid === laudo.patientId` (read own laudo only)

---

### ADR-0025 — IA Strip Classification: Gemini 2.5 Flash Vision API

**Decision:** Gemini 2.5 Flash Vision API for equipment-agnostic OCR (H550 hematology, coagulation, immunology, urinalysis); equipment-specific Zod schemas; confidence scoring (<80% flags manual review); append-only audit trail.

**RDC 978 Coverage:** Arts. 78 (QC validation), 24 (records), 161 (IA audit)

**STRIDE Mitigations:**
- **Spoofing:** Image filename pattern validation; chemically implausible values flagged by range validation
- **Tampering:** Cloud Function stores raw Gemini response + parsed result (both immutable post-creation)
- **Repudiation:** Full Gemini response stored; can be replayed for verification
- **Information Disclosure:** Anonymize patient ID from images if possible; LGPD DPA with Google covers processor liability
- **Denial of Service:** Google Cloud quota limit enforced (e.g., 100 inferences/day)
- **Elevation of Privilege:** Firestore rules deny client-side writes to `/ia-ocr-results` (function creates only)

### ADR-0026 — NOTIVISA Queue Processing: Async Append-Only Outbox Pattern

**Decision:** Async append-only Firestore outbox collection (`/labs/{labId}/notivisa-outbox`) with scheduled processor (every 5 min); exponential backoff (1m → 5m → 15m → 45m → 120m); max 5 retries; idempotent API calls via deduplication key (SHA-256). Phase 8: mock submitter. Phase 12+: real Anvisa SOAP API.

**RDC 978 Coverage:** Art. 66 (24h deadline), Portaria 204 (disease codes), 24 (records)

**STRIDE Mitigations:**
- **Spoofing:** Certificate-based auth for Phase 12+ (lab X.509 cert required for Anvisa API)
- **Tampering:** Submitted form + idempotency key + receipt code logged (immutable); Anvisa receipt proves exact data
- **Repudiation:** Anvisa webhook receives `eventId` + timestamp; bidirectional confirmation (lab → Anvisa + Anvisa → lab)
- **Information Disclosure:** Form includes minimal PII (name_anon, DOB, gender only); LGPD DPA with Anvisa covers disclosure
- **Denial of Service:** Idempotency key deduplication prevents duplicate submissions; Anvisa rejects duplicate keys
- **Elevation of Privilege:** Firestore rules deny client-side writes; operator cannot update form payload after creation

## Auditor RFI Responses

### RFI-1: How does CAPA closure enforce immutability? (ADR-0022)

**Answer:** CAPA 5-state machine transitions are enforced server-side via Cloud Function callable (`capaOpenNewCAPAWorkflow`, `capaSubmitEvidence`, `capaAuditorApprove`). Each transition appends an event to `/labs/{labId}/capaWorkflow/{capaId}/transicoesCAPAs[]` with immutable LogicalSignature (HMAC-SHA256 hash). Evidence files stored in Firebase Storage are immutable post-upload (Firestore rules deny deletion). Auditor approval is non-repudiable: `auditorIdAprovador` field captures approver; chainHash over approval record prevents editing; timestamp enforced. Once state=fechado, Firestore rules block any further updates (read-only thereafter).

**Compliance:** RDC 978 Arts. 5.3 (audit trail immutability), 147 (CAPA closure), 166–180 (record retention)

---

### RFI-2: How are critical values escalated with SLA guarantees? (ADR-0023)

**Answer:** Critical values follow 4-state FSM (NORMAL → CRITICO auto-transition when result <criticalMin OR >criticalMax → ALERTADO escalation with SMS/WhatsApp → RESOLVIDO physician acknowledgment). Auto-transition to CRITICO is deterministic (thresholds from `labSettings.{labId}.criticalThresholds[analyte]`). On CRITICO transition, a scheduled Cloud Function (every 2 min) detects pending escalations and sends Twilio SMS to on-call physicians. Escalation occurs <2 min of result entry. Physician must ACK via SMS reply "ACK" or voice call within 15-min SLA window. SLA timer starts at CRITICO ts; escalation auto-resends at 10-min mark if no ACK. Past 15-min SLA, supervisor alert sent via email + SMS. All events logged immutably in `atualizacaoCritico[]` with chainHash.

**Compliance:** RDC 978 Arts. 184–191 (critical notification, SLA, responsibility), 161 (audit trail)

---

### RFI-3: How is patient portal auth stateless + LGPD compliant? (ADR-0024)

**Answer:** Patient portal uses stateless HMAC-signed email-link tokens (no password, no DB session storage). Lab generates token=HMAC-SHA256(patientId:labId:ts:secret), embeds in email link, expires in 7 days. Patient clicks link → server recomputes HMAC, verifies signature. On success, server issues temporary custom Firebase token (30-day TTL, includes `patientId` claim). Client uses this token to read own laudo (Firestore rule enforces `request.auth.uid === laudo.patientId`). No password means zero password breach risk. HMAC is lab-issued + verified (no 3P OAuth delegation). Email link delivery = proof of identity (RDC 978 Art. 117 compliance). LGPD compliance: patient email includes consent checkbox ("I consent to view my result"); 7-day email TTL + 30-day session TTL minimize PII window; audit trail logs all patient portal access in `/labs/{labId}/audit-patient-portal` (read-only, separate from patient-writable collections).

**Compliance:** LGPD Arts. 9 (lawful basis), 18 (consent), 38 (minimal delegation); RDC 978 Arts. 115 (patient communication), 166–180 (delivery proof)

---

### RFI-4: How does Gemini Vision ensure audit trail for OCR decisions? (ADR-0025)

**Answer:** Gemini Vision OCR is invoked server-side via Cloud Function callable (never client-side). Each invocation stores: (1) raw Gemini response (JSON with confidence per field), (2) equipment-specific Zod schema version, (3) fuzzy-match results (Levenshtein distance + semantic heuristics), (4) operator ID (who submitted image), (5) confidence scores per analyte, (6) final decision (accept/manual-review/reject), (7) timestamp. All stored immutably in `/labs/{labId}/ia-ocr-results/{resultId}` (Cloud Function creates; Firestore rules deny client-side writes). Manual corrections stored separately in `/labs/{labId}/ia-ocr-corrections/{correctionId}` with `correctedBy`, `ts`, `reason`. Monthly accuracy report aggregates Gemini inference results vs. operator corrections (feedback loop for future fine-tuning). Equipment mapping verified: H550 images must match H550 schema; coagulation images must match coagulation schema (Zod validation enforces). Confidence scoring: >95% = auto-accept, 85-95% = yellow warning (manual review recommended), <80% = red flag (manual review required).

**Compliance:** RDC 978 Arts. 78 (QC validation), 161 (IA equipment validation), 24 (records immutability); DICQ 4.2.1 (data acquisition traceability)

---

### RFI-5: How does NOTIVISA queue handle API failures + idempotency? (ADR-0026)

**Answer:** NOTIVISA form submissions are queued asynchronously in `/labs/{labId}/notivisa-outbox` with append-only `submissionAttempts[]` array. Scheduled processor (every 5 min) detects new entries and attempts submission. Phase 8: mock submitter (instant success). Phase 12+: real Anvisa SOAP API with certificate-based auth (lab X.509 cert). Idempotency enforced via deduplication key=SHA-256(labId:diseaseCode:patientDOB:resultDate); Anvisa API rejects duplicate keys (returns same receipt, no duplicate event). Exponential backoff: 1st retry after 1m, 2nd after 5m, 3rd after 15m, 4th after 45m, 5th after 120m. After 5 retries → permanent failure marked; supervisor alert sent (SMS + email). 24-hour deadline enforced: if submission not acknowledged within 24h of result date → escalation to RT manager. Anvisa webhook (`handleAnvisaWebhook`) receives `eventId` + timestamp, updates outbox entry with `anvisa_eventId` + `anvisa_acknowledged_ts` (proof Anvisa processed). All attempts logged immutably (no deletes, no edits).

**Compliance:** RDC 978 Art. 66 (24h deadline, Anvisa notification), Portaria 204 (disease codes), 24 (records immutability); DICQ 4.5.4 (regulatory submissions audit trail)
