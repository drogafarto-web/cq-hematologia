## 5. Contingency Triggers

When risk scores exceed thresholds, contingency plans auto-trigger. This section defines when escalation and action plans activate.

### Severity-Based Triggers

#### 🔴 CRITICAL (Score 7–9)

**Auto-trigger conditions:**
- Risk score 7–9 (P3×I2, P2×I3, or worse)
- Patient safety incident detected
- Production data exposure confirmed
- RDC 978 / DICQ compliance breach in audit

**Automatic actions:**
1. CTO immediate notification (Slack direct message + email)
2. Emergency war room convened (within 4h)
3. Daily standup (until score reduced to ≤6)
4. Escalation to lab director (if compliance/safety incident)
5. Phase pause if blocking critical path

**Threshold to De-Escalate:**
- Score reduced to ≤6 via mitigation evidence
- Root cause resolved + preventive action deployed

### Trigger-Specific Contingencies

#### RISK-402 (Auditor Availability)

**Trigger fires when:** RFI cycle >5 business days for ANY single CAPA

```
ON TRIGGER:
  Day 1: QA Lead contacts auditor + lab director (parallel)
  Day 2: CTO reviews CAPA evidence; identifies if truly auditor-blocked or internal gap
  Day 3: Parallel CAPA gathering without auditor (offline evidence collection)
  Day 5: Escalate to lab director + compliance consultant if no resolution
  Day 7: CTO override; extend Phase 4 timeline by 1 week; accept risk of late audit sign-off

CONTINGENCY PLAN ACTIVITIES:
  - Offline evidence gathering (screenshots, logs, RCA docs prepared independently)
  - Auditor availability negotiation (offer alternative times/formats)
  - Evidence pre-staging (items prepared so auditor only validates, not requests rework)
  - Rollback: if Phase 4 blocked >10 days, pause Wave 2 (Phase 5+) and focus CAPA closure
```

#### RISK-406 (Schema Migration Flaws)

**Trigger fires when:** Rules test failure OR schema bug in Phase 3–4

```
ON TRIGGER:
  Hour 0: Tech Lead + Stream A investigate (root cause in 4h)
  Hour 4: Decision: hot-patch (additive) vs rollback (breaking change)
  Hour 8: If hot-patch: deploy to staging; rules re-test; 24h observation
  Hour 32: Promote to prod if clean
  Day 2: If rollback needed: restore backup; re-plan Phase 3 approach

CONTINGENCY PLAN ACTIVITIES:
  - Staging-only code path (disable new features if risky)
  - Index pre-build (offline before prod deploy)
  - Rollback procedure: firestore restore from backup (24h to restore; acceptable for non-critical)
  - Communication: Phase 4+ delay by 1 week if Phase 3 rollback needed
```

#### RISK-407 (CAPA Evidence Gaps)

**Trigger fires when:** Weekly evidence checklist shows ≥2 CAPAs missing artifacts

```
ON TRIGGER:
  Day 1: CAPA Process Lead identifies gaps; assigns owners to backfill
  Day 2: Evidence gathering sprint (1-day focus); deadline EOD
  Day 3: Auditor notification (if escalating, include evidence status update)
  Day 5: If still ≥2 missing, CTO escalates; extend Phase 4 by 2 days

CONTINGENCY PLAN ACTIVITIES:
  - Evidence template provided to CAPA owner (fill-in guidance)
  - Screenshots: automated via test recordings (faster than manual)
  - Logs: pre-staged in Cloud Logging (query template provided)
  - Auditor pre-call: confirm "good enough" standard (avoid over-documentation)
  - Week 8 contingency: full 1-week buffer dedicated to evidence gap closure
```

#### RISK-408 (Context Switching)

**Trigger fires when:** Bug rate >3/phase OR engineer reports >20% context-switch overhead

```
ON TRIGGER:
  Day 1: Tech Lead + CTO review engineer allocation; identify overlap
  Day 2: Re-allocate to dedicated streams (remove dual-stream assignments)
  Day 3: Increase standup frequency (daily 15-min becomes daily 30-min + Friday review extended)
  Day 5: Engineering velocity survey (confirm improvement)

CONTINGENCY PLAN ACTIVITIES:
  - Reduce parallel streams from 4 to 2 (if bug rate >5/phase persists)
  - Reallocate to critical path (RISK-402, RISK-407 priority)
  - Code review backlog clearance (sprint to reduce PR wait time)
  - Async decision-making: document stream decisions in README per phase (reduce meetings)
```

#### RISK-416 (Scope Creep)

**Trigger fires when:** Customer/sales escalates multi-tenant demand to CTO

```
ON TRIGGER:
  Hour 0: CTO reviews request; determines scope impact (1–8h analysis)
  Hour 4–8: CTO + Sales + Customer call (clarify intent, scope, timeline)
  Day 1: Decision: Accept with scope re-plan / Defer to v1.4.1 / Negotiate phased rollout
  Day 2: Communicate decision + rationale to all stakeholders

CONTINGENCY PLAN ACTIVITIES:
  - If ACCEPTED: Pause v1.4 roadmap; publish revised ROADMAP + REQUIREMENTS (scope expansion)
    - Stakeholder communication: new v1.4 launch date (likely Q3 2026)
    - Engineering impact: +4 weeks (estimated multi-tenant implementation)
  - If DEFERRED: Reaffirm v1.5 commitment; link to product roadmap
    - Sales playbook emphasizes v1.5 features + "early access" option for v1.4.1
  - If PHASED: Negotiate phased rollout (e.g., single-lab v1.4, multi-lab v1.4.1 + hot-fix)

CONTINGENCY ESCALATION AUTHORITY:
  - CTO final decision (no scope change without CTO sign-off)
  - Sales Lead owns customer communication (avoid confusion)
  - Finance approval if scope change impacts budget >$50K
```

#### RISK-418 (Cloud Cost Overrun)

**Trigger fires when:** Any service >80% of monthly budget

```
ON TRIGGER:
  Immediate: GCP Console quota alert (automated)
  Hour 1: Stream D reviews cost report; identifies top consumer (Gemini, Twilio, or Firebase)
  Hour 4: Implement throttle:
    - Gemini: reduce batch size or confidence threshold
    - Twilio: email fallback for non-critical SMS
    - Firebase: increase cold-start timeout (less aggressive pre-warming)
  Day 1: CTO reviews decision; approves throttle or provisioning increase

CONTINGENCY PLAN ACTIVITIES:
  - Defer low-priority features (disable IA on non-critical samples)
  - Batch processing (consolidate calls, fewer invocations)
  - Caching (avoid re-processing identical input)
  - Renegotiate vendor terms (if >$300 overage, contact Gemini/Twilio sales)

CONTINGENCY ESCALATION:
  - Stream D → CTO: decision on throttle vs overage approval
  - Finance: if total overage >$200, Finance approval required (CTO + Finance sign-off)
```
