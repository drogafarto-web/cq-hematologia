# v1.4 Post-Launch Incident Runbook — First 48 Hours

**Status:** PRINTED & TAPED TO MONITOR  
**Monitoring Window:** T+0 (deployment starts) through T+48h (stabilization)  
**Owner:** On-call engineer + RT (técnico responsável) + CTO (escalation)  
**Escalation SLA:** P0: 5 min acknowledgment, P1: 15 min response, P2: 1h response

---

## Table of Scenarios

This runbook covers **7 critical incident scenarios**. Each follows a standard template:

1. **Rules deny legitimate users** (RDC lockout)
2. **Cloud Functions 5xx rate >5%** (function crash)
3. **Firestore quota breach** (read/write spike)
4. **NOTIVISA API timeout** (gov endpoint down)
5. **Gemini Vision API cost spike** (Phase 11 IA)
6. **SMS delivery failure** (Twilio issue)
7. **Critical data loss** (accidental delete)

Plus one **cross-cutting flow**: **Cascading failure** (multiple systems down) → full rollback to v1.3.

## SCENARIO 1: Rules Deny Legitimate Users (RDC Lockout)

### Detection Signal

**Primary:** Error spike in Cloud Logs

```
resource.type="cloud_firestore"
AND text contains ("Permission denied" OR "PERMISSION_DENIED")
AND severity=ERROR
```

**Secondary:** User reports "Access denied to CIQ module" or "Cannot save run"

**Severity Threshold:** >10 permission denied errors in 15 min window = **P0 IMMEDIATE ROLLBACK**

### Decision Tree

```
Q1: Are errors ONLY on reads (not writes)?
  → YES: Likely rules query filter too restrictive
    → Check Cloud Logs for exact collection path + user labId
    → Review deployed rules for `match /labs/{labId}/**` scope
    → Proceed to MITIGATION (deploy rule fix, no full rollback)

  → NO: Errors on writes too (create/update/delete)?
    → CRITICAL: Entire feature blocked
    → Proceed to EMERGENCY ROLLBACK (below)

Q2: Is the lockout affecting ALL users or subset?
  → All users: Rules logic broken (e.g., member check inverted)
  → Subset: Likely onboarding or role assignment bug (investigate in parallel)
    → Still escalate P0 but may NOT require immediate rollback if subset <5%
```

### Mitigation Steps (15–30 min)

**Step 1: Confirm scope (2 min)**

- Run this Cloud Logs query:
  ```bash
  gcloud logging read \
    'resource.type="cloud_firestore" AND severity="ERROR"' \
    --project=hmatologia2 \
    --limit=50 \
    --format='table(timestamp,jsonPayload.message,labels.user_id)' \
    --filter='timestamp >= now - 15m'
  ```
- Count errors; identify affected collections (e.g., `/labs/{labId}/ciq-coagulacao/runs`)
- Check if labId appears in every error (scope = lab-wide) or only specific docs

**Step 2: Assess severity (3 min)**

- If <5 errors AND errors only on read queries → proceed to deploy fix
- If >20 errors AND blocking all writes → trigger EMERGENCY ROLLBACK (skip to end of this scenario)

**Step 3: If deploying a fix (15–20 min total)**

- Checkout a new branch: `git checkout -b incident-rules-fix-<timestamp>`
- Locate `firestore.rules` in project root
- Identify the broken `match` block (referenced in error message)
  - **Example:** if error is on `/labs/{labId}/ciq-coagulacao/runs`, check `match /labs/{labId}/ciq-coagulacao/runs`
- **Common fixes:**
  - Inverted permission: `allow read: if !isActiveMemberOfLab()` → should be `allow read: if isActiveMemberOfLab()`
  - Missing labId check: ensure `request.path[2] == resource.labId`
  - Wrong field name in `member` doc check
- Redeploy only rules (not functions, not hosting):
  ```bash
  firebase deploy --only firestore:rules --project hmatologia2
  ```
- Tail logs for 5 min to confirm permission denied errors stop:
  ```bash
  gcloud logging read \
    'resource.type="cloud_firestore" AND severity="ERROR"' \
    --project=hmatologia2 \
    --follow
  ```

**Step 4: Communicate (2 min)**

- Slack to RT + CTO: "**INCIDENT:** RDC lockout detected on [COLLECTION]. Deployed rules fix [COMMIT HASH]. Error rate: [COUNT]/15min → resuming monitoring."

### If Mitigation Fails → Proceed to Emergency Rollback

**Emergency Rollback (5 min):**

```bash
# Revert to last known-good rules commit
git log --oneline firestore.rules | head -5
# Identify last commit where rules were known working (usually HEAD~1 or earlier)
git checkout <GOOD_COMMIT_HASH> -- firestore.rules

# Deploy
firebase deploy --only firestore:rules --project hmatologia2

# Verify errors stop (tail for 5 min)
gcloud logging read \
  'resource.type="cloud_firestore" AND text contains "Permission denied"' \
  --project=hmatologia2 \
  --follow
```

### Communication Script

**To RT (técnico responsável):**

> "RDC lockout occurred — users unable to access [MODULE]. We've rolled back Firestore rules to the previous version. System is recovering. No data loss. Full incident review in 2h."

**To Auditor (if incident log triggered):**

> "Incident: Temporary access restriction due to security rules deployment. Root cause: [fix description]. Rolled back to v1.3 rules. Impact: [# of users × # of minutes affected]. Preventive: [testing improvement]. Audit trail preserved in auditoria collection."

## SCENARIO 2: Cloud Functions 5xx Rate >5% (Function Crash)

### Detection Signal

**Primary:** Error spike in Cloud Logs

```
resource.type="cloud_function"
AND (httpStatus=500 OR httpStatus=502 OR httpStatus=503)
```

**Secondary:** User reports "Submit failed" or Cloud Function callable times out

**Severity Threshold:** >50 5xx errors in 10-min window OR any single function >20% error rate = **P0**

### Decision Tree

```
Q1: Is this a cold-start issue?
  → Check latency in error log: if >30s, likely timeout after cold-start
  → Cold-starts are expected on first invoke post-deploy
  → If recurring, increase function memory (512MB → 1GB) and redeploy

Q2: Which function is failing? (Check cloud logs for `functionName` field)
  → Single function failing (e.g., `generateMonthlyReportBioquimica`)
    → Can rollback just that function (faster recovery)
  → Multiple functions failing
    → Likely shared dependency or environment issue
    → May need full functions rollback

Q3: Do errors show a pattern?
  → All errors reference same module (e.g., all Bioquímica functions)
    → Check function `tsconfig.json` for import path issue
    → Check if new dependency missing in `functions/package.json`
  → Errors random across modules
    → Likely infra issue (quota, secret not provisioned)
```

### Mitigation Steps (10–20 min)

**Step 1: Identify failing function (2 min)**

```bash
# Get error details
gcloud logging read \
  'resource.type="cloud_function" AND severity="ERROR"' \
  --project=hmatologia2 \
  --limit=20 \
  --format='table(timestamp,labels.function_name,jsonPayload.message)' \
  --filter='timestamp >= now - 15m'
```

Record the `function_name` (e.g., `generateMonthlyReportBioquimica`, `submitLaudo`, etc.)

**Step 2: Rollback single function or all (choice based on Q2 above)**

**Option A: Rollback single function (5 min)**

- Revert function code to previous commit:
  ```bash
  git log --oneline functions/src/<module>/<functionName>.ts | head -1
  git checkout HEAD~1 -- functions/src/<module>/<functionName>.ts
  cd functions && npm run build
  firebase deploy --only functions:<functionName> --project hmatologia2
  ```

**Option B: Rollback all functions (8 min)**

- If multiple functions failing or unclear which is culprit:
  ```bash
  git checkout HEAD~1 -- functions/
  cd functions && npm run build
  firebase deploy --only functions --project hmatologia2
  ```
- **Note:** This reverts ALL functions to previous version. Watch for cross-module impact.

**Step 3: Verify recovery (3 min)**

- Tail errors for 5 min:
  ```bash
  gcloud logging read \
    'resource.type="cloud_function" AND severity="ERROR"' \
    --project=hmatologia2 \
    --follow
  ```
- If error rate <0.1%: recovery successful, continue monitoring
- If errors continue: escalate to CTO + debug

### If Mitigation Fails → Check Environment

**Missing secret check:**

```bash
bash scripts/preflight-secrets-check.sh
```

- Output will list any `defineSecret('NAME')` that resolved to `PENDING_SET_*`
- If found, contact CTO to provision the secret in Firebase:
  ```bash
  firebase functions:secrets:set <SECRET_NAME> --project hmatologia2
  # Then redeploy affected functions
  firebase deploy --only functions --project hmatologia2
  ```

### Communication Script

**To RT:**

> "Cloud Functions error rate spiked to [X]% on [FUNCTION_NAME]. Rolled back to previous version. Investigating root cause. System recovering."

**To Auditor:**

> "Incident: Temporary service unavailability on [FUNCTION]. Root cause: [code issue / missing config]. Rollback time: [duration]. Impact: [# callable invocations failed]. Audit trail: all writes to auditoria collection preserved."

## SCENARIO 3: Firestore Quota Breach (Read/Write Spike)

### Detection Signal

**Primary:** Firestore quota warning in GCP Console

```
Firestore Dashboard → Database Usage → Daily billable read/write operations exceeding quota
```

**Secondary:** Cloud Logs spike

```
resource.type="cloud_firestore"
AND text contains ("Request rate exceeded" OR "quota" OR "too many writes")
```

**Severity Threshold:** Sustained >100 reads/sec above baseline for >5 min = **P1**

### Decision Tree

```
Q1: Is the spike expected?
  → New feature launched that triggers many reads? (e.g., analytics dashboard polling)
  → Check PERFORMANCE_PATTERNS.md for guardrails on polling intervals
  → If interval <30s or reads unbounded: disable feature temporarily, fix code, redeploy

Q2: Is quota actually breached or just spiking?
  → Check GCP Firestore Console: "Daily billable reads" vs quota
  → If <95% of quota: likely auto-throttle happening (not critical, will recover)
  → If >95%: need to request increase or implement rate limiting

Q3: Can you identify the culprit?
  → Run: gcloud logging read 'resource.type="cloud_firestore"' with limit 100
  → Parse for collection accessed (e.g., `analytics/runs`, `sgq/documents`)
  → Check if new listener added or query is unbounded
```

### Mitigation Steps (10–60 min)

**Step 1: Identify spike source (5 min)**

```bash
# Get top collections by read volume
gcloud logging read \
  'resource.type="cloud_firestore" AND operation.type="Read"' \
  --project=hmatologia2 \
  --limit=100 \
  --format='table(timestamp,resource.labels.database)' \
  | awk '{print $NF}' | sort | uniq -c | sort -rn
```

If spike is NOT correlated with recent code deploy, likely **unexpected user behavior** (e.g., automated external tool hitting API). Proceed to rate limiting.

If spike IS correlated with deploy, proceed to code fix.

**Step 2: Code fix (if spike caused by deploy)**

Identify the problematic code:

- **Polling interval too fast:** `useEffect` fires every <30s without meta diff check
- **Unbounded query:** reading entire collection without `.limit()` or date range
- **Nested listener:** creating one listener per item in a list (N listeners for N items)

**Action:**

```bash
# Find the file
grep -r "onSnapshot\|getDocs" src/features/<MODULE> | grep -v node_modules | head -20

# Review the hook for:
# 1. Polling interval (should be >=30s)
# 2. Query limits (should have .limit() or date range)
# 3. Listener count (should be 1 per path, not N per item)

# Fix + test locally + redeploy
git checkout -b incident-quota-fix-<timestamp>
# (fix code)
npm run build && npm run test
firebase deploy --only hosting --project hmatologia2
```

**Step 3: Rate limiting / temporary feature disable**

If fix is complex or user-driven spike (not code bug), temporarily disable the high-read feature:

```bash
# Disable feature flag in client code (conditional render)
# Example: disable analytics dashboard polling
// src/features/analytics/useAnalyticsPolling.ts
const POLLING_ENABLED = false; // Temporarily disabled due to quota spike

firebase deploy --only hosting --project hmatologia2
```

**Step 4: Request quota increase (if needed)**

If spike is legitimate (e.g., lab processing 10x normal volume), request GCP quota increase:

1. Go to [GCP Console → Firestore → Quotas](https://console.cloud.google.com/firestore/quotas?project=hmatologia2)
2. Click **Edit Quotas** for `cloud.firestore/backend_read_requests` or `backend_write_requests`
3. Request increase (e.g., from 100K to 500K daily operations)
4. Google typically approves within 24h

**Step 5: Communicate & monitor (ongoing)**

```bash
# Monitor recovery
gcloud logging read \
  'resource.type="cloud_firestore"' \
  --project=hmatologia2 \
  --follow | grep -E "(quota|rate|limit)"
```

### Communication Script

**To RT:**

> "Firestore quota breach detected. Root cause: [feature causing spike]. Temporarily disabled analytics polling. Quota increase request submitted (24h approval expected). No data loss."

**To Auditor:**

> "Incident: Temporary service degradation due to read quota spike. Root cause: [analytics polling on all runs]. Mitigation: disabled feature. Impact: <5 min latency. System recovered. Quota increase pending."

## SCENARIO 4: NOTIVISA API Timeout (Gov Endpoint Down)

### Detection Signal

**Primary:** Error in Cloud Logs

```
resource.type="cloud_function"
AND text contains ("NOTIVISA" OR "notivisa.anvisa.gov.br" OR "deadline exceeded")
AND severity="ERROR"
```

**Secondary:** User reports "Cannot submit notification" in laudo liberação flow

**Severity Threshold:** >5 consecutive NOTIVISA timeouts = **P1 (not P0, gov timeout is expected occasionally)**

### Decision Tree

```
Q1: Is NOTIVISA endpoint actually down?
  → Check ANVISA status page: https://notivisa.anvisa.gov.br/status
  → Or: try manual curl from Cloud Function environment
  → If down on ANVISA side: this is expected, proceed to queuing strategy

Q2: Are we retrying failed submissions?
  → Check if submission function has exponential backoff + retry logic
  → If no retry: implement as part of fix (don't let failed submissions drop)

Q3: Is the timeout affecting laudo release to patient?
  → If notification is ASYNC (sent separately after laudo released): no blocker
  → If notification is SYNC (must succeed before laudo visible): blocker
```

### Mitigation Steps (5–20 min)

**Step 1: Verify endpoint status (2 min)**

```bash
# From Cloud Shell or local:
curl -I https://notivisa.anvisa.gov.br/api/status --connect-timeout 10

# If timeout or connection refused: ANVISA is down
# If 200 OK but slow: ANVISA is flaky

# Check recent Cloud Logs for timeout pattern
gcloud logging read \
  'text contains "NOTIVISA" AND severity="ERROR"' \
  --project=hmatologia2 \
  --limit=10
```

**Step 2: Queue submissions for retry (5–15 min)**

Implement a **submission queue** that retries failed notifications:

**Option A: Cloud Task queue (recommended for v1.4)**

Create a Cloud Tasks queue to hold failed submissions:

```bash
# Create queue if not exists
gcloud tasks queues create notivisa-retry-queue \
  --location=southamerica-east1 \
  --project=hmatologia2 || echo "Queue already exists"
```

Then in the laudo liberação callable, catch NOTIVISA timeout and enqueue for retry:

```typescript
// functions/src/liberacao/submitNotification.ts
async function submitNotification(laudoId: string) {
  try {
    await callNotivisaAPI(laudoId);
  } catch (error) {
    if (error.message.includes('deadline exceeded')) {
      // Queue for retry in 5 minutes
      const task = {
        httpRequest: {
          uri: `https://southamerica-east1-hmatologia2.cloudfunctions.net/submitNotification`,
          headers: { 'Content-Type': 'application/json' },
          body: Buffer.from(JSON.stringify({ laudoId })).toString('base64'),
          httpMethod: 'POST',
        },
        scheduleTime: new Date(Date.now() + 5 * 60000),
      };
      await tasksClient.createTask({ parent: queuePath, task });
      // Laudo still released even if notification queued
      return { success: true, notificationQueued: true };
    }
    throw error;
  }
}
```

**Option B: Pub/Sub topic (if you need durable long-term queue)**

Similar pattern but using Cloud Pub/Sub instead of Tasks.

**Step 3: Redeploy + monitor (5 min)**

```bash
firebase deploy --only functions:submitNotification --project hmatologia2

# Monitor submission queue
# Check if tasks are being processed
gcloud tasks queues describe notivisa-retry-queue \
  --location=southamerica-east1 \
  --project=hmatologia2
```

**Step 4: Alert RT to manual retry (if queue fills)**

If queue backs up (>100 pending tasks):

- Notify RT: "NOTIVISA notifications queued. Will retry every 5 min. No action needed."
- Monitor ANVISA status for recovery
- Once ANVISA recovers, queue will auto-drain

### Communication Script

**To RT:**

> "NOTIVISA API is temporarily unavailable (ANVISA side). We've queued failed notifications for automatic retry every 5 minutes. Laudos are still being released to patients. No manual action needed."

**To Auditor:**

> "Incident: Third-party API timeout (NOTIVISA endpoint). Mitigation: implemented retry queue. Notifications will be submitted when endpoint recovers. Impact: <1 min delay on notification. Audit trail of queue status in Cloud Logs."

## SCENARIO 5: Gemini Vision API Cost Spike (Phase 11 IA Running Hot)

### Detection Signal

**Primary:** GCP Billing alert or Cloud Logs spike

```
resource.type="cloud_function"
AND labels.function_name="analyzeYumizen*"
AND text contains ("Gemini" OR "vertexai")
```

**Secondary:** User reports slow analyzer response (>30s per sample)

**Severity Threshold:** Gemini token usage >10x baseline in 1h window = **P2 (cost issue, not availability)**

### Decision Tree

```
Q1: Are all API calls from Gemini?
  → Yes: analyzer is calling Gemini on every sample (expected)
  → Check cost: Gemini Vision 1.5 Flash = ~$0.075 per 1M tokens
  → If daily cost >$100: likely something wrong with prompt (too large, unbounded loop)

Q2: Is the problem:
  → Slow response (latency): reduce confidence threshold or batch size
  → High token count: review prompt, maybe using wrong model (4.0 vs Flash)
  → Cost unsustainable: disable feature for low-confidence samples, require manual review instead

Q3: Is this Phase 11 IA or Phase 9 Bioquímica analyzer?
  → Phase 11 (newer imaging IA): may not be production-optimized yet, can disable temporarily
  → Phase 9 Bioquímica (OCR, well-tested): less likely to cause spike, investigate infra
```

### Mitigation Steps (15–45 min)

**Step 1: Identify spike source (3 min)**

```bash
# Check which function is calling Gemini
gcloud logging read \
  'text contains "Gemini" AND severity="WARNING"' \
  --project=hmatologia2 \
  --limit=20 \
  --format='table(timestamp,labels.function_name,jsonPayload.message)'
```

Record which module (e.g., `analyzePhase11ImageAI`, `analyzeYumizen`).

**Step 2: Check confidence threshold (3 min)**

If Phase 11 IA (image analysis) is the culprit:

```typescript
// functions/src/phase11/<MODULE>/analyzeImageAI.ts
const CONFIDENCE_THRESHOLD = 0.85; // Reduce from 0.95 to 0.85
// This filters out low-confidence results before Gemini processing
```

Update and redeploy:

```bash
firebase deploy --only functions:analyzePhase11* --project hmatologia2
```

**Step 3: Disable feature temporarily (if spike critical)**

If cost is spiraling (>$500/day projected):

```typescript
// functions/src/phase11/imageAnalyzer.ts
const PHASE_11_IA_ENABLED = false; // Temporarily disabled due to cost spike
```

Deploy + notify RT to use manual analysis until cost optimized.

**Step 4: Review Gemini calls for inefficiency (10–30 min)**

Check the Gemini prompt in `functions/src/phase11/imageAnalyzer.ts`:

- Is the image being sent at full resolution? (crop to relevant region)
- Is the prompt too long or asking for redundant things?
- Are we calling Gemini for every sample or only when needed?

**Example optimization:**

```typescript
// BEFORE: Full image + long prompt
const response = await model.generateContent({
  contents: [
    { role: 'user', parts: [{ inlineData: { data: base64Image, mimeType: 'image/png' } }] },
  ],
  systemInstruction: 'Analyze this hematology analyzer output and extract all numeric values...',
});

// AFTER: Cropped image + concise prompt + batch mode
const croppedImage = cropToResultRegion(image); // Only send relevant part
const response = await model.generateContent({
  contents: [
    { role: 'user', parts: [{ inlineData: { data: base64Image, mimeType: 'image/png' } }] },
  ],
  systemInstruction: 'Extract hematology values: WBC, RBC, HGB.',
});
```

Redeploy and monitor:

```bash
firebase deploy --only functions --project hmatologia2
gcloud logging read \
  'text contains "Gemini"' \
  --project=hmatologia2 \
  --follow
```

### Communication Script

**To RT:**

> "Gemini Vision API cost spiked to $[amount]/day. Implemented confidence filtering and optimized prompts. Analyzer response time will improve. Monitoring for next 2h."

**To Finance/CTO:**

> "Incident: Phase 11 IA Gemini cost optimization needed. Root cause: [high-volume low-confidence samples]. Actions: reduced threshold, optimized prompts, disabled feature if needed. Estimated new cost: $[amount]/day (vs $[previous]/day)."

## SCENARIO 6: SMS Delivery Failure (Twilio Issue)

### Detection Signal

**Primary:** Error in Cloud Logs

```
resource.type="cloud_function"
AND (text contains "Twilio" OR text contains "SMS")
AND severity="ERROR"
```

**Secondary:** User reports "Didn't receive SMS" (lab staff or patient notification)

**Severity Threshold:** >10 SMS failures in 15 min = **P1 (critical for patient communication)**

### Decision Tree

```
Q1: Is Twilio down?
  → Check Twilio status: https://status.twilio.com
  → If maintenance/outage: wait for Twilio recovery or switch provider

Q2: Are we retrying failed SMS?
  → Check Cloud Logs for retry attempts
  → If no retry: implement queuing (similar to NOTIVISA Scenario 4)

Q3: Is the error auth-related (invalid API key)?
  → Check: gcloud logging read with grep for "Unauthorized" or "401"
  → If yes: secret not provisioned, run preflight check
```

### Mitigation Steps (5–15 min)

**Step 1: Verify Twilio status (2 min)**

```bash
# Check status page manually
# or via curl if available
gcloud logging read \
  'text contains "Twilio" AND severity="ERROR"' \
  --project=hmatologia2 \
  --limit=20
```

Look for pattern:

- `"Unauthorized"` → auth issue (missing secret)
- `"Service unavailable"` → Twilio down
- `"Invalid phone number"` → data issue (validate phone on client side)

**Step 2: If auth issue, check secret (2 min)**

```bash
bash scripts/preflight-secrets-check.sh | grep -i twilio
```

If `TWILIO_AUTH_TOKEN` is `PENDING_SET_*`, contact CTO to provision:

```bash
firebase functions:secrets:set TWILIO_AUTH_TOKEN --project hmatologia2
firebase deploy --only functions --project hmatologia2
```

**Step 3: Fallback to email (5 min)**

If Twilio down and recovery unknown:

Implement email fallback in notification function:

```typescript
// functions/src/shared/sendNotification.ts
async function sendNotification(message: string, recipient: User) {
  try {
    // Try SMS first
    await twilioClient.messages.create({
      from: process.env.TWILIO_PHONE,
      to: recipient.phone,
      body: message,
    });
  } catch (error) {
    if (error.message.includes('Twilio') || error.message.includes('Service unavailable')) {
      // Fallback to email
      console.warn('SMS failed, falling back to email');
      await sendEmail(recipient.email, `Notification: ${message}`);
    } else {
      throw error;
    }
  }
}
```

Deploy:

```bash
firebase deploy --only functions --project hmatologia2
```

**Step 4: Notify RT of fallback (2 min)**

Slack to RT: "SMS delivery failing from Twilio (service issue). Falling back to email notifications. No lab data affected, only delivery channel changed."

### Communication Script

**To RT:**

> "Twilio SMS service is having issues. We've switched to email fallback. All notifications will be sent via email until Twilio recovers. No data impact."

**To Auditor:**

> "Incident: Third-party notification service (Twilio) unavailable. Mitigation: automatic fallback to email. Impact: 15 min without SMS, then email. Audit trail: all notifications logged in Cloud Logs."

## SCENARIO 7: Critical Data Loss (Accidental Delete)

### Detection Signal

**Primary:** Manual report from RT or auditor

"A document was accidentally deleted" (e.g., someone ran DELETE query on production)

**Secondary:** Audit trail shows deletion event

```
db.collection('auditoria').where('operationType', '==', 'DELETE')
  .orderBy('timestamp', 'desc').limit(100)
```

**Severity Threshold:** Any production DELETE outside of soft-delete mechanism = **P0 EMERGENCY**

### Decision Tree

```
Q1: When was the document deleted?
  → Within last 24h: restore from daily backup
  → >24h ago: restore from weekly backup or manual snapshot

Q2: Which document was deleted?
  → Single document: restore just that doc
  → Multiple documents: may indicate systematic issue, investigate

Q3: Do we have a backup?
  → Check: gcloud firestore export --snapshot-time=$(date -u +%Y-%m-%dT%H:%M:%S)Z
  → Backups live in: gs://hmatologia2-firestore-backups/
```

### Mitigation Steps (30–90 min depending on data size)

**Step 1: Confirm deletion in audit trail (5 min)**

```bash
# Query auditoria collection for recent DELETEs
gcloud firestore query '
  SELECT timestamp, operationType, documentPath, operatorId
  FROM auditoria
  WHERE operationType = "DELETE"
  AND timestamp > DATE_SUB(CURRENT_TIMESTAMP(), INTERVAL 24 HOUR)
  ORDER BY timestamp DESC
' --project=hmatologia2 | head -20
```

Record:

- Timestamp of deletion
- Document path (e.g., `/labs/{labId}/ciq-coagulacao/runs/{runId}`)
- Who deleted it (operatorId)

**Step 2: Check available backups (5 min)**

```bash
# List all Firestore backups
gcloud firestore backups list --project=hmatologia2

# Recent backups should be in:
# gs://hmatologia2-firestore-backups/
# (format: YYYY-MM-DD-HH-MM-SS)
```

**Step 3: Restore from backup (30–60 min depending on size)**

**Option A: Restore single collection (faster, ~10–20 min)**

```bash
# Export the deleted collection from backup snapshot
BACKUP_TIME="2026-05-XX-23-59-00" # Use backup time before deletion
gcloud firestore export gs://hmatologia2-firestore-backups/$BACKUP_TIME \
  --collection-ids=ciq-coagulacao \
  --snapshot-time=$(date -d "2026-05-XX 23:59:00" +%s) \
  --project=hmatologia2

# Then import specific documents
gcloud firestore import gs://hmatologia2-firestore-backups/$BACKUP_TIME \
  --project=hmatologia2
```

**Option B: Full database restore (if multiple collections lost, ~30–60 min)**

```bash

gcloud firestore restore --backup-id=<BACKUP_ID> --project=hmatologia2
```

**Step 4: Verify restoration (5 min)**

```bash
# Query the restored document
gcloud firestore documents get \
  /labs/{labId}/ciq-coagulacao/runs/{runId} \
  --project=hmatologia2
```

Should return the document with all fields intact.

**Step 5: Communicate & investigate (ongoing)**

Slack to RT + CTO:

```

Document: /labs/{labId}/ciq-coagulacao/runs/{runId}
Deleted by: {operatorId} at {timestamp}
Action taken: Restored from 2026-05-XX backup
Status: Document verified restored
RCA: User accidentally ran DELETE query / manual intervention needed
Next: Review access controls + audit trail for context
```

### Post-Incident: Strengthen Access Controls

After data loss incident:

1. **Review who has direct Firestore write access**
   - Should be limited to cloud functions + authenticated users with explicit role check
   - Direct client SDK writes should not include DELETE permission on production collections

2. **Implement soft-delete-only rule**
   - Update `firestore.rules` to reject `deleteDoc` calls
   - All deletes must go through `softDelete*` service function

3. **Increase backup frequency**
   - Daily backups insufficient for critical collections
   - Consider hourly snapshots for `ciq-*` collections

### Communication Script

**To RT:**

**To Auditor:**

Preventive: enhanced access controls, soft-delete enforcement, more frequent backups. Audit trail shows deletion event and restoration."

## SCENARIO 8: Cascading Failure (Multiple Systems Down) — Full v1.3 Rollback

**Triggers for cascading failure:**

- > 2 of the above scenarios occurring simultaneously
- Rules + Functions + Hosting all erroring (P0 across the board)
- > 1% of all requests returning 5xx errors
- Unable to isolate root cause within 10 minutes

### Full Rollback Procedure (15–25 min)

**Step 1: Declare incident (1 min)**

Slack:

```
🚨 P0 CASCADING FAILURE — Initiating full v1.3 rollback
All systems: Rules + Functions + Hosting reverting to last known-good state
ETA: 15 min to stable state
```

**Step 2: Rollback all three layers (10–15 min)**

```bash
# 1. Identify last known-good commit
# (should be the v1.3 stable tag or merge commit to main)
GOOD_COMMIT=$(git log --oneline --grep="v1.3" | head -1 | awk '{print $1}')
echo "Rolling back to: $GOOD_COMMIT"

# 2. Rollback firestore rules
git checkout $GOOD_COMMIT -- firestore.rules
firebase deploy --only firestore:rules --project hmatologia2

# 3. Rollback functions
git checkout $GOOD_COMMIT -- functions/
cd functions && npm install && npm run build && cd ..
firebase deploy --only functions --project hmatologia2

# 4. Rollback hosting
git checkout $GOOD_COMMIT -- src/
npm install && npm run build
firebase deploy --only hosting --project hmatologia2
```

**Note:** Each deploy step (rules, functions, hosting) takes 2–5 min. Total time 10–15 min.

**Step 3: Verify stability (5 min)**

```bash
# Tail error logs
gcloud logging read 'severity="ERROR"' \
  --project=hmatologia2 \
  --follow &
TAIL_PID=$!

# Hit the app with a few manual requests
# (hard reload hmatologia2.web.app in browser)
# Wait 5 min for logs to settle

kill $TAIL_PID

# If error rate <0.01%: rollback successful
```

**Step 4: Communicate incident & recovery (2 min)**

Slack:

```
✅ Full v1.3 rollback complete
Status: STABLE — 0 errors in last 5 min tail
All features operational
Root cause investigation starting
Next update: 1h review call with CTO + engineering team
```

**Step 5: Post-incident RCA (within 24h)**

See **Post-Incident Review Template** (below) for full RCA process.
