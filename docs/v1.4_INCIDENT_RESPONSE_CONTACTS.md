## 2. Escalation Tree

### Severity Definitions

| Severity | Impact | Response SLA | Owner | Example |
|----------|--------|--------------|-------|---------|
| **P0** | System down, data corruption, security breach, RDC 978 violation | <5 min | CTO | Portal not loading, NOTIVISA queue stuck, auth bypass |
| **P1** | Feature broken, significant degradation, missing audit trail | <2 hours | Stream Lead + Owner | Laudo draft failing to save, signature validation errors |
| **P2** | Minor bug, cosmetic issue, workaround exists | <24 hours | Assigned Eng | Button alignment, typo in label |
| **P3** | Enhancement, docs, post-launch items | No SLA | Backlog | Future: add dark mode toggle |

### Escalation Flow

```
INCIDENT DETECTED (any user/system)
        │
        ├─ Is system DOWN or DATA CORRUPTED or SECURITY BREACH?
        │  └─ YES → P0 → Notify CTO IMMEDIATELY (phone call)
        │  └─ NO → continue
        │
        ├─ Is feature BROKEN or AUDIT TRAIL MISSING?
        │  └─ YES → P1 → Notify Stream Lead (Slack + phone if no response in 5 min)
        │  └─ NO → continue
        │
        └─ Is it a BUG that doesn't block production?
           └─ YES → P2 → Slack @channel, assign to team
           └─ NO → P3 → Add to backlog
```

### Decision Authority

| Decision | Authority | Notification |
|----------|-----------|--------------|
| **Declare P0 incident** | CTO or Stream Lead (if CTO unreachable) | Slack #incidents, @channel |
| **Approve rollback** | CTO (final) | Slack + all on-call team |
| **Pause new deploys** | CTO or DevOps (if CTO unreachable) | #deployments |
| **Sign-off recovery** | CTO + QA Lead | Document in incident log |
| **Public communication** | CTO (customer-facing) | Email + support portal |

## 6. Incident Severity Assessment Matrix

### P0 Triggers (Declare Immediately)

- [ ] Portal login **not working** for any user
- [ ] NOTIVISA **queue not processing** events (>5 min backlog)
- [ ] Laudo **draft data lost or corrupted** (cannot recover)
- [ ] **Audit trail not recording** signatures or timestamps
- [ ] **Auth token injection** or security bypass detected
- [ ] **Gemini Vision API** completely unavailable (can't submit forms)
- [ ] Firebase **Firestore read/write failing** for critical paths (>30s errors)
- [ ] **Database** connection pooling exhausted (>1min)
- [ ] **Rollback required** (previous deploy broke critical flow)

### P1 Triggers (Escalate to Stream Lead)

- [ ] Form submission **failing for some users** (not global)
- [ ] Signature validation **occasionally failing** but retries work
- [ ] Cloud **Logs showing warnings** for non-critical functions
- [ ] **E2E test flaking** but manual testing passes
- [ ] SMS/email delivery **delayed** (>5 min, but eventually arrives)
- [ ] Portal UI **partially broken** (one screen, workaround exists)
- [ ] Gemini Vision **API timeout** (retry succeeds)

### P2 Triggers (Slack #dev-general)

- [ ] Typo or grammar in UI text
- [ ] Button misalignment or missing spacing
- [ ] Responsive layout issue on mobile (but desktop OK)
- [ ] Console warning (non-fatal)
- [ ] Outdated documentation

## 7. Rollback Procedures

### Go/No-Go Decision (CTO Only)

**CTO asks team:**

1. **What broke?** (root cause clear or unknown?)
2. **Can we fix forward?** (patch in 15 min or no?)
3. **Is rollback safe?** (previous version stable?)

**Decision Tree:**

```
Root cause + fix time <15 min?
  └─ YES → Proceed with forward fix
  └─ NO → Rollback approved
     └─ Previous version tested in prod 24h+ ago?
        └─ YES → Roll back immediately
        └─ NO → Investigate (may roll back earlier version)
```

### Rollback Execution (DevOps)

**Command sequence (in order, with verification):**

```bash
# 1. Revert to previous stable commit
git log --oneline hmatologia2/main | head -5
# Pick the stable commit hash

# 2. Verify rollback intent
git show <hash> --stat
# Check: only legitimate changes, no hidden deps

# 3. Create rollback branch
git checkout -b rollback/<incident-date>-<issue>
git reset --hard <hash>

# 4. Type-check
npm run typecheck

# 5. Test (quick smoke)
npm run test:smoke

# 6. Deploy (rules first, then functions, then hosting)
firebase deploy --only firestore:rules --project hmatologia2
firebase deploy --only functions --project hmatologia2
firebase deploy --only hosting --project hmatologia2

# 7. Notify team (Slack #incidents)
echo "✅ Rollback complete. Testing in progress."
```

**Sign-Off:** CTO verifies "all green" + QA re-runs smoke tests (10 min).

## 9. Critical Functions to Monitor (48h Post-Deploy)

### Firestore Paths

```
Monitor these collections for errors or delays:

✓ /portal-configuracao/{labId}
✓ /notivisa-outbox/{labId}/events
✓ /criticos-escalacoes/{labId}/
✓ /laudos-draft/{labId}/{rtId}
✓ /auditTrail/{labId}/events

Alert thresholds:
  - Write latency > 2 sec (Firestore issue)
  - Read latency > 1 sec (Indexing issue)
  - Rules errors > 5/hour (Security rule violation)
  - Missing documents in audit trail (Data consistency issue)
```

### Cloud Functions

```
Monitor these callables for errors:

✓ rtPortalLogin() → Auth failure rate >5%
✓ listLaudosDraft() → Timeout rate >2%
✓ fetchLaudoForReview() → Signature validation errors
✓ escalateCriticalValue() → SMS/email delivery failures >10%
✓ notivisaQueueProcessor() → Event processing latency >30s

Alert thresholds:
  - Error rate > 1% (any function)
  - P99 latency > 5 sec (timeout risk)
  - Cold start >3 sec (performance regression)
  - Out of quota (contact Google Cloud)
```

### Gemini Vision API

```
Monitor integration for availability:

✓ API response time < 10 sec (model inference)
✓ API error rate < 0.5%
✓ Quota remaining > 50% (check daily)
✓ Vision model accuracy (manual spot-checks)

If issues:
  - Retry with exponential backoff
  - Fall back to Gemini Flash (non-Vision)
  - Contact Google AI support
```

### Network & Auth

```
Monitor user experience:

✓ Portal load time (LCP) < 2 sec
✓ Auth token refresh success > 99%
✓ Logout/re-login cycles working
✓ Session hijacking attempts: 0
✓ CORS errors: 0

If issues:
  - Check Firebase Auth quotas
  - Verify CORS rules in firestore.rules
  - Monitor Cookie SameSite compliance
```
