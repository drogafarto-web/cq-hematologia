## System Components

### 1. Pre-Launch Readiness (`v1.4_PRE_LAUNCH_INCIDENT_READINESS.md`)

**Purpose:** Gate deployment until infrastructure & team are ready.

**Sections:**
- Documentation ready (runbooks, quick cards, monitoring guides)
- Tooling ready (scripts, secret checks, testing)
- Human readiness (on-call engineer, CTO, RT, auditor briefed)
- Infrastructure ready (GCP access, gcloud CLI, dashboards)
- Code ready (no known bugs, rollback tested, secrets provisioned)
- Testing complete (smoke tests passing, load test baseline)
- Compliance & audit trail verified
- Communication & escalation plans prepared
- Deployment day preparation

**Gate:** All items checked before deployment. CTO final sign-off required.

**Timeline:** Complete 72h before v1.4 go-live.

---

### 2. Incident Runbook (`v1.4_POST_LAUNCH_INCIDENT_RUNBOOK.md`)

**Purpose:** Detailed procedures for 8 critical incident scenarios + cascading failure + RCA.

**8 Scenarios Covered:**

| # | Scenario | Detection | P0 Threshold | Primary Action |
|---|----------|-----------|---|---|
| 1 | Rules deny users | "Permission denied" errors >10/15min | Rules block all users | ROLLBACK rules or FIX |
| 2 | Cloud Functions 5xx | httpStatus 500+ >50/10min | Any function >20% error | ROLLBACK function(s) |
| 3 | Firestore quota breach | "rate exceeded" OR quota >95% | N/A (P1) | Disable feature OR request quota ↑ |
| 4 | NOTIVISA API timeout | "deadline exceeded" >5 times | N/A (P1) | Queue for retry (auto-handled) |
| 5 | Gemini Vision API cost | Token usage >10x baseline | N/A (P2) | Reduce threshold OR disable IA |
| 6 | SMS delivery failure | Twilio errors >10/15min | N/A (P1) | Fallback to email |
| 7 | Critical data loss | Manual report: "doc deleted" | Any production delete | RESTORE from backup |
| 8 | Cascading failure | >2 scenarios OR >1% error rate | Multiple systems down | FULL v1.3 ROLLBACK |

**Each scenario includes:**
- Detection signal (what metric/log indicates problem)
- Decision tree (is this P0? call CTO? rollback?)
- Mitigation steps (5–15 min procedures, with CLI commands)
- Communication script (what to tell RT, auditor, CTO)
- If mitigation fails, escalation path

**Additional Sections:**
- Post-incident review template (RCA within 24h)
- Quick reference commands (copy-paste ready)
- Escalation contacts & procedures
- Pre-incident checklist (before going live)

**Timeline:** Activate after v1.4 deploy, use for 48h monitoring window.

### 3. Quick Reference Card (`v1.4_INCIDENT_QUICK_CARD.txt`)

**Purpose:** Single-page printable reference for desk/monitor during 48h window.

**Printable?** Yes — designed to be printed on 1–2 pages and taped to monitor.

**Contains:**
- Key contacts (CTO, on-call engineer, RT, auditor)
- Severity definitions (P0/P1/P2)
- 8 scenarios at a glance (1-line summary + quick action)
- Copy-paste CLI commands for all common operations
- Communication templates (Slack, RT, auditor)
- Post-incident checklist (quick reference version)
- References to full runbook for details

**How to use:** When incident detected, grab card and reference relevant scenario within 10 sec. Full procedures in runbook.

**Print:** 2–3 copies (one per monitor + backup).

**Purpose:** Detailed guide to Cloud Logs setup, filtering, and baseline metrics.

**Sections:**

- What to monitor (Firestore, Cloud Functions, Hosting)
- Common issues & troubleshooting

**How to use:** Reference before 48h window to understand Cloud Logs structure. Use during incident for detailed filter queries.

### 5. Monitoring Scripts (`scripts/monitor-cloud-logs.*`)

**Purpose:** Automated 48h monitoring with periodic reports.

**Bash version:** `scripts/monitor-cloud-logs.sh`
```bash
bash scripts/monitor-cloud-logs.sh 48 10
# Monitors for 48h, samples every 10 min, auto-generates report at end
```

**PowerShell version:** `scripts/monitor-cloud-logs.ps1`
```powershell
.\scripts\monitor-cloud-logs.ps1 -Hours 48 -IntervalMinutes 10
# Same behavior on Windows
```

**Output:** Auto-generates `docs/MONITORING_REPORT_<timestamp>.md` with:
- Error summaries by service
- Latency metrics
- Function invocation counts
- Quota usage
- Recommendations for post-48h SRE monitoring

**How to use:** Start immediately after v1.4 deploy. Runs in background; alerts on each sampling interval. Don't wait for it to finish — parallel with other post-deploy tasks.

## Deployment Phases

### Phase: Deploy (T-0)

**Owner:** Deployer  
**Duration:** 10–15 min total  
**Steps:**

1. **Type-check:** `npx tsc --noEmit` (verify no TS errors)
2. **Build:** `npm run build` (create optimized bundle)
3. **Preflight:** `bash scripts/preflight-secrets-check.sh` (verify secrets)
4. **Deploy Rules:** `firebase deploy --only firestore:rules --project hmatologia2`
5. **Deploy Functions:** `firebase deploy --only functions --project hmatologia2`
6. **Deploy Hosting:** `firebase deploy --only hosting --project hmatologia2`

**Gate:** Each step must succeed. If any fails, stop and debug. Do not proceed to next step with errors.

**Reference:** [deploy-protocol.md](.claude/rules/deploy-protocol.md)

---

### Phase: Immediate Post-Deploy (T+0 to T+5min)

**Owner:** On-call engineer  
**Duration:** 5 min  
**Actions:**

- [ ] Hard reload app in browser (`Ctrl+Shift+R`) → verify page loads
- [ ] Check Firebase Console for any immediate function errors
- [ ] Tail Cloud Logs: `gcloud logging read 'severity >= ERROR' --follow`
- [ ] No errors? Proceed to monitoring phase
- [ ] Any error? Reference runbook Scenario 1–8, declare incident

---

### Phase: Intensive Monitoring (T+5min to T+48h)

**Owner:** On-call engineer  
**Duration:** 48h continuous or spot-check every 15–30 min  
**Cadence:**

| Window | Cadence |
|--------|---------|
| T+0 to T+6h | Real-time (don't leave desk) |
| T+6h to T+24h | Spot-check every 30 min |
| T+24h to T+48h | Spot-check every 1h |

**Actions (per interval):**

```bash
# Run each sampling interval
gcloud logging read 'severity >= ERROR' \
  --project=hmatologia2 \
  --limit=50 \
  --filter='timestamp >= now - <INTERVAL>'

# Count errors
... | wc -l

# If >5 errors in interval: investigate
# If >10 errors in interval: declare P1
# If cascading (multiple services): declare P0 EMERGENCY
```

**Automated alternative:** Let script run in background

```bash
bash scripts/monitor-cloud-logs.sh 48 10 &
# Runs for 48h, samples every 10 min, generates report at end
```

**Critical decisions:**
- Error rate >1% sustained >10 min → P0 (call CTO)
- Error rate 0.1–1% → P1 (investigate, escalate if unresolved >15 min)
- Error rate <0.1% → P2 (log, monitor)
- Any data loss reported → P0 EMERGENCY (restore backup immediately)

**Reference:** Runbook Scenario 1–8 (depending on error pattern)

---

### Phase: Incident Response (When Triggered)

**Owner:** On-call engineer + CTO  
**Duration:** 10–60 min depending on scenario  
**Flow:**

1. **Declare:** Post in Slack `#ops-incidents` with scenario name + impact
2. **Assess:** Determine if P0 (rollback) or P1 (mitigation)
3. **Mitigate:** Execute steps from runbook scenario
4. **Verify:** Tail logs for 5 min — confirm error rate recovered
5. **Communicate:** Update RT + CTO every 5 min
6. **Resolve:** Document resolution + time-to-recovery
7. **RCA:** Start RCA document (complete within 24h)

**Reference:** Runbook + quick card

---

### Phase: Post-48h Sign-Off (T+48h)

**Owner:** CTO + on-call engineer  
**Duration:** 1h review  
**Actions:**

- [ ] Review monitoring report (auto-generated)
- [ ] Tally incidents: ____ P0, ____ P1, ____ P2
- [ ] If >5 incidents (P0+P1): extend monitoring to T+72h
- [ ] If <5 incidents (P0+P1): proceed to Phase 2 planning
- [ ] Archive monitoring artifacts: `docs/v1.4-MONITORING-REPORT-<date>.md`
- [ ] CTO sign-off: v1.4 stable, transition to normal SRE monitoring

## Escalation Paths

### Real-time Escalation (During Incident)

**Severity P0 (Emergency):**
- [ ] Declare in Slack `#ops-incidents`
- [ ] Call CTO (phone): ____
- [ ] Page on-call engineer (if not already active)
- [ ] CTO makes decision: fix vs rollback
- [ ] Execute from runbook
- [ ] Update status every 5 min until resolved

**Severity P1 (Urgent):**
- [ ] Post in Slack `#ops-incidents`
- [ ] Slack CTO (no phone call unless unresolved >15 min)
- [ ] Start mitigation from runbook
- [ ] Escalate to call if error rate not improving

**Severity P2 (Normal):**
- [ ] Log in Slack `#ops-incidents`
- [ ] Execute fix within 1h
- [ ] RCA within 24h

### Post-Incident Escalation (Compliance/Audit)

**If incident impacts RDC 978 compliance:**
- Notify auditor within 1h of resolution
- Include in RCA: "Impact on RDC 978 Art. X: [description]"
- Provide audit trail from auditoria collection + Cloud Logs

**If data was lost:**
- Notify auditor immediately
- Include in RCA: restore action + any data loss quantification
- Update compliance baseline

**If repeating pattern detected:**
- Document in risk register (`.planning/v1.4-RISK-REGISTER.md`)
- Escalate to architecture review (may require multi-phase fix)

## Success Criteria

### 48h Window Success

- [ ] <5 incidents in total (P0+P1+P2 combined)
- [ ] All P0 incidents resolved <15 min
- [ ] All P1 incidents resolved <1h
- [ ] Error rate <0.1% for >80% of window
- [ ] No data loss
- [ ] All RCAs completed within 24h of incident

### Post-48h Sign-Off

- [ ] Monitoring report generated automatically
- [ ] <5 P0/P1 incidents detected
- [ ] CTO approves transition to normal SRE monitoring
- [ ] Runbook & quick card updated if improvements needed
- [ ] Team debriefs on lessons learned (optional but recommended)

## Quick Links for Common Tasks

### I need to...

**...start monitoring**
```bash
bash scripts/monitor-cloud-logs.sh 48 10
# or Windows: .\scripts\monitor-cloud-logs.ps1 -Hours 48 -IntervalMinutes 10
```

**...check current error rate**
```bash
gcloud logging read 'severity >= ERROR' --project=hmatologia2 --follow
```

**...identify which function is failing**
```bash
gcloud logging read 'resource.type="cloud_function" AND severity="ERROR"' \
  --project=hmatologia2 --limit=50 --format='table(labels.function_name,jsonPayload.message)'
```

**...rollback a single function**
```bash
git checkout HEAD~1 -- functions/src/<module>/<function>.ts
cd functions && npm run build && cd ..
firebase deploy --only functions:<functionName> --project=hmatologia2
```

**...rollback all code (full v1.3 restore)**
```bash
COMMIT=$(git log --oneline --grep="v1.3" | head -1 | awk '{print $1}')
git checkout $COMMIT -- firestore.rules functions/ src/
firebase deploy --only firestore:rules,functions,hosting --project=hmatologia2
```

**...verify secrets**
```bash
bash scripts/preflight-secrets-check.sh
```

**...restore a deleted document from backup**
```bash
# 1. Identify backup time
gcloud firestore backups list --project=hmatologia2

# 2. Restore (CTO only)
gcloud firestore restore --backup-id=<ID> --project=hmatologia2
```
