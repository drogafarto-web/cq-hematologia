## Executive Summary

This document defines a **complete test data provisioning framework** for v1.4 execution (Phases 4–15, 14 weeks). Each phase receives:
- ✅ **Seed data** (analytes, thresholds, procedures, documents, regulatory records)
- ✅ **Mock users** (RT, physicians, operators, patients for portal testing)
- ✅ **Sample collections** (laudos, runs, feedback, CAPA records, audit events)
- ✅ **Staging isolation** (separate `labId` from production, no cross-contamination)
- ✅ **Reset procedures** (between test cycles, keep data fresh and isolated)

**Key principle:** All test data lives in **fixtures** (JSON files in `test/fixtures/<phase>/`) that teams can reset between runs. Each phase has independent test data sets that do not interfere with production or other phases.

**Success metric:** Every phase kickoff has realistic, ready-to-test data on day 1.

## Architecture Overview

### Data Isolation Strategy

```
Production (labId: "lab-001-prod")
  └─ Real operators, real patients, real data
  └─ No test data writes
  └─ Separate Firestore database (manual toggle)

Staging (labId: "lab-001-staging", "lab-002-staging", etc.)
  └─ Test data isolated per phase
  ├─ Phase 4: portal-test, notivisa-test labIds
  ├─ Phase 5: criticos-test, ia-test labIds
  ├─ Phase 6: capa-test, auditoria-test labIds
  └─ Phases 7–15: modular isolation per feature
  └─ Auto-reset between cycles (nightly or on-demand)

Local Emulator (for unit/integration tests)
  └─ Complete fixture sets replicated
  └─ No external API calls (mocked)
  └─ Fast feedback loop
```

### Multi-Tenant Data Layout

All test data follows the **multi-tenant convention**:

```
/labs/{labId}/
  ├─ analitos/
  ├─ criticos-escalacoes/
  ├─ laudos/
  ├─ runs/
  ├─ members/
  ├─ turnos/
  ├─ risks/
  ├─ lab-apoio/
  └─ [other collections]

/users/{uid}/
  ├─ profile/
  ├─ permissions/
  └─ audit-trail/

/shared/
  ├─ labs/
  └─ configs/
```

```javascript
import fs from "fs";
import path from "path";

export class FixtureLoader {
  static async load(fixturePath) {
    const filePath = path.join(
      process.cwd(),
      "test/fixtures",
      `${fixturePath}.json`
    );
    return JSON.parse(fs.readFileSync(filePath, "utf-8"));
  }

  static async loadBinary(fixturePath) {
    const filePath = path.join(
      process.cwd(),
      "test/fixtures",
      fixturePath
    );
    return fs.readFileSync(filePath);
  }

  static async save(fixturePath, data) {
    const filePath = path.join(
      process.cwd(),
      "test/fixtures",
      `${fixturePath}.json`
    );
    fs.mkdirSync(path.dirname(filePath), { recursive: true });
    fs.writeFileSync(filePath, JSON.stringify(data, null, 2));
  }
}

// Usage in tests
```

```javascript
export async function validateFixtures(phase) {
  const results = {
    phase,
    timestamp: new Date().toISOString(),
    checks: {},
    status: "PASSED"
  };

  // Check 1: All required files present
  results.checks.filesExist = await checkRequiredFiles(phase);

  // Check 2: JSON schema validation
  results.checks.jsonSchema = await validateJSONSchema(phase);

  // Check 3: Multi-tenant isolation (labId consistency)
  results.checks.labIdConsistency = await checkLabIdConsistency(phase);

  // Check 4: Referential integrity (FKs valid)
  results.checks.referentialIntegrity = await checkReferences(phase);

  // Check 5: Test data completeness (min 80% coverage)
  results.checks.completeness = await checkCompleteness(phase);

  // Check 6: No production data leakage
  results.checks.noProductionData = await checkForProdLabIds(phase);

  // Fail fast if any check fails
  Object.values(results.checks).forEach(check => {
    if (!check.passed) results.status = "FAILED";
  });

  return results;
}
```
