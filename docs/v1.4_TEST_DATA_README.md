## The Problem Solved

**Before v1.4 Test Data Strategy:**
- Each phase started with ad-hoc test data (created during execution, no fixtures)
- Tests waited for QA to provision data (delay of days)
- No isolation between phases (contamination risk)
- No reset procedures (stale data, manual cleanup)
- Teams didn't know what test users existed (confusion, wasted time)

**After v1.4 Test Data Strategy:**
- ✅ Day 1 of phase kickoff: All test data ready in fixtures
- ✅ Complete isolation: Each phase has separate `labId` (lab-001-staging, lab-002-staging, etc.)
- ✅ Fast reset: `node test/utils/reset-staging.mjs --phase X` (30s)
- ✅ Known users: All mock users documented, credentials in `.env.test`
- ✅ Validation: Automated checks (JSON schema, referential integrity, data completeness)
- ✅ No delays: Zero test execution time lost to "data not ready"

## Key Features

### Data Isolation

- ✅ Separate `labId` per phase (no cross-contamination)
- ✅ Staging data never touches production
- ✅ Automated checks prevent production `labId` in test fixtures
- ✅ Multi-tenant rules enforced at Firestore Rules level

### Fast Reset

```bash
# Full reset in ~30 seconds
node test/utils/reset-staging.mjs --phase 4 --firebase-project hmatologia2-staging
```

### Validation

```bash
# Pre-deploy checklist
npm run test:validate-fixtures -- --all
# Checks:
# ✅ All required files present
# ✅ JSON schema valid
# ✅ labId consistency (no production leakage)
# ✅ Referential integrity (FKs valid)
# ✅ Data completeness (80%+ coverage per phase)
```

### Automated Reset

- Nightly 02:00 UTC auto-reset (delete + reload + validate)
- Backup before reset (keep 7 days)
- Logs and monitoring via Cloud Logging

## FAQ

**Q: Where do I find mock user passwords?**  
A: In `.env.test` (created before Phase 4, never committed). Use `grep TEST_PASSWORD .env.test`

**Q: What if I need to reset data mid-phase?**  
A: `node test/utils/reset-staging.mjs --phase 4` (30s, full reset)

**Q: Where are the 100 test laudos for Portal?**  
A: `test/fixtures/phase-4/laudos.json` (created May 15)

**Q: Can I use this test data in production by accident?**  
A: No. All fixtures use separate `labId` (lab-001-staging, not lab-001-prod). Rules enforce isolation.

**Q: What if validation fails?**  
A: Run detailed check: `npm run test:validate-fixtures -- --phase X --verbose`

**Q: How do I add more test data during a phase?**  
A: Edit fixture JSON file, save, reload via `node test/utils/load-fixtures.mjs --phase X`
