# HC Quality — v1.4 NOTIVISA Sandbox Setup & Government API Onboarding

## Executive Summary

This document describes the **complete onboarding process** for integrating HC Quality with NOTIVISA (Notificação Imediata de Vítima de Violência / ANVISA's adverse event notification system). Phase 4 Task 04-03 implements sandbox integration only; production submission deferred to v1.5 (pending certificate provisioning).

**Timeline expectation:** Government sandbox provisioning takes **3–5 business days** after registration. **Start immediately (by 2026-05-10)** to avoid Phase 4 critical path delays.

**Success criteria:**
- [ ] HC Quality lab registered with NOTIVISA (CNPJ, RT credentials, contact submitted)
- [ ] Sandbox API credentials provisioned (key + endpoint URL confirmed)
- [ ] Portaria 204/2016 Art. 6º schema validated (all 15 mandatory fields mapped)
- [ ] Test payload generation working (draft forms generated for syphilis, dengue, HIV)
- [ ] Test submission workflow passing (draft → RT review → seal → sandbox API call)
- [ ] Error handling verified (validation errors, rate limits, retry windows)
- [ ] Audit trail logs complete (every submission + response immutable in Firestore)
- [ ] Ops runbook drafted (alerts, manual retries, escalation path)

## Part 1: Government Registration & Sandbox Credentials (3–5 days)

### Step 1.1: Prepare Laboratory Registration Documents

NOTIVISA registration requires legal/operational setup. Coordinate with lab director + RT (Responsável Técnico).

**Required documents:**

| Document | Source | Use | Notes |
|----------|--------|-----|-------|
| **CNPJ** | Lab incorporation papers | Lab identifier | Must be active (status "ativa" with Receita Federal) |
| **RT CPF + RG** | Personnel records | Responsável Técnico signature authority | RT must be registered with ANVISA |
| **RT Crédenciais ANVISA** | ANVISA portal (anvisa.gov.br) | API authentication | RT account activation required first |
| **Health Authority Contact** | Regional health dept | Emergency escalation contact | Secretária de Saúde (municipal or state) phone + email |
| **Lab Address** | Corporate records | Registration address | Full street, city, state, ZIP, phone |

**Checklist before contacting ANVISA:**
- [ ] CNPJ status is "ativa" (verify via https://www.cnpj.net/)
- [ ] RT has active ANVISA account (login test: https://portalanvisa.gov.br/)
- [ ] RT CPF/RG scanned + uploaded
- [ ] Health authority (Secretária de Saúde) contact identified
- [ ] Lab director has signed authorization letter (template below)

**Authorization letter template (RT must sign):**

### Step 1.2: Register with NOTIVISA Sandbox Portal

**Portal URL:** https://portalanvisa.gov.br/notivisa (sandbox endpoint suffix may vary; confirm with ANVISA during registration)

**Registration flow:**
1. **Log in** with RT credentials (ANVISA account)
2. **Select "New Laboratory Registration"** → fill mandatory fields:
   - CNPJ
   - Lab name
   - Contact email (for sandbox updates)
   - Phone (emergency escalation)
   - Secretária de Saúde contact (regional health authority)
3. **Attach authorization letter** (signed by RT, PDF)
4. **Select integration type:** "API Integration (Sandbox)"
5. **Confirm RT signature** (electronic signature via ICP-Brasil cert, or PIN-based if available)
6. **Submit** → ANVISA returns `registration-id` (ticket number for tracking)

**Expected response time:** 3–5 business days. ANVISA will:
- Verify CNPJ status with Receita Federal
- Validate RT credentials
- Issue **API key** (alphanumeric, ~32 chars)
- Issue **sandbox endpoint URL** (typically `https://sandbox.notivisa.gov.br/api/v1/` or similar)
- Email both to the contact email provided

**Record these details in a secure location immediately:**
- API Key: `NOTIVISA_SANDBOX_API_KEY` → Firebase Secrets Manager
- Sandbox URL: `NOTIVISA_SANDBOX_ENDPOINT` → Firebase environment config or Secrets Manager
- Registration ID: `NOTIVISA_REGISTRATION_ID` → Secrets Manager (for manual lookup/debugging)

```bash
  exit 1
fi

echo "[✓] All connectivity checks passed. Ready for schema validation."
```

## Part 2: Portaria 204/2016 Art. 6º Schema Validation

### Step 2.1: Review Mandatory Fields

Portaria 204/2016 defines 15 mandatory fields for NOTIVISA notifications. All fields must be present in the submission; missing any field causes API rejection (400 Bad Request).

**Mandatory fields (Art. 6º §1):**

| # | Field | Type | Format | Example | Validation |
|---|-------|------|--------|---------|------------|
| 1 | **versao** | string | fixed | `"1.0"` | Must be `"1.0"` |
| 2 | **doenca_notificavel** | string | MS code (5-6 digits) | `"99078"` (syphilis) | Must exist in MS Portaria 204 list (99 diseases) |
| 3 | **tipo_notificacao** | enum | `"CONFIRMADA"` \| `"SUSPEITA"` | `"CONFIRMADA"` | Case-sensitive uppercase |
| 4 | **paciente_cpf** | string | 11 digits, masked | `"123.456.789-**"` | No hyphens in transmitted form; PII masked (last 2 digits only in logs) |
| 5 | **paciente_data_nascimento** | timestamp | ISO 8601 | `"1980-05-15"` | YYYY-MM-DD format |
| 6 | **paciente_sexo** | enum | `"M"` \| `"F"` \| `"O"` | `"F"` | Case-sensitive single char (M=male, F=female, O=other) |
| 7 | **resultado_teste** | string | freetext | `"Reagente"` | Lab-specific result (positive, negative, inconclusive) |
| 8 | **data_resultado** | timestamp | ISO 8601 | `"2026-05-07T14:30:00Z"` | UTC timestamp with timezone |
| 9 | **laboratorio_cnpj** | string | 14 digits | `"12345678000195"` | Your lab CNPJ (no hyphens) |
| 10 | **operador_cpf** | string | 11 digits, masked | `"987.654.321-**"` | RT or authorized operator who signed |
| 11 | **operador_assinatura_hash** | string | SHA-256 hex | `"e3b0c44298fc1c1..."` | 64-char hex hash of signing payload |
| 12 | **operador_assinatura_ts** | timestamp | Unix epoch (ms) | `1714077000000` | Milliseconds since 1970-01-01 UTC |
| 13 | **auditoria_trail_url** | string | HTTPS URL | `"https://hmatologia2.web.app/audit/123abc"` | Public-facing audit log URL (optional, v1.5) |
| 14 | **observacoes** | string | freetext, max 500 chars | `"Result from Yumizen H550"` | Lab notes (optional) |
| 15 | **timestamp_criacao** | timestamp | ISO 8601 | `"2026-05-07T14:35:22Z"` | When notification created (server-generated) |

**Validation rules:**

```typescript
// Zod schema (functions/src/shared/notivisa.ts)
export const notivisaDraftSchema = z.object({

  doenca_notificavel: z.string().regex(/^\d{5,6}$/, 'MS disease code 5-6 digits'),

  paciente_data_nascimento: z.string().regex(/^\d{4}-\d{2}-\d{2}$/, 'YYYY-MM-DD'),

  data_resultado: z.string().datetime(),

  operador_cpf: z.string().regex(/^\d{3}\.\d{3}\.\d{3}-\*\*$/, 'Masked CPF format'),
  operador_assinatura_hash: z.string().regex(/^[a-f0-9]{64}$/, 'SHA-256 hex'),
  operador_assinatura_ts: z.number().int().positive(),
  auditoria_trail_url: z.string().url().optional(),
```

### Step 2.2: Map HC Quality Laudo → NOTIVISA Payload

Create a transformation layer (`functions/src/modules/notivisa/notivisaPayloadBuilder.ts`) that maps laudo data to NOTIVISA schema.

**Mapping reference:**

| HC Quality Field | NOTIVISA Field | Transformation |
|-----------------|----------------|-----------------|
| `laudo.resultadoEm` (Firestore timestamp) | `data_resultado` | Convert to ISO 8601 UTC string |
| `laudo.paciente.cpf` | `paciente_cpf` | Mask all but last 2 digits: `"123.456.789-**"` |
| `laudo.paciente.dataNascimento` | `paciente_data_nascimento` | Format as `YYYY-MM-DD` |
| `laudo.paciente.sexo` (enum: 'M' \| 'F' \| 'O') | `paciente_sexo` | Pass through as-is |
| `laudo.resultados[].valor + .status` | `resultado_teste` | Concatenate: `"${analito}: ${valor} ${status}"` |
| Lab config `cnpj` | `laboratorio_cnpj` | Lookup from lab settings doc |
| `laudo.assinatura.operatorId` (UID) | `operador_cpf` | Fetch operator CPF from admin store + mask |
| `laudo.assinatura.hash` | `operador_assinatura_hash` | Use HMAC hash from signature |
| `laudo.assinatura.ts` | `operador_assinatura_ts` | Millisecond timestamp as-is |
| Static value | `versao` | `"1.0"` |
| Disease list lookup | `doenca_notificavel` | Map laudo analito → MS Portaria 204 code |

**Example transformation:**

```typescript
export async function buildNotivisaPayload(
  laudo: LaudoDoc,
  labId: LabId,
  diseaseMapping: DiseaseMap,
): Promise<NotivisaDraft> {
  const lab = await getLabSettings(labId);
  const operator = await getOperator(laudo.assinatura.operatorId);
  
  // Determine disease code from laudo results
  const diseaseCode = determineNotivisaDiseaseCode(laudo.resultados, diseaseMapping);
  if (!diseaseCode) throw new Error('No notifiable disease detected in laudo');
  
  // Mask CPF (preserve anonymity in draft)
  const maskCPF = (cpf: string) => cpf.slice(0, 9) + '**';
  
  return {
    versao: '1.0',
    doenca_notificavel: diseaseCode,
    tipo_notificacao: 'CONFIRMADA', // v1.4: always confirmed (sandbox only)
    paciente_cpf: maskCPF(laudo.paciente.cpf),
    paciente_data_nascimento: formatDate(laudo.paciente.dataNascimento), // YYYY-MM-DD
    paciente_sexo: laudo.paciente.sexo,
    resultado_teste: formatResult(laudo.resultados),
    data_resultado: new Date(laudo.resultadoEm).toISOString(),
    laboratorio_cnpj: lab.cnpj,
    operador_cpf: maskCPF(operator.cpf),
    operador_assinatura_hash: laudo.assinatura.hash,
    operador_assinatura_ts: laudo.assinatura.ts,
    timestamp_criacao: new Date().toISOString(),
    observacoes: `Laudo ID: ${laudo.id} (sandbox draft, not yet submitted)`,
  };
}
```

### Step 3.2: Test Draft → RT Review → Seal Workflow

HC Quality workflow for sandbox:
1. **Draft generation** (automatic on notifiable result)
2. **RT review** (RT sees draft in UI, approves or rejects)
3. **Sealing** (HMAC signature applied, immutable)
4. **Sandbox submission** (async via queue processor, v1.4 sends to sandbox; v1.5 will send to prod)

**Firestore schema for notivisa-outbox drafts:**

```typescript
/labs/{labId}/notivisa-outbox/{docId}
├── status: 'draft' | 'rt_reviewing' | 'sealed' | 'submitted' | 'acknowledged' | 'rejected'
├── laudo_id: string
├── disease_code: string (MS Portaria 204 code)
├── paciente_cpf_masked: string (e.g., '123.456.789-**')
├── payload: {
│   // Full Portaria 204 Art. 6º §1 object (see Part 2)
│   versao: string
│   doenca_notificavel: string
│   tipo_notificacao: string
│   // ... 12 more fields
│ }
├── submission_attempts: [{
│   timestamp: Timestamp
│   status: 'pending' | 'sent' | 'error'
│   error_code?: string (if applicable)
│   error_message?: string
│   response_status_code?: number
│   anvisa_response?: object
│ }]
├── rt_review: {
│   reviewed_by: UID
│   reviewed_at: Timestamp
│   decision: 'approved' | 'rejected'
│   reason?: string
│ }
├── sealed_by: UID
├── sealed_at: Timestamp
├── seal_hash: string (HMAC over payload)
├── created_at: Timestamp
├── deleted_at: Timestamp | null
```

## Part 4: Error Handling & Retry Strategy

### Step 4.1: Validation Error Handling

NOTIVISA API rejects payloads with **400 Bad Request** if any mandatory field is malformed.

**Common validation errors:**

| Error Code | Field | Issue | Fix in v1.4 |
|------------|-------|-------|-----------|
| `MISSING_FIELD` | Any | Null or undefined value | Check all 15 fields before submit |
| `INVALID_FORMAT` | `doenca_notificavel` | Non-existent MS disease code | Validate against seed list before submit |
| `INVALID_FORMAT` | `paciente_cpf` | Wrong masking (not `XXX.XXX.XXX-**`) | Use `maskCPF()` utility |
| `INVALID_FORMAT` | `data_resultado` | Not ISO 8601 UTC | Validate with `z.string().datetime()` |
| `INVALID_ENUM` | `paciente_sexo` | Not `'M'` \| `'F'` \| `'O'` | Normalize in payload builder |
| `LENGTH_EXCEEDED` | `observacoes` | >500 chars | Truncate in payload builder |

**Validation error handler:**

```typescript
export async function handleNotivisaValidationError(
  error: AxiosError<{ error_code: string; error_message: string }>,
  draftId: string,
  labId: LabId,
): Promise<void> {
  const errorCode = error.response?.data?.error_code || 'UNKNOWN';
  const errorMessage = error.response?.data?.error_message || error.message;

  // Log validation error (permanent, requires manual fix)
  await logAuditEvent('notivisa_validation_failed', {
    draft_id: draftId,
    error_code: errorCode,
    error_message: errorMessage,
    operator_uid: request.auth.uid,
  });

  // Update draft status to indicate manual intervention needed
  const draftRef = doc(db, `labs/${labId}/notivisa-outbox/${draftId}`);
  await updateDoc(draftRef, {
    status: 'validation_error',
    validation_error_code: errorCode,
    validation_error_message: errorMessage,
    requires_manual_intervention: true,
  });

  // Alert ops (email + dashboard flag)
  await notifyOpsTeam('NOTIVISA validation error', {
    draft_id: draftId,
    lab_id: labId,
    error: errorMessage,
    action: 'Manual review required in HC Quality UI',
  });
}
```

### Step 4.3: Transient Error Retry Strategy

**Transient errors (5xx, network timeouts):** Use exponential backoff.

```typescript
const RETRY_BACKOFF_MS = [
  60 * 1000,           // 1 min (attempt 1)
  5 * 60 * 1000,       // 5 min (attempt 2)
  30 * 60 * 1000,      // 30 min (attempt 3)
  2 * 60 * 60 * 1000,  // 2 hours (attempt 4)
  24 * 60 * 60 * 1000, // 24 hours (attempt 5)
];

export async function submitNotivisaWithRetry(
  payload: NotivisaPayload,
  options: { maxAttempts?: number } = {},
): Promise<SubmissionResult> {
  const maxAttempts = options.maxAttempts || 5;
  let lastError: Error | null = null;

  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      const response = await submitNotivisaToSandbox(payload);
      return { status: 'success', response, attempts: attempt };
    } catch (error) {
      lastError = error;
      const isTransient =
        error.response?.status >= 500 ||
        error.code === 'ECONNREFUSED' ||
        error.code === 'ENOTFOUND';

      if (!isTransient || attempt === maxAttempts) {
        throw error;
      }

      // Wait before retry
      const backoffMs = RETRY_BACKOFF_MS[attempt - 1];
      console.log(`[NOTIVISA] Attempt ${attempt} failed. Retrying in ${backoffMs}ms...`);
      await sleep(backoffMs);
    }
  }

  throw lastError || new Error('Retry exhausted');
}
```

```typescript
// v1.5: Real production submission
export async function submitNotivisaToProduction(
  payload: NotivisaPayload,
): Promise<SubmissionResult> {
  const cert = await secretsManager.getSecret('NOTIVISA_CERTIFICATE_PFX');
  const password = await secretsManager.getSecret('NOTIVISA_CERTIFICATE_PASSWORD');

  // Create HTTPS agent with mTLS client certificate
  const httpsAgent = new https.Agent({
    pfx: Buffer.from(cert, 'base64'),
    passphrase: password,
    rejectUnauthorized: true, // Validate ANVISA server cert

    'https://notivisa.saude.gov.br/api/v1/submit',
    payload,
    {
      httpsAgent,
      headers: { 'Content-Type': 'application/json' },
      timeout: 30000,
    }
  );

  return {
    status: 'success',
    transaction_id: response.data.transaction_id,
```

**Cause:** Sandbox API has request limits (typically 100 req/hour or 1000 req/day)

1. Wait for `Retry-After` header (default 3600s / 1 hour)
2. Client automatically retries with exponential backoff
3. Check submission cadence in logs
4. Contact ANVISA for rate limit policy

**Monitor rate limit status:**
```bash
node scripts/notivisa-sandbox-inspect.mjs --lab lab-001 --status rate_limited
```

#### Timeout Errors

**Problem:** `Request timeout` or `ECONNREFUSED`

**Cause:** Network latency or ANVISA sandbox under maintenance

**Solution:**
1. Automatic retry triggered (up to 5 attempts)
2. Backoff: 1, 2, 4, 8, 16 seconds
3. Check ANVISA maintenance window (typically Saturdays 22:00–04:00 UTC)
4. Verify firewall allows outbound HTTPS to `*.notivisa.gov.br`

#### Validation Errors (400 Bad Request)

**Problem:** `INVALID_FORMAT` or `MISSING_FIELD` on submission

**Debug:**

2. Inspect last submission attempt:
   ```bash

   ```

3. Review error message in `validation_error_message` field

4. **Manual fix:** RT must review + re-seal draft in UI
