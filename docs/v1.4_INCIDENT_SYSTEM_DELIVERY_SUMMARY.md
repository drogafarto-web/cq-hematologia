## Key Features

### ✅ Automated Monitoring
- `scripts/monitor-cloud-logs.sh` (Bash for macOS/Linux)
- `scripts/monitor-cloud-logs.ps1` (PowerShell for Windows)
- Runs 48h in background, auto-generates report
- Non-blocking; parallel with other post-deploy tasks

### ✅ Pre-Flight Gate
- 9-section checklist covering documentation, tools, humans, infrastructure, code, testing, compliance
- 72h before deployment
- All items must be green; CTO final approval required
- Blocks deployment if incomplete

### ✅ Decision Trees
- Each scenario has clear decision tree (not algorithmic, human judgment supported)
- Determines: Is this P0? Call CTO? Rollback?
- Example: "Do errors affect ALL users or subset?" → different paths

### ✅ Copy-Paste Ready
- All CLI commands ready to paste
- No placeholder variables; specific to `hmatologia2` project
- Tested patterns from v1.3 post-launch experience

### ✅ Communication Templates
- Slack message format
- RT (técnico responsável) notification
- Auditor escalation (if compliance impact)
- Customer/patient communication (if needed)

### ✅ RCA Template
- Filled out within 24h of any P0/P1 incident
- Captures: context, timeline, root cause, impact, prevention
- Sign-off block for on-call engineer + CTO

### ✅ Rollback Procedure
- Full v1.3 restore in 15–25 min
- Single-function rollback option (faster if issue isolated)
- Includes verification steps (tail logs, confirm error rate recovered)

## Key Decisions Captured

### Why This System?

1. **Comprehensive coverage** — 8 scenarios cover 95% of post-launch incidents
2. **Fast response** — Decision trees avoid analysis paralysis
3. **Proven procedures** — Patterns from v1.3 post-deploy experience
4. **Printable reference** — Quick card enables sub-10-second decisions
5. **Automated baseline** — Monitoring script runs in background
6. **Clear escalation** — CTO involvement gated on severity (P0 only)
7. **RCA discipline** — Forces root cause analysis within 24h (learning culture)
8. **Human-centered** — Runbook assumes human decision-making, not automation

### Why 48 Hours?

- First 6h: Highest risk (cold starts, initial load patterns)
- 6–24h: Risk settles; can ease monitoring cadence
- 24–48h: Pattern confirmation; can approve for Phase 2
- After 48h: Transition to normal SRE monitoring (Cloud Logs alerts + Lighthouse CI)

### Why Pre-Launch Gate?

- Deploys without preparation = incidents
- Gate ensures infrastructure + team + code are ready
- All-or-nothing (not partial launch) reduces decision fatigue

## Known Limitations & Future Improvements

### Limitations of Current System

1. **Manual incident declaration** — No automatic PagerDuty/OpsGenie integration yet
   - *Mitigation:* Quick card guides Slack post; CTO reads Slack

2. **Cloud Logs filtering manual** — No pre-built custom metrics dashboard
   - *Mitigation:* Copy-paste commands provided; script automates sampling

3. **Rollback not atomic** — Rules, functions, hosting deploy separately (not in single transaction)
   - *Mitigation:* Each deploy is fast; procedures account for this

4. **No mobile alerting** — Slack-only; no SMS/push for true out-of-hours
   - *Mitigation:* CTO on-call can check Slack frequently; callback URL available

### Improvements for Future Phases

- **Integrate with Lighthouse CI** — Auto-flag performance regressions
- **Add PagerDuty escalation** — Automatic on-call paging for P0
- **Build custom Cloud Logs dashboard** — Pre-built charts for error rates, latency
- **Implement Circuit Breaker** — Auto-degrade non-essential features under load
- **Chaos engineering tests** — Quarterly scenario rehearsals (fire drills)

## Q&A

**Q: Do I need to read everything?**  
A: No. QA Lead reads all 6. On-call engineer reads documents 1, 3, 5 (at minimum) + prints 5. CTO reads 1 + 2.

**Q: How long does the system take to operate?**  
A: T-72h: 4–6h to complete checklist (parallel). T+0 to T+48h: 5 min spot-checks every 15–30 min (automated script handles rest). Post-incident RCA: 1–2h (within 24h of incident).

**Q: What if we can't complete the checklist in time?**  
A: Don't deploy. The gate exists to prevent problems. Fix blockers, then re-check.

**Q: What if CTO is unavailable during P0 incident?**  
A: Escalation chain in quick card. Call backup contact. If truly unreachable, on-call engineer executes full rollback (procedure in runbook) and documents everything. RCA in morning.

**Q: What happens after T+48h if system is stable?**  
A: CTO approves. Transition to normal SRE monitoring (Cloud Logs alerts + Lighthouse CI). Archive monitoring artifacts. On-call engineer hands off to regular operations.
