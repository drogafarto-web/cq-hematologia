# v1.3 Deployment — Security Audit Checklist

**Your job:** Verify security controls are live and compliance baseline met. 30 min pre-deploy + 20 min post-deploy.

---

## Pre-Deployment Security Review (10 min)

Execute **before** DevOps deploys anything.

### 1. Code Security Audit
- [ ] **Read:** [SECURITY_SIGN_OFF_v1.3.md](./SECURITY_SIGN_OFF_v1.3.md) (page 1, section 1-4)
- [ ] **Verify in document:**
  - ✅ Soft-delete enforcement: 73 hard-delete blocks present
  - ✅ Callable-only writes: `allow create: if false` on `/laudos`, `/reclamacoes`, `/bioquimica-analitos`
  - ✅ LogicalSignature validation: 64-char hex hash + timestamp + operatorId
  - ✅ Rate limiting on callables: 60/min per uid minimum
  - ✅ LGPD consent enforcement: mandatory on reclamacoes

**All 5 checks present?**
- ✅ YES → Continue
- ❌ NO → BLOCK deployment, file issue

### 2. Compliance Baseline Audit
- [ ] **Read:** [COMPLIANCE_SUMMARY_v1.3.md](./COMPLIANCE_SUMMARY_v1.3.md) (page 1-3)
- [ ] **Verify in document:**
  - ✅ RDC 978 Art. 167 (Laudos): RT signature + chain of custody documented
  - ✅ RDC 978 Art. 179 (CIQ): Bioquímica + 4 other CIQ modules deployed
  - ✅ DICQ 4.3 compliance: 78.5% coverage (target 75% for audit-ready)
  - ✅ DICQ 4.4 audit trail: deletadoEm field + operatorId + timestamps
  - ✅ LGPD integration: consent capture + data anonymization (90d old feedback)

**Compliance % ≥ 75%?**
- ✅ YES (78.5%) → Continue
- ❌ NO → BLOCK deployment, escalate to compliance officer

### 3. Firestore Rules Validation
- [ ] **Check:** No `z.any()` in Zod schemas (unvalidated data)
  - Command: `grep -r "z\.any" firestore.rules`
  - Expected: 0 results
  - If found: BLOCK deployment

- [ ] **Check:** No client-side `deleteDoc` calls
  - Command: `grep -r "deleteDoc" src/`
  - Expected: 0 results (all soft-delete via `softDelete*` service methods)
  - If found: BLOCK deployment

- [ ] **Check:** All auth tokens validated
  - Review: `/functions/src/**/authenticate.ts` or equivalent auth check
  - Expected: `request.auth !== null` validation present on all callable functions
  - If missing: BLOCK deployment

### Status: Pre-Deploy Security = ✅ GREEN
Proceed to DevOps deployment.

## Post-Deployment Security Verification (20 min)

Execute **after** DevOps confirms Steps 1–3 LIVE (Rules + Functions + Hosting).

### 1. Rules Enforcement Tests

#### Test 1.1: Soft-Delete Enforcement
**Goal:** Verify hard-delete is blocked (RN-06)

1. Open **Firebase Console** → **Firestore** → **Console** tab
2. Try to delete a test document:
   ```javascript
   // This should FAIL with "Permission denied"
   db.collection('labs').doc('riopomba')
     .collection('reclamacoes').doc('[test-doc-id]')
     .delete()
   ```
3. Expected: **Error: "Permission denied"**
4. ✅ **PASS** if denied; ❌ **FAIL** if deletion succeeds

#### Test 1.2: Cross-Lab Access Block
**Goal:** Verify multi-tenant isolation

1. Get a user ID from **Riopomba lab**
2. Try to read data from **different lab** (Mercês or staging):
   ```javascript
   // Login as Riopomba user, try to read Mercês data
   db.collection('labs').doc('merces')
     .collection('bioquimica-runs').get()
   ```
3. Expected: **Error: "Permission denied"**
4. ✅ **PASS** if denied; ❌ **FAIL** if reads succeed (data leak)

#### Test 1.3: Callable-Only Write Enforcement
**Goal:** Verify client-side writes to regulated collections are blocked

1. Try to create a laudo document from browser console:
   ```javascript
   // This should FAIL (create only via callable)
   db.collection('labs').doc('riopomba')
     .collection('laudos').add({
       paciente: 'test',
       resultado: 'normal'
     })
   ```
2. Expected: **Error: "Permission denied"**
3. ✅ **PASS** if denied; ❌ **FAIL** if write succeeds (security bypass)

### Status: Rules Enforcement = ✅ PASS (all 3 tests)
Proceed to data protection tests.

### 2. Data Protection Verification

#### Test 2.1: HTTPS/TLS Enforcement
**Goal:** Verify all traffic is encrypted

1. Open **https://hmatologia2.web.app** in browser
2. Check **address bar:** Lock icon visible, no warnings
3. Right-click → **Inspect** → **Security** tab
4. Expected:
   - [ ] Certificate valid (green checkmark)
   - [ ] Protocol: TLS 1.3 or 1.2
   - [ ] Cipher suite: strong (AES-256 or similar)
5. ✅ **PASS** if all green; ❌ **FAIL** if any warnings

#### Test 2.2: PII Anonymization (Feedback Module)
**Goal:** Verify old feedback is automatically anonymized (LGPD compliance)

1. **Manual check** (or ask Security Team):
   - Go to feedback collection in Firestore
   - Filter: `createdAt < [90 days ago]`
   - Spot-check 3 documents
   - Expected: `operatorId` = null or redacted; `feedback` = anonymized
2. ✅ **PASS** if anonymized; ❌ **FAIL** if PII present (LGPD violation)

#### Test 2.3: Audit Trail Logging
**Goal:** Verify write operations are logged

1. In Firestore, create a test run (via Bioquímica app)
2. Check **Cloud Audit Logs** (CloudAudit or Cloud Logging):
   ```bash
   gcloud logging read "resource.type=firestore_instance AND 
     protoPayload.methodName=google.firestore.v1.Firestore.Write" \
     --limit 10 --project=hmatologia2
   ```
3. Expected: Entry appears showing:
   - `timestamp`: when you created the run
   - `operatorId`: your user ID
   - `document`: path to the run (e.g., `/labs/riopomba/bioquimica-runs/...`)
   - `action`: "create" or "update"
4. ✅ **PASS** if entry logged; ❌ **FAIL** if no entry (audit trail broken)

### Status: Data Protection = ✅ PASS (all 3 tests)

## Issue Escalation

### Critical Issues (Block Deployment)
- ❌ Rules enforcement fails (hard-delete succeeds, cross-lab read works)
- ❌ Callable-only write bypass (client can write to `/laudos`)
- ❌ HTTPS/TLS missing or invalid certificate
- ❌ Compliance < 75%

**Action:** 
1. File **critical** issue in project tracker
2. Notify CTO immediately
3. Do NOT deploy until fixed and re-verified

### Medium Issues (Requires Fix, Can Delay)
- ⚠️ `z.any()` found in schema (unvalidated input)
- ⚠️ Anonymization cron not running (feedback still has PII)
- ⚠️ Audit trail incomplete (some operations not logged)

**Action:**
1. File **high** priority issue
2. Fix on current release or next sprint
3. Can deploy with mitigation plan documented

### Low Issues (Monitor, No Block)
- ℹ️ Non-critical warnings in console
- ℹ️ Optional security hardening (e.g., CSP headers)

**Action:**
1. Document in post-deploy report
2. Add to v1.4 backlog
3. Does not block deployment
