# v1.4 Post-Launch Incident Response System — START HERE

You are here because v1.4 deployment is imminent. This document is your entry point to a complete incident response infrastructure built for the **first 48 hours** after production go-live.

**Good news:** All tools, procedures, and documentation already exist. You just need to know where they are and how to use them.

---

## What This System Covers

- **8 critical incident scenarios** (rules lockout, function crash, quota breach, API timeouts, data loss, etc.)
- **Full rollback procedure** to v1.3 (if cascading failure detected)
- **Post-incident RCA template** (root cause analysis within 24h)
- **Pre-launch readiness checklist** (deployment gate, 72h before go-live)
- **Automated monitoring** (48h background script with periodic reports)
- **Quick reference card** (printable, taped to monitor)
- **Escalation paths** (when to call CTO, how to communicate with RT/auditor)

### Document 3: v1.4_POST_LAUNCH_INCIDENT_RUNBOOK.md

**The procedure manual. Use this during incidents.**

**8 Scenarios:**

1. Rules deny users (RDC lockout)
2. Cloud Functions 5xx (crash)
3. Firestore quota breach (spike)
4. NOTIVISA API timeout (gov down)
5. Gemini Vision API cost spike (IA running hot)
6. SMS delivery failure (Twilio down)
7. Critical data loss (accidental delete)
8. Cascading failure (multiple systems down) → full rollback

**Each scenario includes:**

- Detection signal (what metric indicates the problem)
- Decision tree (is this P0? call CTO? rollback?)
- Mitigation steps (in order, with CLI commands)
- Communication script (what to tell RT, auditor, CTO)
- If mitigation fails, escalation path

**Additional sections:**

- Full v1.3 rollback procedure (15–25 min)
- Post-incident RCA template (within 24h)
- Quick reference commands (copy-paste ready)
- Monitoring commands (Cloud Logs filters)

**When to use:** During 48h monitoring window OR active incident  
**Length:** ~1,185 lines  
**Reference time:** 5–10 min per scenario

### T-0h: Deployment

**Owner:** Deployer (follow standard deploy steps from `.claude/rules/deploy-protocol.md`)

```bash
# 1. Type-check
npx tsc --noEmit

# 2. Build
npm run build

# 3. Preflight
bash scripts/preflight-secrets-check.sh

# 4. Deploy (in order)
firebase deploy --only firestore:rules --project hmatologia2
firebase deploy --only functions --project hmatologia2
firebase deploy --only hosting --project hmatologia2
```

**After deploy:**

- Hard reload app in browser
- Check Firebase Console for immediate errors
- Start Cloud Logs monitoring

---

### T+0 to T+48h: Intensive Monitoring

**Owner:** On-call engineer

**What to do:**

| Time Window    | Cadence      | Action                                                                 |
| -------------- | ------------ | ---------------------------------------------------------------------- |
| T+0 to T+6h    | Real-time    | Don't leave desk. Watch Cloud Logs continuously.                       |
| T+6h to T+24h  | Every 30 min | Spot-check Cloud Logs for errors. Automated script runs in background. |
| T+24h to T+48h | Every 1h     | Lighter monitoring. Automated script continues.                        |

**Tool:** Keep this running in background

```bash
bash scripts/monitor-cloud-logs.sh 48 10
# Monitors for 48h, samples every 10 min, auto-generates report
```

**Tool:** Monitor by hand if automated script unavailable

```bash
gcloud logging read 'severity >= ERROR' --project=hmatologia2 --follow
```

**Decision triggers:**

| Error Rate             | Action                                      | Severity     |
| ---------------------- | ------------------------------------------- | ------------ |
| >1% sustained >10 min  | Call CTO immediately                        | P0           |
| 0.1–1%                 | Investigate, escalate if unresolved >15 min | P1           |
| <0.1%                  | Log, monitor (expected baseline)            | P2           |
| Any reported data loss | Restore from backup immediately             | P0 EMERGENCY |

**Reference:** v1.4_INCIDENT_QUICK_CARD.txt (at monitor) or runbook (detailed procedures)

---

### T+48h: Post-Launch Sign-Off

```
□ If >5 incidents (P0+P1): extend monitoring to T+72h

```

## If an Incident Occurs (During 48h Window)

### Quick Flow

1. **Declare:** Post in Slack `#ops-incidents`

   ```
   🚨 P[0|1] INCIDENT: [Scenario name]
   Status: Detected
   Impact: [users/modules affected]
   ETA: [recovery estimate]
   ```

2. **Reference:** Grab printed quick card at monitor

3. **Find scenario:** Takes <10 sec

4. **Follow action:** Takes 1–3 min

5. **Open full runbook:** If action says "see runbook"

6. **Execute:** Follow decision tree + mitigation steps

7. **Monitor:** Tail Cloud Logs for 5 min — confirm error rate recovered

8. **Communicate:** Update CTO + RT every 5 min until resolved

9. **Document:** Start RCA (complete within 24h)

## Common Questions

**Q: Do I need to read all 4 documents before launch?**  
A: No. QA Lead reads all 4 and completes the readiness checklist. On-call engineer reads 1 (overview) + 3 (runbook) + prints 4 (quick card).

**Q: What if I don't have time to complete the pre-launch checklist?**  
A: Don't deploy. The checklist exists to prevent problems. Rushing it defeats the purpose.

**Q: What if an incident occurs at 3am and I can't reach CTO?**  
A: Quick card has escalation path. Call backup contact. If truly alone and it's P0, execute full rollback (procedure in runbook). Document + RCA in morning.

**Q: How long will 48h monitoring take?**  
A: Automated script runs in background (non-blocking). Spot-checks take 5 min per 30-min interval. Real-time monitoring T+0 to T+6h only.

**Q: What if we see 0 errors in 48h?**  
A: That's success. Complete RCA template for documentation (even if "RCA: No incidents"). Hand off to Phase 2. Monitor script will auto-generate a clean report.

**Q: Do I need to memorize the 8 scenarios?**  
A: No. Quick card is your memory. But familiarize yourself with decision trees for Scenarios 1–3 (most likely: rules, functions, quota).
